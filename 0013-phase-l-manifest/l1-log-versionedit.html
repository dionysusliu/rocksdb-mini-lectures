TIER 3 · PHASE L · L1

共享 log 模块 + VersionEdit — MANIFEST 的两块地基

K 收官,M2 多版本引擎达成。进入 Tier 3 — Compaction。但在归并文件之前,引擎得先知道自己到底有哪些文件——而现在它是靠扫目录猜的(discover_ssts:列出 *.sst、按号排序、max+1 当下一个号)。这是 Phase C 就埋下的"故意空洞"。Phase L 把它换成 RocksDB 的做法:一份权威元数据日志 MANIFEST,记录"哪些文件活着"。

L1 打两块地基:① 把 WAL 的记录帧抽成一个 payload-无关的 log 模块——因为 MANIFEST 就是另一个日志文件,复用同一套帧;② VersionEdit——一条"文件集增量"的 tag 化记录,就是 MANIFEST 里 append 的那一条条 payload。

GOAL

① 新建 src/log.rs:把 wal.rs 里 crc+len+payload 的帧 + torn-tail 容错抽出来,成 LogWriter::append(&[u8]) / log::read_all(path),payload 无关。重构 wal.rs 用它。② 新建 src/version_edit.rs:VersionEdit{ new_files, deleted_files, next_file_number, last_sequence } + tag 化 encode/decode。

Done when: wal.rs 重构后既有 WAL 测试全回归(帧没变,只是搬了家);VersionEdit 任意字段组合 decode(encode(e)) == e;把一串 VersionEdit 当 payload 写进一个 log 文件、再 read_all + decode 能逐条还原(MANIFEST 的雏形)。

为什么"扫盘猜"不够

discover_ssts 在 MVP 阶段顶得很好,但它有三宗根本性的不足,每一宗都指向"需要一份权威记录":

扫盘的问题后果MANIFEST 怎么解
文件号靠 max+1 现推崩溃后若有"写了一半没记账"的文件,号可能撞、可能覆盖把 next_file_number 持久化,启动直接读
只能表达"有哪些文件",无法表达"删了哪个"compaction(Phase N)要丢弃被归并的旧文件,扫盘模型根本没有"删除"这个动作VersionEdit 带 deleted_files 增量
没地方记 last_sequence 等全局状态seq 只能靠扫 WAL/SST 重新推断全局状态作为 edit 字段落盘

本质:目录列表只是"文件的物理存在",而引擎需要的是"逻辑上哪些文件构成当前数据库"——这两者在 compaction 出现后会分叉(物理上文件还在,逻辑上已被取代)。MANIFEST 记的是后者。

地基一:framing ≠ payload

看一眼现在的 wal.rs:append(key,value) 把 KV 拼成 payload,外面再套 crc+len 帧。帧逻辑和 KV 编码是缠在一起的。但 MANIFEST 需要的是同一套帧、不同的 payload(payload 是 VersionEdit 而非 KV)。RocksDB 正是这么做的:db/log_writer.cc 一套 writer,WAL 和 MANIFEST 都用它。

帧(framing)载荷(payload)
WALcrc32 + len + bytes
torn-tail 容错
(共享 log 模块)		seq + klen + k + vlen + v
MANIFEST序列化的 VersionEdit

所以第一步是解耦:把帧抽进 log,让 wal.rs 退化成"KV 编解码 + 调 log",MANIFEST 将来就是"VersionEdit 编解码 + 调 log"。

代码 1 — 共享帧 log.rs

    //! frame = crc32(4) + payload_len(4) + payload。payload 无关。
    pub struct LogWriter { w: BufWriter<File> }

    impl LogWriter {
        pub fn open_append(path: &Path) -> Result<Self> {
            let f = OpenOptions::new().create(true).append(true).open(path)?;
            Ok(Self { w: BufWriter::new(f) })
        }
        pub fn create_truncate(path: &Path) -> Result<Self> {       // 轮换/reset 用
            let f = OpenOptions::new().create(true).write(true).truncate(true).open(path)?;
            Ok(Self { w: BufWriter::new(f) })
        }
        // 追加一条不透明记录——payload 的 schema 是调用方的事
        pub fn append(&mut self, payload: &[u8]) -> Result<()> {
            let crc = crc32fast::hash(payload);
            self.w.write_all(&crc.to_le_bytes())?;
            self.w.write_all(&(payload.len() as u32).to_le_bytes())?;
            self.w.write_all(payload)?;
            Ok(())
        }
        pub fn sync(&mut self) -> Result<()> {
            self.w.flush()?; self.w.get_ref().sync_all()?; Ok(())  // buffer→OS→disk
        }
    }

    // 读出全部完好记录,遇撕裂/损坏停在尾部(容半写尾)。文件不存在 → 空。
    pub fn read_all(path: &Path) -> Result<Vec<Vec<u8>>> {
        let mut buf = Vec::new();
        match File::open(path) {
            Ok(mut f) => { f.read_to_end(&mut buf)?; }
            Err(e) if e.kind() == ErrorKind::NotFound => return Ok(vec![]),
            Err(e) => return Err(e.into()),
        }
        let (mut out, mut pos) = (Vec::new(), 0);
        while pos + 8 <= buf.len() {
            let crc = u32::from_le_bytes(buf[pos..pos+4].try_into().unwrap());
            let len = u32::from_le_bytes(buf[pos+4..pos+8].try_into().unwrap()) as usize;
            let start = pos + 8;
            if start + len > buf.len() { break; }              // 没写全 → 撕裂尾
            let payload = &buf[start..start+len];
            if crc32fast::hash(payload) != crc { break; }     // 损坏
            out.push(payload.to_vec());
            pos = start + len;
        }
        Ok(out)
    }

代码 2 — wal.rs 瘦身(现在只管 KV payload)

    use crate::log::{self, LogWriter};
    pub struct Wal { w: LogWriter }            // 不再自己持 BufWriter / 写 crc

    impl Wal {
        pub fn open_append(p: &Path) -> Result<Self> { Ok(Self { w: LogWriter::open_append(p)? }) }
        pub fn reset(p: &Path) -> Result<Self> { Ok(Self { w: LogWriter::create_truncate(p)? }) }
        pub fn sync(&mut self) -> Result<()> { self.w.sync() }

        pub fn append(&mut self, seq: u64, key: &[u8], value: &[u8]) -> Result<()> {
            let mut p = Vec::new();                        // payload = seq|klen|k|vlen|v
            p.extend_from_slice(&seq.to_le_bytes());
            p.extend_from_slice(&(key.len() as u32).to_le_bytes()); p.extend_from_slice(key);
            p.extend_from_slice(&(value.len() as u32).to_le_bytes()); p.extend_from_slice(value);
            self.w.append(&p)                            // 帧的事全交给 log
        }
        pub fn replay(p: &Path) -> Result<Vec<(u64, Bytes, Bytes)>> {
            let mut out = Vec::new();
            for rec in log::read_all(p)? { out.push(decode(&rec)?); }  // log 吐完好记录,这里只解 KV
            Ok(out)
        }
    }

注意 append 现在多带了一个 seq——这是 H 阶段引入 InternalKey 后 WAL 本就该存的(replay 要恢复 seq)。重构顺手把签名对齐。关键是帧逻辑彻底没了:crc、长度、torn-tail 全下沉到 log,wal.rs 只剩 KV schema。

代码 3 — VersionEdit(version_edit.rs)

    // 字段 tag。预留空间;Phase M 加 level/key-range 只是新增 tag。
    const TAG_NEXT_FILE_NUMBER: u8 = 1;
    const TAG_LAST_SEQUENCE: u8 = 2;
    const TAG_NEW_FILE: u8 = 3;
    const TAG_DELETED_FILE: u8 = 4;

    // 一条对当前 Version 的增量。字段稀疏:只带变化的部分。
    #[derive(Debug, Default, Clone, PartialEq, Eq)]
    pub struct VersionEdit {
        pub new_files: Vec<u64>,        // 本 edit 新增的 SST
        pub deleted_files: Vec<u64>,    // 本 edit 删除的 SST(compaction 起,Phase N+)
        pub next_file_number: Option<u64>,
        pub last_sequence: Option<u64>,
    }

    impl VersionEdit {
        // 编码 = 一串 (tag, 值)。数值用定长 u64 LE(RocksDB 用 varint,这里求简单清晰)。
        pub fn encode(&self) -> Vec<u8> {
            let mut out = Vec::new();
            if let Some(n) = self.next_file_number { out.push(TAG_NEXT_FILE_NUMBER); out.extend_from_slice(&n.to_le_bytes()); }
            if let Some(s) = self.last_sequence    { out.push(TAG_LAST_SEQUENCE);   out.extend_from_slice(&s.to_le_bytes()); }
            for &f in &self.new_files     { out.push(TAG_NEW_FILE);     out.extend_from_slice(&f.to_le_bytes()); }
            for &f in &self.deleted_files { out.push(TAG_DELETED_FILE); out.extend_from_slice(&f.to_le_bytes()); }
            out
        }
        // 解码 = 按 tag 分派,每个 tag 自管自己的 codec。新增字段 = 多一个 match 臂。
        pub fn decode(buf: &[u8]) -> Result<Self> {
            let (mut edit, mut i) = (VersionEdit::new(), 0);
            while i < buf.len() {
                let tag = buf[i]; i += 1;
                match tag {
                    TAG_NEXT_FILE_NUMBER => edit.next_file_number = Some(read_u64(buf, &mut i)?),
                    TAG_LAST_SEQUENCE    => edit.last_sequence    = Some(read_u64(buf, &mut i)?),
                    TAG_NEW_FILE     => edit.new_files.push(read_u64(buf, &mut i)?),
                    TAG_DELETED_FILE => edit.deleted_files.push(read_u64(buf, &mut i)?),
                    other => bail!("unknown VersionEdit tag {other}"),
                }
            }
            Ok(edit)
        }
    }

为什么 deleted_files 现在就要? Phase L 没有任何东西删文件(删文件是 compaction,Phase N)。但"删除"恰恰是 edit 这个设计的灵魂——若只会"加",那直接存一个文件列表就够了,何必搞增量日志?正因为将来要"加一些、减一些",才用 delta。删除分支现在就建好、用合成 edit 测(下一课 L2 的 Version fold 会真正消费它),Phase N 直接拿来用。这与"惰性字段(level/key-range)推迟到 M"不矛盾:那些这期没有行为,而 delete 是 L2 折叠的核心行为。

🧰 Coding style:解耦 · 可扩展协议 · 稀疏增量

  • framing ≠ payload:把"怎么把一段字节安全存进文件"(帧:长度、校验、容错)和"这段字节是什么"(schema:KV / VersionEdit)分到两层。一旦分开,新载荷(MANIFEST)就能零成本复用整套帧 + 容错。判据:当两个东西"存储方式相同、内容不同",存储方式就该独立成层。
  • tag 化协议,为扩展而设计:不用定长 struct,用 tag + 值 的自描述流。加字段 = 加一个 tag 常量 + 一个 match 臂,老数据照样解。这是所有要长期演进的磁盘/网络格式的通用形态(Protobuf field number、TLV、BSON 都是这个骨架)。
  • 稀疏增量(Option 字段):一条 edit 只编码变了的字段。flush 的 edit 只带 new_files + 两个计数器,不带 delete。这让"edit = 一个 delta"在编码层就是字面真相,而非全量快照。

🧠 System mindset:一切皆日志(state = fold(events))

MANIFEST 不是一张"当前文件表",而是一串增量事件的 append-only 日志;当前文件集 = 从头把这些事件折叠一遍的结果(下一课 L2 的 Version::apply)。这正是 event sourcing 的核心:不存当前状态,存导致状态的事件流,状态是事件的折叠。一旦你认出这个形,到处都是它:

系统append-only 事件当前状态 = fold(events)
RocksDB / 本项目MANIFEST 里的 VersionEditVersion(活文件集)
Gitcommit 对象(只追加)HEAD 指向的树 = 历史折叠
Kafka分区 logconsumer 折叠出的物化视图
数据库 redo log事务 redo 记录重放恢复出的页
会计 / 区块链分录 / 交易余额 = 全部分录之和

迁移收获:当你纠结"该存当前状态还是状态变更"时,日志派的答案几乎总更稳——事件不可变、可追加、可重放、可审计,而"当前状态"只是一个可随时从日志重建的缓存。RocksDB 把 WAL 和 MANIFEST 都建在同一个 log_writer 上,不是巧合:KV 数据和元数据,都用"先写日志"这一招扛崩溃。

RocksDB 锚: db/log_writer.cc · db/log_reader.cc(WAL 与 MANIFEST 复用同一套 log writer/reader)

🧪 Harness 系统:编码 round-trip + 日志 torn-tail

两条性质

① VersionEdit round-trip:任意字段组合(全字段 / 只 add+计数器的 flush edit / 空 edit),decode(encode(e)) == e。这把磁盘格式钉成可逆,是后面一切恢复正确的前提。② 日志 torn-tail(白嫖 WAL):往 log 写几条记录,set_len 砍掉尾几字节,read_all 应吐出撕裂前的前缀、丢残尾、不 panic。这条性质 MANIFEST 直接继承——半写的 VersionEdit = 撕裂的尾记录。

    #[test] // VersionEdit 全字段 round-trip
    fn roundtrip_full_edit() {
        let mut e = VersionEdit::new();
        e.new_files = vec![7, 8, 9]; e.deleted_files = vec![3];
        e.next_file_number = Some(10); e.last_sequence = Some(4242);
        assert_eq!(VersionEdit::decode(&e.encode()).unwrap(), e);
    }

    #[test] // 日志撕裂尾:丢残尾、留前缀
    fn torn_tail_drops_last_record() {
        let p = dir.path().join("t.log");
        { let mut w = LogWriter::open_append(&p).unwrap();
          w.append(b"keep").unwrap(); w.append(b"lose-me").unwrap(); w.sync().unwrap(); }
        let len = std::fs::metadata(&p).unwrap().len();
        OpenOptions::new().write(true).open(&p).unwrap().set_len(len - 3).unwrap();  // 撕
        assert_eq!(log::read_all(&p).unwrap(), vec![b"keep".to_vec()]);
    }
