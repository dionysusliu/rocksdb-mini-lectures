TIER 3 · PHASE L · L3

接进 Engine — 扫盘退役

MANIFEST 子系统(L1 的 log+VersionEdit、L2 的 Version+recover+install)造齐了,但还悬在引擎之外。L3 把它焊上:让 open 用 recover 替换 discover_ssts(扫盘正式退役)、flush 发一条 VersionEdit 并定下崩溃排序、next_sst 升级成共享的 next_file_number。收尾用一条"往目录扔孤儿 SST、断言读不到"的回归守卫,把"引擎信 manifest、不信目录"钉死。

GOAL

改 engine.rs:① open = replay WAL + Manifest::recover 重建 Version + open-time 强制 checkpoint(删 discover_ssts);② flush 发 edit,排序 SST(fsync)→ manifest edit(fsync=提交点)→ 截 WAL;③ next_sst → next_file_number(SST 与 MANIFEST 共用);④ last_sequence = max(manifest 值, WAL replay 最大 seq)。

Done when: 重开经 manifest 取回全部数据(已 flush 的来自 SST、未 flush 的来自 WAL);文件号跨重开单调不撞;孤儿 SST(目录里有、manifest 没记)读不到;既有引擎测试全回归。

open:从"扫盘猜"到"读账本"

对照旧 open 的 discover_ssts(列目录、排序、max+1)和新 open:数据来源从"文件系统目录列表"换成"MANIFEST 这本账"。四步:

    pub fn open(dir: &Path) -> Result<Self> {
        std::fs::create_dir_all(dir)?;
        let wal_path = dir.join("wal.log");

        // 1. replay WAL → memtable,记下见过的最大 seq
        let (mut mem, mut max_wal_seq) = (MemTable::new(), 0u64);
        for (seq, k, v) in Wal::replay(&wal_path)? { mem.put(k, v); max_wal_seq = max_wal_seq.max(seq); }
        let wal = Wal::open_append(&wal_path)?;

        // 2. 从 MANIFEST 恢复活文件集(替换 discover_ssts)。None = 全新库。
        let (version, mut next_file_number, manifest_seq) = match Manifest::recover(dir)? {
            Some(r) => (r.version, r.next_file_number, r.last_sequence),
            None    => (Version::new(), 0, 0),
        };
        let seq = manifest_seq.max(max_wal_seq);          // ← manifest 给地板,WAL 续高

        // 3. 只为 manifest 列出的文件开 reader,新→旧
        let mut ssts = Vec::new();
        for num in version.files_newest_first() { ssts.push(SstReader::open(&sst_path(dir, num))?); }

        // 4. open-time 强制 checkpoint:把恢复出的 Version 快照进新 MANIFEST、原子切 CURRENT(折叠历史)
        let manifest_num = next_file_number; next_file_number += 1;
        let manifest = Manifest::install(dir, manifest_num, &version, next_file_number, seq)?;

        Ok(Self { mem, wal, ssts, version, manifest, dir: dir.into(),
                  next_file_number, seq, flush_threshold: 4<<20, max_manifest_size: 64<<10 })
    }

第 4 步:open 现在带写副作用——每次开库都写一个新 MANIFEST、切 CURRENT。这是故意的:把上次开库以来累积的一长串增量 edit 折成一条全量快照,下次开库 replay 量恒定(L2 🧠 的"checkpoint")。它复用 L2 的 install,所以每次 open 都跑一遍原子安装——崩溃 harness 因此每次重开都有靶子。全新库(recover 返回 None)走同一条路:空 Version 也 install,写出 MANIFEST-000000 + CURRENT,完成 bootstrap。

last_sequence 为什么取 max? manifest 里存的是上次 flush 时的 last_sequence——那之后、崩溃前的写都还在 WAL 里(没进 SST,所以没进 manifest)。这些 WAL 记录带着更高的 seq。所以 manifest 给"地板",WAL replay 把它续到真正的最高水位。漏了这个 max,重启后新写的 seq 会和未 flush 的旧写撞号,破坏多版本定序。

flush:发一条 edit,定一个提交点

旧 flush 是"写 SST → 截 WAL"。现在中间多一条记账,三个动作的顺序就是崩溃安全的全部:

1. 写 SST 文件,fsync // 数据先落盘
2. manifest.append_edit(&edit),fsync ← 提交点
3. 截断 WAL + 更新内存 // 只有 manifest 认账了才敢丢 WAL 
    pub fn flush(&mut self) -> Result<()> {
        if self.mem.is_empty() { return Ok(()); }
        // (1) 写 SST,fsync(在 finish() 内)。号从共享计数器取。
        let num = self.next_file_number; self.next_file_number += 1;
        let path = sst_path(&self.dir, num);
        let mut b = SstBuilder::new();
        for (k, v) in self.mem.iter() { b.add(&k, &v); }
        b.finish(&path)?;

        // (2) 记账并 fsync —— 这条 fsync 是提交点。
        let mut edit = VersionEdit::new();
        edit.new_files = vec![num];
        edit.next_file_number = Some(self.next_file_number);
        edit.last_sequence = Some(self.seq);
        self.manifest.append_edit(&edit)?;

        // (3) 此刻才安全:更新内存视图、丢 WAL。
        self.version.apply(&edit);
        self.ssts.insert(0, SstReader::open(&path)?);
        self.wal = Wal::reset(&self.dir.join("wal.log"))?;
        self.mem = MemTable::new();

        // 体积过阈轮换(与 open-time checkpoint 同一台机器)。
        if self.manifest.size() > self.max_manifest_size { self.rotate_manifest()?; }
        Ok(())
    }

    fn rotate_manifest(&mut self) -> Result<()> {
        let manifest_num = self.next_file_number; self.next_file_number += 1;
        self.manifest = Manifest::install(&self.dir, manifest_num, &self.version, self.next_file_number, self.seq)?;
        Ok(())
    }

崩在每个点,都不丢数据

崩在哪磁盘状态恢复结果
(1)后 (2)前SST 在盘但 manifest 没记SST 成孤儿(不在 Version);数据仍在 WAL,replay 补回 ✅
(2)后 (3)前SST 进 Version 且 WAL 还在manifest 给 SST;WAL 把同批 KV 按原 seq 再灌回 mem;按 seq 去重幂等 ✅

孤儿 + 文件号复用(讲但不治):崩在 (1)(2) 之间,号 num 没持久化进 manifest,下次 open 的 next_file_number 是旧值,可能复用 num 覆盖那个孤儿。安全,因为孤儿的数据还在 WAL、覆盖无损。孤儿文件的物理回收是 Phase X 的 DeleteObsoleteFiles;本期靠 WAL 兜底,只讲不治。WAL 仍整截(不引入 log number 标记法,那也是 Phase X)。

🧠 System mindset:提交点 —— 元数据也走 write-ahead

看这个对称:

谁的提交记录提交点提交前崩 → 视为没发生
一次 putWAL 记录WAL 那条 fsync数据不进 memtable
一次 flushmanifest VersionEditedit 那条 fsyncSST 成孤儿,当没 flush 过

同一招,两个层级。 用户数据靠 WAL 先写日志扛崩溃;元数据(哪些文件构成数据库)靠 MANIFEST 先写日志扛崩溃。一次 flush 的"原子提交点"不是"写完 SST"——SST 写完但没记账,等于没发生——而是"manifest edit 落盘"。这是 write-ahead logging 的精髓:先把"我要做什么"持久化成一条日志记录,再去做;那条记录的 fsync 就是不可逆的提交线,线之前一切可当作没发生过而安全回滚。

迁移收获:数据库事务的 redo-before-data、文件系统 journaling 的 journal-commit-before-checkpoint、2PC 的 prepare→commit、甚至 TCP 的"ack 才算送达"——都是同一个骨架:把"生效"绑定到一个单一、持久、原子的提交标记上。设计任何"多步骤、中途可能崩"的操作,先问:我的提交点是哪一行?它之前的步骤都可重做吗?

RocksDB 锚: db/version_set.cc VersionSet::LogAndApply(把 edit 写进 MANIFEST = 一次原子的元数据提交)

🧪 Harness + 🔄 CI/CD:孤儿 SST 回归守卫

本课旗舰 harness 也是这条流水线的关键守卫:证明扫盘真的死透了。构造一个目录里物理存在、但 manifest 从未记录的 SST,断言引擎读不到它——若引擎还在偷偷扫目录,这个孤儿就会被读出来,测试就红。

三条性质

① 经 manifest 重开:put 几个、flush(进 manifest)、再 put(留 WAL),重开后全部读回(SST + WAL 两路)。② 孤儿 SST 不可见:flush 后手工伪造一个合法 SST 丢进目录(manifest 未记),重开断言它的 key 读不到、真数据照常。③ 文件号跨重开单调:重开后再 flush 不复用号、不覆盖旧 SST。

    #[test] // 头牌回归:引擎信 manifest、不信目录
    fn orphan_sst_is_ignored() {
        { let mut e = Engine::open(dir).unwrap();
          e.put("real".., "yes"..).unwrap(); e.flush().unwrap(); }
        // 伪造一个完全合法、但 manifest 从未提及的 SST
        let mut forged = SstBuilder::new();
        forged.add(b"ghost", b"boo");
        forged.finish(&dir.join("999999.sst")).unwrap();

        let e2 = Engine::open(dir).unwrap();
        assert_eq!(e2.get(b"real").as_deref(), Some(&b"yes"[..]));
        assert_eq!(e2.get(b"ghost"), None, "孤儿 SST 不可被读到");   // 扫盘若没死,这里会红
    }
