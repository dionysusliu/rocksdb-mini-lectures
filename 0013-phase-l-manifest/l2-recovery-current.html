TIER 3 · PHASE L · L2

Version 折叠 + 恢复 + CURRENT 原子切换

L1 造了两块零件:log(帧)和 VersionEdit(增量记录)。L2 把它们拼成真正的 MANIFEST 子系统:① Version——把一串 edit 折叠成"当前活文件集";② 恢复——open 时读 CURRENT → 打开它指向的 MANIFEST → replay 重建 Version;③ 原子安装——写新 MANIFEST + 原子切 CURRENT 的那套序列。

这是整个 Phase L 的旗舰。压轴是一个新 harness 系统:元数据崩溃恢复——沿安装序列每一步注入崩溃,断言恢复永远落在某个一致状态(全旧或全新),绝不撕裂。而 CURRENT 的 rename 这个"原子发布",正是你 backlog 里"无锁发布新状态"在文件粒度上的同一招。

GOAL

① src/version.rs:Version(活文件集) + apply(edit)(add/delete 双分支折叠) + to_snapshot_edit() + files_newest_first()。② src/manifest.rs:recover(dir)(读 CURRENT→replay→重建) + install(...)(7 步原子安装,含 fsync 父目录) + append_edit()。

Done when: 一串 edit 折叠出正确的活文件集(含 delete 移除分支);写 MANIFEST + 切 CURRENT 后 recover 拿回同一个 Version;崩溃恢复四例(a/b1/b2/c)全绿——撕裂尾取前缀、切换前崩落全旧、残 tmp 被无视。

地基三:Version = fold(edits)

MANIFEST 是一串 VersionEdit。"当前有哪些文件"不直接存,而是从空集开始,把每条 edit 叠一遍:add 插入、delete 移除。这就是 L1 🧠 讲的 event-sourcing 折叠。扁平 MVP 下 Version 就是一组文件号(分层留给 Phase M):

    // 扁平 MVP:就是一组活文件号(无 level)。读序由号导出:号越大=越晚 flush=越新。
    #[derive(Debug, Default, Clone)]
    pub struct Version { files: BTreeSet<u64> }

    impl Version {
        // 折叠一条 edit:add 插入,delete 移除。
        pub fn apply(&mut self, edit: &VersionEdit) {
            for &f in &edit.new_files     { self.files.insert(f); }
            for &f in &edit.deleted_files { self.files.remove(&f); }   // ← edit 的灵魂分支
        }
        // 活文件号,新→旧(降序),喂给读路径。
        pub fn files_newest_first(&self) -> Vec<u64> {
            self.files.iter().rev().copied().collect()
        }
        // 把当前活集压成一条全量 edit——新 MANIFEST 的首记录(把历史折成一个 checkpoint)。
        pub fn to_snapshot_edit(&self, next_file_number: u64, last_sequence: u64) -> VersionEdit {
            VersionEdit {
                new_files: self.files.iter().copied().collect(),
                deleted_files: Vec::new(),
                next_file_number: Some(next_file_number),
                last_sequence: Some(last_sequence),
            }
        }
    }

读序为什么能"由号导出"? Phase L 文件只来自 flush,号单调递增,号大 = 后 flush = 数据新。所以"号降序"就是"新→旧",与原 discover_ssts(sort + rev)同一规则。注意这个假设到 Phase O(compaction)会失效——一个 L1 输出文件号很大却装着旧数据——那正是 Phase M 要引入 level 的原因。本期诚实地继续用号序。

地基四:CURRENT 与原子安装

恢复要先知道"读哪个 MANIFEST"。这就是 CURRENT 文件的活:它存当前活跃 MANIFEST 的裸文件名(如 MANIFEST-000007\n)。而每当我们写一个新 MANIFEST(open 强制 checkpoint,或运行期体积过阈轮换——见 L3),都要把 CURRENT 原子地切过去。原子序列(对标 RocksDB SetCurrentFile):

1. 写快照 edit 进 MANIFEST-<n> (CURRENT 仍指旧)
2. fsync(MANIFEST-<n>) // 新文件落盘后才敢切
3. 写 CURRENT.tmp = "MANIFEST-<n>\n"
4. fsync(CURRENT.tmp)
5. rename(CURRENT.tmp → CURRENT) ← 原子提交点(POSIX rename 原子)
6. fsync(父目录) // 让 rename 这个目录改动本身持久

第 5 步是唯一的提交点:rename 在 POSIX 上是原子的,要么旧名要么新名,绝无"半个文件名"。崩在 5 之前 → CURRENT 仍旧 → 恢复读到完整旧 MANIFEST;崩在 5 之后 → 读到完整新 MANIFEST。两种都是一致状态。第 6 步常被漏掉:rename 改的是目录项,不 fsync 目录,这条改动可能没落盘,崩溃后 CURRENT 退回旧值——所以要 fsync 父目录,让"切换"这个动作本身持久。

代码 1 — manifest.rs(recover + install)

    pub struct Recovered { pub version: Version, pub next_file_number: u64, pub last_sequence: u64 }
    pub struct Manifest { writer: LogWriter, size: u64 }   // size:轮换阈值用(L3)

    fn manifest_name(num: u64) -> String { format!("MANIFEST-{num:06}") }

    impl Manifest {
        // 读 CURRENT 指向的 MANIFEST、replay 重建。None = 全新库(无 CURRENT)。
        pub fn recover(dir: &Path) -> Result<Option<Recovered>> {
            let name = match fs::read_to_string(dir.join("CURRENT")) {
                Ok(s) => s.trim().to_string(),
                Err(e) if e.kind() == ErrorKind::NotFound => return Ok(None),  // 全新库
                Err(e) => return Err(e.into()),
            };
            let (mut version, mut nfn, mut ls) = (Version::new(), 0, 0);
            for rec in log::read_all(&dir.join(&name))? {   // log 容错:撕裂尾自动停
                let edit = VersionEdit::decode(&rec)?;
                version.apply(&edit);                          // 折叠
                if let Some(n) = edit.next_file_number { nfn = n; }   // 后写覆盖先写
                if let Some(s) = edit.last_sequence    { ls = s; }
            }
            Ok(Some(Recovered { version, next_file_number: nfn, last_sequence: ls }))
        }

        // 原子安装一个新 MANIFEST(首记录=version 的全量快照),再把 CURRENT 指过去。
        // open 强制 checkpoint 与 体积过阈轮换 共用这一台机器。
        pub fn install(dir: &Path, manifest_num: u64, version: &Version,
                       next_file_number: u64, last_sequence: u64) -> Result<Self> {
            let path = dir.join(manifest_name(manifest_num));
            let mut writer = LogWriter::create_truncate(&path)?;
            let payload = version.to_snapshot_edit(next_file_number, last_sequence).encode();
            let size = (payload.len() + 8) as u64;
            writer.append(&payload)?;
            writer.sync()?;                                    // (2) 新 MANIFEST 落盘

            let tmp = dir.join("CURRENT.tmp");
            { let mut f = fs::File::create(&tmp)?;
              writeln!(f, "{}", manifest_name(manifest_num))?;        // (3)
              f.sync_all()?; }                                // (4)
            fs::rename(&tmp, dir.join("CURRENT"))?;             // (5) 原子提交点
            fsync_dir(dir)?;                                   // (6) 让 rename 持久
            Ok(Self { writer, size })
        }

        // 追加一条增量 edit 并 fsync——这条 fsync 是 flush 的提交点(L3 用)。
        pub fn append_edit(&mut self, edit: &VersionEdit) -> Result<()> {
            let payload = edit.encode();
            self.size += (payload.len() + 8) as u64;
            self.writer.append(&payload)?; self.writer.sync()?; Ok(())
        }
    }

    // fsync 一个目录,让其中的 rename/create 持久。
    fn fsync_dir(dir: &Path) -> Result<()> { fs::File::open(dir)?.sync_all()?; Ok(()) }

恢复里"后写覆盖先写":一个 MANIFEST 开头是全量快照(带 next_file_number/last_sequence),后面跟若干增量 edit(flush 每次也带新的计数器值)。replay 时按出现顺序覆盖,最后留下的就是最新值。这正是日志折叠:同一字段的多次写,时间靠后的赢。

🧰 Coding style:制造-再发布,绝不原地改

  • 从不原地改活状态:要更新 CURRENT,不是打开它写两个字节(那会留下"改了一半"的窗口),而是在旁边造一个完整的新文件(CURRENT.tmp),再用一步 rename 让它整体顶替。读者任一时刻看到的 CURRENT 要么是完整旧的、要么是完整新的。
  • 把"提交"收缩成单点:整个安装序列里,只有 rename 那一行是不可分割的提交。它之前的所有写都是"准备",可随意重来或丢弃;它一旦完成,新状态整体生效。设计崩溃安全的通法:把多步操作压成"一堆可重做的准备 + 一个原子提交"。
  • 顺序即正确性:fsync 新文件 必须 在切 CURRENT 之前(否则 CURRENT 指向一个可能没落盘的文件);fsync 目录 必须 在最后(否则切换本身可能丢)。这些顺序不是装饰,是约束。

🧠 System mindset ★:原子指针切换 —— 三种粒度,同一个不变量

CURRENT 的 rename 是一次原子发布:把一个新状态"发布"出去,让所有后续读者要么完全看到、要么完全看不到,绝无中间态。这个模式你已经见过、还会反复见到——它在三种粒度上是同一回事:

粒度原子操作发布什么在本课程
字级CAS(compare-and-swap)无锁跳表插一个新节点backlog:并发 MemTable
指针级原子换 Arc / SuperVersion 指针RCU 发布新只读 VersionD 阶段 Arc+freeze
文件级renameCURRENT 切到新 MANIFEST本课

三者本质同构:

先在旁边把新状态造完整,再用一步原子操作让它可见;读者永远看不到半成品。 字级靠 CPU 的 CAS 指令保证那一步原子;指针级靠原子指针写;文件级靠文件系统的 rename。机制不同,纪律相同。

连上你的 backlog(并发写 / 无锁发布)。 你标记过要深挖"LSM 写并发模型:无锁跳表 CAS+epoch、RocksDB 用 Arc 发布 SuperVersion"。那道题的核心难点就是这一个:一个写者如何在不阻塞、不让读者看到撕裂状态的前提下,发布一份新数据? 答案在三种粒度上都是"制造-再发布"。本课先在文件粒度把它吃透(rename 最直观、可在测试里手工注入崩溃);到 Tier 4 重构成多线程后,同一招会上升到指针粒度(原子换 SuperVersion),那时你对"为什么发布必须原子、为什么不能原地改"已经有了肌肉记忆。RocksDB 自己就是这么统一的:InstallSuperVersion(指针级)和 SetCurrentFile(文件级)是同一个"原子发布"哲学的两次落地。

RocksDB 锚: db/version_set.cc SetCurrentFile(写 tmp → rename → fsync dir,与本课 install 逐行对应) · 对照 InstallSuperVersion(指针级同款)

🧪 Harness 系统:元数据崩溃恢复(新系统,累计第 8 种)

测试里没有真崩溃。手法:手工构造"崩在第 X 步"会留在磁盘上的字节状态,再 recover,断言落在一致状态。安装序列每步之间都是崩溃点。旗舰不变量:恢复出的 Version 永远是某个一致状态(全旧或全新),绝不撕裂。

代码 2 — 崩溃恢复 harness

    fn recovered_files(dir: &Path) -> Vec<u64> {     // 恢复出活文件集,新→旧
        Manifest::recover(dir).unwrap().map(|r| r.version.files_newest_first()).unwrap_or_default()
    }

    #[test] // (a) 增量 edit 撕裂尾 → 取前缀
    fn crash_a_torn_incremental_edit() {
        let mut v = Version::new(); let mut e0 = VersionEdit::new(); e0.new_files = vec![1]; v.apply(&e0);
        let mut m = Manifest::install(dir, 0, &v, 1, 10).unwrap();        // 快照 {1}
        let mut e1 = VersionEdit::new(); e1.new_files = vec![2]; e1.next_file_number = Some(3);
        m.append_edit(&e1).unwrap();                                // 追加 {add 2}
        let mp = manifest::current_target(dir).unwrap().unwrap();
        let len = fs::metadata(&mp).unwrap().len();
        OpenOptions::new().write(true).open(&mp).unwrap().set_len(len - 3).unwrap();   // 撕
        assert_eq!(recovered_files(dir), vec![1]);              // 残尾被丢,只剩快照 {1}
    }

    #[test] // (b1) 新 MANIFEST 完整但 CURRENT 没切 → 全旧
    fn crash_b1_new_manifest_complete_not_switched() {
        let mut old = Version::new(); let mut e0 = VersionEdit::new(); e0.new_files = vec![1]; old.apply(&e0);
        Manifest::install(dir, 0, &old, 1, 10).unwrap();             // CURRENT → MANIFEST-000000
        let mut newv = old.clone(); let mut e1 = VersionEdit::new(); e1.new_files = vec![2]; newv.apply(&e1);
        { let mut w = LogWriter::create_truncate(&dir.join("MANIFEST-000002")).unwrap();
          w.append(&newv.to_snapshot_edit(3, 20).encode()).unwrap(); w.sync().unwrap(); }  // 完整新文件,但没切
        assert_eq!(current_name(dir), "MANIFEST-000000");
        assert_eq!(recovered_files(dir), vec![1]);              // 仍全旧 {1};新文件是孤儿
    }

    #[test] // (b2) 新 MANIFEST 只写一半 → 全旧,半文件根本不被读
    fn crash_b2_new_manifest_half_written() {
        /* install old {1} 同上 */
        fs::write(dir.join("MANIFEST-000002"), b"\x00\x01\x02half").unwrap();  // 半截垃圾
        assert_eq!(recovered_files(dir), vec![1]);   // CURRENT 没切 → 半文件永不被打开
    }

    #[test] // (c) 残留 CURRENT.tmp → 被无视
    fn crash_c_leftover_current_tmp() {
        /* install old {1} 同上 */
        fs::write(dir.join("CURRENT.tmp"), b"MANIFEST-000002\n").unwrap();
        assert_eq!(recovered_files(dir), vec![1]);   // 恢复只认 CURRENT,tmp 是噪音
    }

b1 vs b2 的细分意义: b1 证明"即便新 MANIFEST 写得完美无瑕,只要 CURRENT 没切,它就等于不存在";b2 进一步证明"哪怕新 MANIFEST 是残的也无所谓,反正 CURRENT 没切、没人会去读它"。两例合起来钉死一句话:新文件的完整性根本不影响一致性——决定一切的是那个 rename 切没切。这就是把提交收缩成单点的回报。

🔄 CI/CD:崩溃恢复进回归门

本课增量

四个崩溃恢复测试是纯文件手术 + 公开 recover 断言,无需特殊环境,直接并入 cargo test --all。意义重大:崩溃安全这种"平时看不见、出事才致命"的属性,被钉成了每次提交都自动验证的回归测试。以后任何人改动安装序列(比如手贱把 fsync 顺序调换、或省掉 fsync dir),只要破坏了"全旧/全新"不变量,CI 立刻红。这是把"崩溃一致性"从口头约定变成可执行规格。L1 的格式可逆性 + L2 的崩溃一致性,两道砝码叠起来,这条流水线已经在守护引擎最难肉眼复查的两类正确性。

核心问题(自己答)

L2 想透这几个
  1. 为什么 install 第 2 步(fsync 新 MANIFEST)必须在切 CURRENT 之前,而第 6 步(fsync 父目录)必须在最后? 把这两个顺序各自调换,分别会在什么崩溃时机暴露什么 bug?
  2. b1 和 b2 恢复结果完全一样(全旧),为什么还要拆成两个测试? 各自证明了什么独立的事实?
  3. CURRENT 的 rename、RCU 的 Arc 指针交换、无锁跳表的 CAS,共享哪一个不变量? 用一句话概括,并说出三者各自"那一步原子"是谁保证的。
  4. 恢复时 next_file_number 用"后写覆盖先写"取最后一个值——如果某条增量 edit 没带这个字段(Option 为 None)会怎样? 这对"稀疏增量"编码提了什么要求?
