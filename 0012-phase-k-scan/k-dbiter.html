STEP K · Phase K 对外范围扫描

DBIter + scan_as_of — 范围读的时间旅行

G2 的 scan 把"去重"塞进了 MergeIterator,取的是全局最新版本——不认快照。本课把这层语义拆出来,做成一个对外游标 DBIter,让范围读和点读一样能按任意快照读历史。

GOAL

新模块 DBIter:包 MergeIterator::new(吐所有版本),上层做 ① 快照过滤(seq < snapshot)② 按 user_key 取最新可见版本去重 ③ 墓碑跳过。Engine::scan_as_of(start, end, snapshot) 用它;scan = scan_as_of(.., self.seq)。

Done when: 范围读时间旅行(历史快照扫到当时视图,无视之后写/删);跨 flush 仍正确;scan ≡ get 在每个 key 上一致(随机交错 + 多快照对拍);既有 G/H/J 测试全回归。

分工:MergeIterator 归并,DBIter 懂语义

K 的核心不是新算法,是把三件事放回该在的层。回顾两层各自的职责:

层职责懂什么
MergeIterator(底层)纯 k 路归并,吐所有版本,顺序 = InternalKey 字节序(user_key 升、同 key seq 降)只懂字节序。不去重、不懂 seq、不懂墓碑
DBIter(本层)① 快照过滤 ② user_key 去重 ③ 墓碑跳过懂 seq、懂 type(value/deletion)
为什么去重必须在 DBIter,不能留在 MergeIterator? 去重 = "同一个 user_key 只取最新可见版本"。但"最新可见"依赖快照:在快照 s 下,"最新"是 seq 最大且 < s 的那个版本——不是全局最大。G2 在归并层去重时取的是堆顶(全局最新),这在没有快照时碰巧对;一旦要历史快照,归并层不懂 seq、无法判断"对这个快照哪个版本才算最新"。所以去重要上移到懂 seq 的 DBIter:它在一个 user_key 的整组版本里,跳过 seq >= snapshot 的,取第一个 seq < snapshot 的(组内 seq 降序 → 第一个即最新可见)。

代码 1 — DBIter(db_iter.rs)

    pub struct DBIter<'a> {
        inner: MergeIterator<'a>,
        snapshot: ik::Seq,            // 独占上界:可见 iff version.seq < snapshot
        cur: Option<(Vec<u8>, Vec<u8>)>,  // 当前停留的 (user_key, value),owned
    }

    impl<'a> DBIter<'a> {
        pub fn new(sources: Vec<Box<dyn LsmIterator + 'a>>, snapshot: ik::Seq) -> Self {
            Self { inner: MergeIterator::new(sources), snapshot, cur: None }
        }

        // 从 inner 当前位置起,推进到下一个"可吐"的 user_key(最新可见版本是 VALUE)。
        fn advance(&mut self) {
            while self.inner.valid() {
                let uk = ik::user_key(self.inner.key()).to_vec();
                let mut chosen: Option<(u8, Vec<u8>)> = None;
                // 消费本 user_key 的整组版本(组内 seq 降序)
                while self.inner.valid() && ik::user_key(self.inner.key()) == uk.as_slice() {
                    if chosen.is_none() && ik::seq(self.inner.key()) < self.snapshot {
                        chosen = Some((ik::value_type(self.inner.key()), self.inner.value().to_vec()));
                    }                                          // 第一个 seq
                    self.inner.next();
                }
                match chosen {
                    Some((t, v)) if t == ik::TYPE_VALUE => { self.cur = Some((uk, v)); return; }
                    _ => {}                                    // 墓碑 / 无可见版本 → 跳过此 key
                }
            }
            self.cur = None;                                   // 扫完
        }
    }

    impl LsmIterator for DBIter<'_> {
        fn seek(&mut self, target: &[u8]) {
            // target 是 user_key;lookup_key(.., MAX) trailer 最小 → 落到该 user_key 组首
            self.inner.seek(&ik::lookup_key(target, ik::Seq::MAX));
            self.advance();
        }
        fn next(&mut self) { self.advance(); }  // advance 已推过当前组
        fn valid(&self) -> bool { self.cur.is_some() }
        fn key(&self) -> &[u8] { &self.cur.as_ref().unwrap().0 }   // 吐 user_key,非 InternalKey
        fn value(&self) -> &[u8] { &self.cur.as_ref().unwrap().1 }
    }

为什么 cur 要 owned(Vec<u8>)? 定位"该吐什么"必须把一个 user_key 的整组版本消费完(才能确认组内最新可见、且后续没有更高优先级覆盖)。消费完后 inner 已经移到下一组,原来的 key/value 切片失效——所以要在消费时拷一份存进 cur,key()/value() 才能稳定返回。这是"前瞻整组才能决定当前输出"型游标的通用代价(RocksDB DBIter 同样维护 saved_key_ / saved_value_)。

代码 2 — scan_as_of(engine.rs)

    pub fn scan(&self, start: &[u8], end: &[u8]) -> Vec<(Key, Value)> {
        self.scan_as_of(start, end, self.seq)   // "读最新" = 当前快照
    }

    pub fn scan_as_of(&self, start: &[u8], end: &[u8], snapshot: Seq) -> Vec<(Key, Value)> {
        let mut sources: Vec<Box<dyn LsmIterator + '_>> = Vec::new();
        sources.push(Box::new(self.mem.iter_lsm()));       // 源顺序 = 优先级降序,与 get 一致
        for m in &self.imm { sources.push(Box::new(m.iter_lsm())); }
        for r in &self.ssts { sources.push(Box::new(r.iter())); }
        let mut it = DBIter::new(sources, snapshot);
        it.seek(start);
        let mut out = Vec::new();
        while it.valid() {
            if it.key() >= end { break; }                 // 半开区间 [start, end)
            out.push((Bytes::copy_from_slice(it.key()), Bytes::copy_from_slice(it.value())));
            it.next();
        }
        out
    }

注意 scan 主循环肉眼可见地瘦了:不再手动判 value_type、不再手动 strip user_key——那些都被 DBIter 吞掉了。引擎层只剩"建源 → seek → 收集到 end"。源优先级顺序(mem→imm→ssts)就是去重时的"谁更新":同一 user_key 同 seq 不会跨源出现(seq 全局唯一),但更高优先级源天然排在前,DBIter 取组内第一个可见的即对。

🧰 Coding style:语义归位 · 前瞻型游标 · 主循环越薄越对

  • 把语义放在懂它的那一层:去重/快照/墓碑都依赖 seq 和 type,就该在懂 seq/type 的 DBIter 做,而非归并层。G2 的"在 MergeIterator 去重"是没有快照时的权宜;需求一变(要历史快照),错放的逻辑就拦路。分层的判据是"这段逻辑依赖的信息在哪一层可见"。
  • 前瞻型游标必然要缓存:当"当前该输出什么"要看完一整组才能定,就得把结论拷进 owned 字段,因为底层迭代器已经走过去了。这不是浪费,是这类游标的本质代价。
  • 对外 API 主循环越薄,说明分层越对:scan_as_of 收缩成"建源→seek→收集",所有脏活在 DBIter——主循环的厚薄是分层质量的体感指标。

🧠 System mindset:一次遍历,三重过滤

DBIter 是 LSM 读路径的收口:底层无数版本散在 active/imm/多个 SST,经 MergeIterator 汇成一条有序流,DBIter 一遍走过、就地完成三重过滤(快照 / 去重 / 墓碑),吐出干净的 user-level KV。这套"归并 + 语义过滤分层"几乎是所有 LSM 引擎的统一读路径:

系统底层归并语义层
RocksDB / LevelDBMergingIteratorDBIter(快照 + 去重 + 墓碑)
本项目MergeIteratorDBIter(本课)
Cassandra多 SSTable 归并row resolution(时间戳取最新 + tombstone)

迁移收获:面对"多个有序源、每个源有重复/过期/删除标记"的合并问题(日志聚合、事件溯源、CRDT 收敛、版本化文件系统),通用解都是先归并成一条有序流,再用一个懂语义的游标做一遍过滤,而不是把过滤揉进归并。两层各自简单、各自可测。

🧪 Harness 系统:范围时间旅行三问

三条性质

① 时间旅行:put a,b → 抓 s1 → 覆盖 a、删 b、新增 c。scan 当前 = {a2, c};scan_as_of(s1) = {a1, b1}(覆盖/删除/新增全不可见)。② 跨 flush:抓 s1 后把 v1/v2/墓碑全 flush 进 SST,scan 当前为空(已删),scan_as_of(s1) 仍 = {v1}。③ scan≡get:随机交错 put/delete/freeze/flush,沿途抓多个快照;对每个快照,scan_as_of 全量 == 对每个 key 调 get_as_of 的非空集合。这条把范围读和点读钉死成同一真相。

代码 3 — harness(engine.rs tests)

    #[test] // 范围时间旅行
    fn scan_as_of_time_travel() {
        e.put("a".., "a1"..); e.put("b".., "b1"..);  let s1 = e.snapshot();
        e.put("a".., "a2"..); e.delete("b"..); e.put("c".., "c1"..);
        assert_eq!(e.scan(b"", b"\xff"), vec![("a","a2"), ("c","c1")]);   // 当前
        assert_eq!(e.scan_as_of(b"", b"\xff", s1), vec![("a","a1"), ("b","b1")]); // s1
    }

    #[test] // 跨 flush
    fn scan_as_of_survives_flush() {
        e.put("k".., "v1"..); let s1 = e.snapshot();
        e.put("k".., "v2"..); e.delete("k"..); e.flush();   // v1,v2,墓碑全进 SST
        assert!(e.scan(b"", b"\xff").is_empty());                  // 当前 = 已删
        assert_eq!(e.scan_as_of(b"", b"\xff", s1), vec![("k","v1")]);
    }

    #[test] // scan ≡ get,随机交错 + 多快照对拍
    fn scan_equiv_get_under_snapshot() {
        // 2000 步随机 put/delete/freeze/flush,每 100 步抓一个快照
        for &snap in &snaps {
            let scanned = e.scan_as_of(b"", b"\xff", snap);    // 全量
            for p in 0..30 {                                   // 逐 key 对拍 get_as_of
                assert_eq!(scanned.get(k), e.get_as_of(k, snap));
            }
        }
    }

核心问题(自己答)

K 想透这几个
  1. 为什么去重不能留在 MergeIterator,必须上移到 DBIter?(去重要取"该快照下最新可见版本",这依赖 seq 和 snapshot 的比较;MergeIterator 只懂字节序、不懂 seq,无法判断"对某个历史快照哪个版本算最新"。G2 在归并层去重碰巧对,只因当时永远读全局最新。需求加入快照后,逻辑必须搬到懂 seq 的层。)
  2. DBIter 怎么在一个 user_key 的整组版本里挑中"该快照下最新可见"?(组内版本按 seq 降序排(InternalKey 编码保证)。顺序扫这组,跳过 seq >= snapshot 的(不可见),取第一个 seq < snapshot 的——因为降序,第一个满足的就是 seq 最大的可见版本 = 最新可见。它若是墓碑则整 key 不吐,是 value 则吐。)
  3. 为什么 cur 要存 owned 拷贝,不能返回底层切片?(决定"当前吐什么"要把整组版本消费完,消费完 inner 已走到下一组,底层切片失效。所以在消费时把选中的 (user_key, value) 拷进 cur,key()/value() 才能稳定返回。前瞻整组型游标的固有代价。)
  4. scan 主循环为什么比 G2 薄那么多?(G2 在主循环里手动判 value_type 跳墓碑、手动 strip user_key、还依赖 MergeIterator 去重;K 把这三件全压进 DBIter,主循环只剩 seek + 收集到 end。对外 API 的主循环厚薄,是内部分层是否到位的体感指标——脏活都在该在的层,调用方就干净。)

下一站:Tier 3 — Compaction. 至此引擎"只追加、不回收",旧版本和墓碑永久堆积。Compaction 会归并多个 SST、物理丢弃被遮蔽的旧值与已无快照引用的墓碑,并把"最老活跃快照 seq"当作回收安全水位(早于它且被覆盖的才能丢)。这正是 J/K 反复提到的"快照 ↔ GC 协调"落地之处。
