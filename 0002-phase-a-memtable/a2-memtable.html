STEP A2 · Phase A MemTable

MemTable 点写读

把 A1 的"序"装进一张并发有序表，包出 put / get。这是 RocksDB 写路径的终点，你在 learning-record 0005 读过它。

GOAL

交付 src/memtable.rs：MemTable 包一个 SkipMap<Bytes,Bytes>，put(k,v) 写入或覆盖，get(&[u8]) 返回最新值或 None。

Done when: 模型差分 harness 跑 1000 步随机操作，MemTable 与 BTreeMap oracle 逐步一致。

为什么是跳表（mission 锚）

RocksDB 默认 MemTable 用 InlineSkipList——支持无锁并发读 + 单写者并发写，且天然有序（flush 成 SST 时无需再排序）。你不自己手写跳表（那是另一个 lab），用 crossbeam-skiplist 这个工业级实现当零件，把注意力放在 MemTable 的语义上：覆盖写、点查、有序。

代码 1 — src/memtable.rs（核心）

// src/memtable.rs — 内存写缓冲，LSM 写路径终点
use bytes::Bytes;
use crossbeam_skiplist::SkipMap;
use crate::kv::{Key, Value};

pub struct MemTable {
    // SkipMap：无锁并发跳表。K 用自身 Ord（= A1 的字节序，见核心问题 1）。
    map: SkipMap<Bytes, Bytes>,
}

impl MemTable {
    pub fn new() -> Self {
        Self { map: SkipMap::new() }
    }

    /// 写入或覆盖。同 key 再写 = 新值盖旧值（LSM 的"逻辑更新即追加新版本"）。
    pub fn put(&self, key: Key, value: Value) {
        self.map.insert(key, value);
    }

    /// 点查。返回该 key 的当前值；不存在返回 None。
    pub fn get(&self, key: &[u8]) -> Option<Value> {
        self.map.get(key).map(|e| e.value().clone()) // Bytes clone = O(1) 浅拷贝
    }
}

impl Default for MemTable {
    fn default() -> Self { Self::new() }
}
值得停一秒. get 传 &[u8] 就能查 SkipMap<Bytes,_>——它能编过，但为什么？ map.get(key) 的签名是 get<Q>(&self, &Q) where Bytes: Borrow<Q>, Q: Ord，而标准库给了 impl Borrow<[u8]> for Bytes，于是 Q=[u8] 成立。 这和 HashMap<String,_>.get(&str) 是同一个 Borrow 机制：查找不必拥有 key 的所有权，省掉一次为了查询而做的堆分配。 试试把参数改成 &Bytes 或 Vec<u8>，体会差别——这正是 Rust 把这条性能性质编进类型系统的地方。

Harness 系统 ②：模型差分 (oracle)

为什么这里用模型差分

点写读的 bug 往往是状态依赖的：第 37 次覆盖后某 key 读出旧值。手写表驱动用例覆盖不到这种序列态。 办法：拿一个"显然正确但慢"的参考实现当 oracle（这里是 std::BTreeMap），对两者喂同一串随机操作，每步比对。 任何分叉 = 你的实现有 bug，且失败序列可打印复现。这是 RocksDB db_stress 的核心思想，也是验数据结构最强的可迁移 harness。

代码 2 — 模型差分 harness（接 memtable.rs 末尾）

#[cfg(test)]
mod tests {
    use super::*;
    use std::collections::BTreeMap;

    // 无第三方依赖的确定性伪随机（xorshift）——可复现，失败种子能重跑。
    struct Rng(u64);
    impl Rng {
        fn next(&mut self) -> u64 {
            self.0 ^= self.0 << 13; self.0 ^= self.0 >> 7; self.0 ^= self.0 << 17; self.0
        }
    }

    #[test]
    fn memtable_matches_btreemap_oracle() {
        let mt = MemTable::new();
        let mut model: BTreeMap<Vec<u8>, Vec<u8>> = BTreeMap::new();
        let mut rng = Rng(0x1234_5678);

        for step in 0..1000 {
            let k = format!("k{}", rng.next() % 50).into_bytes(); // 50 个 key，逼出大量覆盖写
            if rng.next() % 4 == 0 {
                // 25% 概率读：两边必须一致
                let got = mt.get(&k).map(|v| v.to_vec());
                assert_eq!(got, model.get(&k).cloned(), "第 {} 步分叉, key={:?}", step, k);
            } else {
                // 75% 概率写：两边同样写
                let v = format!("v{}", rng.next()).into_bytes();
                mt.put(Bytes::from(k.clone()), Bytes::from(v.clone()));
                model.insert(k, v);
            }
        }
    }
}

核心问题（自己答）

想清楚再进 A3
  1. 序从哪来：A1 写了 Comparator trait，可 SkipMap<Bytes,_> 用的是 Bytes 自身的 Ord。这两个序现在一致（都字节序）所以没事——但如果用户要自定义比较器，SkipMap 接不进去。RocksDB 怎么解决的？（提示：它不直接存 user_key，而是存编码进比较信息的 key。这是 Phase 后期 InternalKey 的伏笔，先记住这个张力。）
  2. 覆盖写的真相：map.insert 同 key 直接替换值。但真正的 LSM 不是"原地改"——它是追加一个新版本，旧版本仍在更老的 SST 里。MemTable 这层为什么可以"原地覆盖"而整体仍是 append-only？（提示：MemTable 是最新一层。）
  3. Delete 去哪了：现在没有 delete。LSM 怎么删一个 key 而不去翻所有 SST 找到它？（关键词：墓碑 tombstone。这是 MVP 后引入 InternalKey + ValueType 的动机，先想想朴素 map.remove 为什么不够。）
  4. 并发：put 签名是 &self 不是 &mut self——多线程能同时写。为什么 SkipMap 能允许这样而 BTreeMap 不行？这对应 RocksDB MemTable 的什么并发模型（learning-record 0006/0007）？
