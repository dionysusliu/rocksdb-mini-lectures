Phase A — MemTable

从零手写 LSM 引擎的第一块：内存写缓冲。能写、能查、知道自己多大、会喊"我满了"。

Mission 锚点. 这是 mission 第四段「参考 RocksDB 架构从零实现简化版 LSM 引擎」的起点。 你已读过 RocksDB 写路径源码（learning-record 0004/0005）—— WriteBatch → WAL → MemTable。MemTable 是那条链的终点，也是读路径第一站。 先把它单独跑通，WAL（Phase B）只是在前面加持久化，flush（Phase C）只是把它倒进 SST。
本 Lab 宣称的 GOAL

交付一个 MemTable 类型，满足三条可验证断言：

  1. 正确：任意 put 序列后，get 返回每个 key 的最后写入值；缺失 key 返回 None。
  2. 有序：底层按字节序维护 key 顺序（为后续 flush 成有序 SST、范围扫描铺路）。
  3. 自知：能报告近似内存占用，并在超过阈值时回答 is_full() == true。

Done when: cargo test 全绿，且三类 harness（见下）各自就位。

路线：三个最小步

每步一节课，几十行代码，单独 cargo test 可验。按数据依赖排序——先有 key 的"序"，才能建有序表，才能谈"满"。

本 Lab 的第二主题：harness 设计

你要练的不只是引擎，还有怎么证明引擎是对的。存储引擎的 bug 多是"低概率、状态依赖、静默损坏数据"—— 靠肉眼读输出抓不到。三步分别教一种工业可迁移的 harness 系统：

Harness 系统抓什么 bug本 Lab 用在
表驱动测试
table-driven		 边界/分类遗漏。一份用例表，一段断言逻辑，加用例零成本。RocksDB/Go 标准库主力风格。A1 比较器
模型差分 (oracle)
model-based		 与"显然正确的参考实现"行为分叉。拿 BTreeMap 当 oracle，随机操作序列两边对拍。A2 MemTable
不变量检查
invariant		 违反"任何时刻都该成立"的性质（如大小单调、阈值边界）。RocksDB 满身 assert 即此道。A3 大小/冻结
带着这些问题进 A1（每节课末尾留了核心问题，先别看答案，自己想）： 为什么 RocksDB 把"比较器"抽成可替换接口，而不是写死字节序？一个 Comparator 必须满足哪几条数学性质，违反会怎样？
