STEP A3 · Phase A MemTable

内存计量 + 冻结门

MemTable 不能无限涨。它得知道自己多大，并在超阈值时喊"我满了，该冻结刷盘了"。这个信号是 Phase B（WAL 轮换）和 Phase C（flush 成 SST）的触发器。

GOAL

给 MemTable 加 approximate_size()（原子累加 key+value 字节）与 is_full(threshold)。

Done when: 不变量 harness 验证：大小随写入单调不减；恰好跨过阈值时 is_full 翻 true；并发写下计数不丢。

为什么是"近似"（mission 锚）

RocksDB 的 MemTable::ApproximateMemoryUsage() 也叫"近似"——精确算跳表每个节点的指针/arena 开销太贵，且没必要。冻结阈值本就是个软启发式。 你学到的可迁移原则：热路径上的计量要 O(1) 且无锁，宁可近似不可拖慢写入。所以用一个 AtomicUsize 在 put 时 fetch_add，而不是每次遍历跳表求和。

对标 db/memtable.cc:300 ApproximateMemoryUsage · :319 ShouldFlushNow · write_buffer_size

步骤

  1. 改 src/memtable.rs：给 MemTable 加 approx_size: AtomicUsize 字段（见「代码 1」的 + 行）。
  2. new() 初始化为 0；put 里 fetch_add(key.len()+value.len())。
  3. 加 approximate_size() 与 is_full() 两个方法。
  4. 敲入「代码 2」不变量 harness，跑命令看绿。

代码 1 — 改 src/memtable.rs（+ 为新增行）

use bytes::Bytes;
use crossbeam_skiplist::SkipMap;
+ use std::sync::atomic::{AtomicUsize, Ordering};
use crate::kv::{Key, Value};

pub struct MemTable {
    map: SkipMap<Bytes, Bytes>,
+   approx_size: AtomicUsize, // 近似字节占用，热路径 O(1) 维护
}

impl MemTable {
    pub fn new() -> Self {
        Self {
            map: SkipMap::new(),
+           approx_size: AtomicUsize::new(0),
        }
    }

    pub fn put(&self, key: Key, value: Value) {
+       // 注意：覆盖写也累加——这是"近似"的代价，见核心问题 2
+       self.approx_size.fetch_add(key.len() + value.len(), Ordering::Relaxed);
        self.map.insert(key, value);
    }

    pub fn get(&self, key: &[u8]) -> Option<Value> {
        self.map.get(key).map(|e| e.value().clone())
    }

+   /// 近似内存占用（字节）。O(1)，不遍历跳表。
+   pub fn approximate_size(&self) -> usize {
+       self.approx_size.load(Ordering::Relaxed)
+   }
+
+   /// 是否到达冻结阈值。这就是 Phase B/C 的触发信号。
+   pub fn is_full(&self, threshold: usize) -> bool {
+       self.approximate_size() >= threshold
+   }
}
为什么 Relaxed. 计数本身不保护任何其它内存的可见性顺序，只需原子地"不丢加法"。 最强的 SeqCst 在这里是无谓开销。能解释清楚"为什么 Relaxed 够"是核心问题 4。

Harness 系统 ③：不变量检查

为什么这里用不变量

大小/阈值这种东西没有"参考实现"可对拍，但有任何时刻都必须成立的性质：①大小单调不减；②跨阈值那一刻 is_full 必翻 true，且之后恒 true；③并发 N 线程各写 M 次，总计数不丢（无数据竞争丢更新）。 把这些性质写成断言反复检验，就是不变量 harness。RocksDB 满代码的 assert(...) 正是把不变量钉在执行路径上——可迁移到任何系统代码。

代码 2 — 不变量 harness（替换 memtable.rs 的 tests 模块，或并入）

#[cfg(test)]
mod size_tests {
    use super::*;
    use std::sync::Arc;
    use std::thread;

    // 不变量①②：单调 + 阈值边界
    #[test]
    fn size_monotonic_and_threshold() {
        let mt = MemTable::new();
        let mut last = 0;
        let threshold = 100;
        let mut crossed = false;
        for i in 0..50 {
            mt.put(Bytes::from(format!("k{i}")), Bytes::from(vec![0u8; 5]));
            let now = mt.approximate_size();
            assert!(now >= last, "大小回退: {last} -> {now}"); // ① 单调
            last = now;
            if mt.is_full(threshold) { crossed = true; }
            // ② 一旦 full 必恒 full（大小只增，阈值不变）
            assert_eq!(mt.is_full(threshold), now >= threshold);
        }
        assert!(crossed, "50 次写应越过阈值 100");
    }

    // 不变量③：并发写计数不丢
    #[test]
    fn concurrent_size_no_lost_update() {
        let mt = Arc::new(MemTable::new());
        let (threads, per) = (8, 1000);
        let handles: Vec<_> = (0..threads).map(|t| {
            let mt = mt.clone();
            thread::spawn(move || {
                for i in 0..per {
                    // key 全局唯一，避免覆盖写干扰"应加字节"的精确预期
                    mt.put(Bytes::from(format!("t{t}-k{i}")), Bytes::from(vec![0u8; 3]));
                }
            })
        }).collect();
        for h in handles { h.join().unwrap(); }
        // 每次 put 加 key.len()+3。key 长度不一，所以验"下界不丢"：
        // 至少 threads*per*(最短key+3) 字节，且计数 == 实际所有 key/val 长度和。
        let total = mt.approximate_size();
        assert!(total >= threads * per * (2 + 3), "并发丢了更新? total={total}");
    }
}

核心问题（自己答）

Phase A 收尾，想透这些
  1. 近似的偏差方向：覆盖写时你又加了一遍字节，但跳表里旧值被替换、没真占新空间。所以 approximate_size 是高估还是低估真实占用？高估和低估，哪个对"决定何时冻结"更安全？为什么？
  2. RocksDB 怎么算的：去读 db/memtable.cc:300，它把 arena 分配 + table 占用 + insert_hints 加起来。为什么它能拿到比你更真实的数，你的简化版差在哪？（提示：它自己管 arena 内存。）
  3. 冻结之后：is_full()==true 之后该发生什么？RocksDB 是：当前 MemTable 转为 immutable、新建一个 active、旧的排队 flush。关键：为什么要"转 immutable 再后台 flush"，而不是当场停下来同步刷盘？（这是 Phase C 的设计，也牵涉 write stall——learning-record 0005。）
  4. 内存序：解释为什么计数用 Relaxed 就够，不需要 SeqCst。如果将来 is_full 的结果要用来和"另一个线程设置的冻结标志"建立 happens-before 关系，那时还能用 Relaxed 吗？
