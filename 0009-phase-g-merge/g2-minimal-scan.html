STEP G2 · Phase G 归并

minimal scan — 对外范围扫描雏形

G1 的 MergeIterator 还只认 Vec<Box<dyn LsmIterator>>。G2 把它接上引擎真实的三层源——active memtable、若干 immutable、一摞 SST——切出引擎对外的第一个**范围读 API**:scan(start, end)。这是 LSM "读时归并"第一次以**用户能调的形态**落地。

GOAL

engine.rs 加 scan(&self, start: &[u8], end: &[u8]) -> Vec<(Key, Value)>:把 mem / imm[..] / ssts[..] 按优先级各做一个 LsmIterator,塞进 MergeIterator,seek(start) 起、收到 key >= end 止(半开区间 [start, end))。

Done when: 跨层扫描差分绿——数据散在 active/imm/SST、含覆盖写,engine.scan(a, b) 逐条 == BTreeMap::range(a..b)(取最新值)。

源的顺序 = 优先级,必须和 get 一致

归并要靠源优先级区分新旧(G1 讲过:plain key 分不出谁新)。所以喂进 MergeIterator 的顺序必须是优先级降序,且和 get 的查找顺序同一个:

    // get:  mem → imm(新→旧)→ ssts(新→旧),命中即停
    // scan: 同一个顺序塞进 sources[0..],idx 小 = 优先级高
    sources = [ mem.iter,  imm[0].iter, imm[1].iter, …,  ssts[0].iter, ssts[1].iter, … ]
                ↑最高                                                        最低↑

get 和 scan 走两条代码路径(get 是逐层命中即停、护 Phase R 的 bloom;scan 是归并),但它们对"谁是某个 key 的最新值"必须给出一致答案。这条一致性后面用 get(k) ≡ scan(k..=k).first() 差分锁死(Phase H 起)。同一份优先级约定,喂给两条路径——顺序写错,两条就会分叉。

步骤

  1. engine.rs 引入 MergeIterator / LsmIterator / Bytes;加 scan 方法(「代码 1」)。
  2. 按优先级 mem → imm → ssts 收集 sources,MergeIterator::new,seek(start),循环收到 key >= end(「代码 1」)。
  3. 敲入「代码 2」跨层扫描差分,cargo test scan 全绿。

代码 1 — engine.scan(engine.rs)

    use crate::iterator::LsmIterator;
    use crate::merge::MergeIterator;
    use bytes::Bytes;

    impl Engine {
        /// 范围扫描 [start, end)(plain key,半开区间)。新值遮旧值由源优先级去重保证。
        pub fn scan(&self, start: &[u8], end: &[u8]) -> Vec<(Key, Value)> {
            // 源顺序 = 优先级降序,与 get 的 mem→imm→ssts 一致
            let mut sources: Vec<Box<dyn LsmIterator + '_>> = Vec::new();
            sources.push(Box::new(self.mem.iter_lsm()));
            for m in &self.imm { sources.push(Box::new(m.iter_lsm())); }
            for r in &self.ssts { sources.push(Box::new(r.iter())); }

            let mut it = MergeIterator::new(sources);
            it.seek(start);
            let mut out = Vec::new();
            while it.valid() && it.key() < end {       // 半开:>= end 即停
                out.push((Bytes::copy_from_slice(it.key()),
                           Bytes::copy_from_slice(it.value())));
                it.next();
            }
            out
        }
    }

生命周期在这兑现。 sources 里每个 iter 都借 &self(mem/imm/ssts):MemTableIter<'_> 借跳表(F3 那个 'a)、SstIter<'_> 借 reader。MergeIterator<'_> 把这些 '_ 统一成 &self 的生命周期。最后 copy_from_slice 把结果拷成 owned Bytes——这样返回的 Vec 不再借 self,函数能干净返回。Bytes::copy_from_slice 这里是真拷(出借用边界),不是引用计数。

为什么先 collect 成 Vec,而不返回一个 scan 迭代器? 返回 impl LsmIterator + '_ 也行、还更惰性,但它会**借住 &self 整段扫描期间**——调用方拿着这个迭代器时不能再 &mut self(put/flush)。G2 选 Vec(物化、立刻释放借用)是 MVP 取舍:简单、够用、避开生命周期外溢。Phase K 做对外 scan 迭代器(seek/next 流式、快照+墓碑感知)时再正面处理这个借用 / 快照 pin 问题。

🧰 Coding style:按优先级建源 · 物化释放借用 · 半开区间

  • 源按优先级建,和 get 同序:把"谁更新"的约定只写一处(mem→imm→ssts)、两条读路径共用。约定散成两份迟早分叉;集中成一个顺序,改一处全改。
  • 物化释放借用:copy_from_slice 把借 self 的切片拷成 owned,返回值与 self 脱钩。"出借用边界就拷贝"是 Rust 把内部借用挡在 API 边界内的常规手法(对照 F2 SstIter 内部零拷贝、点查边界才 copy)。
  • 半开区间 [start, end):key() < end 停。半开是范围 API 的事实标准(BTreeMap::range、std::ops::Range、SQL 之外几乎都是)——拼接区间不重不漏([a,b) + [b,c) = [a,c)),空区间 [x,x) 自然为空。

🧠 System mindset:scan = 读时归并的对外面孔

到这一步,LSM 的读路径骨架齐了:点查(get,逐层命中即停)和范围读(scan,归并所有层)。两者都建在"数据分层散落、读时合一"之上。scan 就是 RocksDB DBIter 的雏形——一个跨整个 LSM、对用户呈现"一张有序逻辑表"的游标。它把底下 active/imm/L0/L1… 几十个物理片段的复杂度,收敛成一条干净的有序流。

迁移收获:任何"底层分片 + 对外统一视图"的系统都有这层。难点不在归并本身,在边界语义:① 区间开闭(半开);② 快照隔离(扫描期间并发写可见吗?——Phase J/K);③ 删除(被删的 key 怎么从扫描结果里消失?——墓碑,Phase H3/K)。G2 把骨架立起来,这三件语义留给后续逐个补——先正确地"扫出存在的东西",再正确地"扫出该看见的版本"。

🧪 Harness 系统:跨层扫描差分 vs BTreeMap::range

把数据故意摊到三层

G1 用 VecIter 在内存里验归并逻辑;G2 要验引擎真把三层源接对了。手法:写一批 → flush(进 SST)→ 覆盖偶数 key → freeze(进 imm)→ 再写一批留 active。于是同一个 key 的新旧版本跨层分布,只有"源优先级 + 归并"都对,scan 才能逐条等于 BTreeMap::range。多个区间(头/中/尾/全量)各打一遍,抓:漏层、层序反、区间边界错(把 end 也吐了)。这条差分到 Phase K 升级成"快照 + 墓碑感知"的扫描差分。

代码 2 — 跨层扫描差分(engine.rs tests)

    #[test]
    fn scan_matches_btreemap_range() {
        let dir = tempdir().unwrap();
        let mut e = Engine::open(dir.path()).unwrap();
        let mut model: BTreeMap<Vec<u8>, Vec<u8>> = BTreeMap::new();

        for i in 0..40u32 {                          // 一批 → flush 进 SST
            let (k, v) = (format!("k{i:03}"), format!("old{i}"));
            e.put(k.clone().into(), v.clone().into()).unwrap();
            model.insert(k.into_bytes(), v.into_bytes());
        }
        e.flush().unwrap();
        for i in (0..40u32).step_by(2) {               // 覆盖偶数 key → freeze 进 imm
            let (k, v) = (format!("k{i:03}"), format!("new{i}"));
            e.put(k.clone().into(), v.clone().into()).unwrap();
            model.insert(k.into_bytes(), v.into_bytes());
        }
        e.freeze();
        for i in 50..60u32 {                         // active 独有
            let (k, v) = (format!("k{i:03}"), format!("act{i}"));
            e.put(k.clone().into(), v.clone().into()).unwrap();
            model.insert(k.into_bytes(), v.into_bytes());
        }

        for (a, b) in [(&b"k000"[..], &b"k010"[..]), (&b"k005"[..], &b"k055"[..]),
                       (&b"k038"[..], &b"k052"[..]), (&b""[..], &b"\xff"[..])] {
            let got = e.scan(a, b);
            let want: Vec<(Bytes, Bytes)> = model.range(a.to_vec()..b.to_vec())
                .map(|(k, v)| (Bytes::copy_from_slice(k), Bytes::copy_from_slice(v))).collect();
            assert_eq!(got, want, "scan 与 BTreeMap::range 分叉");
        }
    }

核心问题(自己答)

G2 / Phase G 收尾
  1. 为什么 scan 物化成 Vec、get 却能逐层命中即停?(get 找一个 key,逐层问、命中即返回,大多数情况只碰最上层 → 快、且能用 bloom 跳过整层(Phase R)。scan 要一个范围的全部,绕不开把每层都开游标归并。点查和范围读的最优策略天生不同,所以 RocksDB 也保留两条路径,而非让 get 走 scan。)
  2. 源顺序写反会怎样?(若把 ssts 排在 mem 前面,同 key 的旧 SST 值优先级变高 → scan 吐旧值。和 get 的 mem→imm→ssts 一比就分叉。源优先级是个必须两路一致的全局约定,这也是为什么把它收敛成一处。)
  3. 删除的 key 现在会被 scan 扫出来吗?(会!现在没有删除——没墓碑、没 delete API。要让"删过的 key 在 scan 里消失",得 Phase H3 引入 kTypeDeletion 墓碑:删 = 写一条墓碑,归并时墓碑遮蔽同 key 旧值、且自己不吐。想清楚:删除在 LSM 里也是一次写,不是原地抹除。)
  4. 扫描期间并发 put,scan 看得到吗?(G2 是单线程同步引擎,scan 跑完才返回,问题不显。但语义上:MemTableIter 是活游标(F3),会看到扫描中途的写 → 弱一致。真正要"扫一个一致快照"得 Phase J 的 snapshot seq——按 seq 过滤,只看 ≤ 快照号的版本。G2 先不管,K 补齐。)

下一步:Phase H — InternalKey(user_key + seq + type). 给每个 key 编上序列号 seq 和类型 type(Put/Delete)。seq 让"新旧"从"靠源顺序猜"变成"key 自带、比较器可读";type 让"删除"成为一种可归并的写(墓碑)。H3 会重写 MergeIterator 的比较器(plain [u8]::cmp → InternalKeyComparator)——本课刻意把比较规则外置,就是为这次重写铺路。
