STEP G1 · Phase G 归并

MergeIterator — k 路归并

Phase F 把三种数据源(block / SST / memtable)全装成同一个 LsmIterator。零件齐了,现在组装:把 N 个各自有序的源,合成一条全局有序的流。这就是 LSM 的灵魂——读时归并(merge-on-read):数据散在 active memtable、若干 immutable、一摞 SST 里,谁也不动谁,读的时候按 key 归并、新值盖旧值。本课只做纯归并(plain key);快照与墓碑感知留到 H/K。

GOAL

新建 merge.rs:MergeIterator<'a> 实现 LsmIterator,内部只有一个 min-heap,堆元素 HeapItem 直接拥有迭代器(Box<dyn LsmIterator + 'a>)+ 优先级 idx。源按优先级传入(idx 小 = 高);同 key 多源出现时,只吐最高优先级源的值、其余推进跳过(源优先级去重)。lib.rs 加 pub mod merge;。

Done when: 归并差分绿——full scan == 按优先级合并后的 BTreeMap;同 key 吐最高优先级值且只一次;seek(t) 跨源落第一个 ≥t;空源集 invalid。零 key 拷贝(堆拥有源、比较读活 key)。

为什么是「看齐所有头 → 取最小 → 只推一个」(F1 的解耦在这兑现)

k 路归并的核心动作:看齐 N 个源的当前头 → 比出最小 key → 只推进出最小的那个源,其余源纹丝不动、下一轮还要再被看一次。这正是 F1 反复强调的"窥视(valid/key/value)与前进(next)解耦"——语言自带的 Iterator::next() 把"看 + 进"捆死,根本写不出归并;我们的 LsmIterator 把它俩拆开,归并才成立。这一课是 Phase F 全部铺垫的兑现点。

min-heap:堆里存什么?(这次让堆直接拥有源)

"看齐 N 个头取最小"最朴素是每步线性扫 N 个源(O(N))。用 min-heap 把每步降到 O(log N):堆顶永远是当前最小。但堆里存什么有讲究——三个选项:

  • ❌ 存"指向源的引用"(堆借 sources 兄弟字段)→ 自引用借用,编译器拒(F3 刚踩过)。
  • 🅱️ 存 (key 的拷贝, idx),源另放 sources Vec —— 堆不借源、无自引用,但每次推进拷一次 key 进堆(to_vec())。这是最朴素的破法。
  • ✅ 让堆直接拥有迭代器:堆元素 HeapItem { idx, iter: Box<dyn LsmIterator> } 把源 move 进堆。没有兄弟字段可借 → 根本不存在自引用;比较时读 iter.key() 的活 key → 零拷贝。推进靠 pop 出堆顶 → next() → 压回。

这是本课相对旧版的关键改动:struct 从「sources + heap 两字段」塌成「只剩 heap」。所有权不再分裂,to_vec() 消失。

但"堆独占所有权"逼出两个新约束:
① 自定义 Ord 把大顶堆当小顶用:std 只有大顶 BinaryHeap,给 HeapItem 写 Ord——主序 key 升、key 同则 idx 升,最后 .reverse() → 最小 (key, idx) 落堆顶。idx 即优先级,"同 key 谁赢"被堆序顺手解决。
② 失效源必须沉底、绝不丢:堆是唯一所有者,seek 要重定位所有源,所以耗尽的源 next() 后照样压回,不能丢(丢了 seek 捞不回)。它没 key,Ord 里用 valid() 分支让它恒沉堆底——绝不上顶污染比较。于是 valid() 也得改判「堆顶是否 valid」,而非「堆非空」。
源优先级 = plain key 阶段区分新旧的唯一手段。 同一个 key 写了两次(memtable 新值 + SST 旧值),plain key 字节完全一样、分不出谁新。只能靠"源的排列顺序"约定:active memtable > immutable > 新 SST > 旧 SST(引擎 get 的 mem → imm → ssts 同序)。所以 idx=0(传入 sources 的第 0 个)必须是最新的源。Phase H3 把 seq 编进 key、比较器换成 InternalKeyComparator 后,新旧就由 seq 直接定序,不再单靠源顺序——那次"比较器重写"是刻意的教学点。

步骤

  1. 新建 merge.rs,定义 HeapItem<'a>(持 idx + iter)及其 Ord(活 key + idx + invalid 沉底),再定义 MergeIterator<'a>(唯一字段 heap)与 new(「代码 1」)。lib.rs 加 pub mod merge;。
  2. 实现 LsmIterator:seek* 把全员 drain 出堆、各自 seek、重建堆;next 用 pop 锚 + 同 key 全推进 + 压回;key/value/valid 走堆顶(「代码 2」)。
  3. 敲入「代码 3」归并差分 + 去重 + seek harness,cargo test 全绿。

代码 1 — HeapItem + Ord + MergeIterator 结构(merge.rs)

    use std::cmp::Ordering;
    use std::collections::BinaryHeap;
    use crate::iterator::LsmIterator;

    // 堆元素:idx(优先级,小=高) + 迭代器本身。堆直接持有源 → struct 无兄弟字段 → 无自引用。
    struct HeapItem<'a> {
        idx: usize,
        iter: Box<dyn LsmIterator + 'a>,
    }
    // Ord 读"活 key":主序 key 升、key 同则 idx 升,末尾 .reverse() → 喂大顶堆得小顶语义。
    // invalid 源没 key:用 valid() 分支让它恒"最小",沉到堆底、绝不上顶。seek 还能把它捞回重定位。
    impl<'a> Ord for HeapItem<'a> {
        fn cmp(&self, o: &Self) -> Ordering {
            match (self.iter.valid(), o.iter.valid()) {
                (true, true)  => self.iter.key().cmp(o.iter.key())
                                    .then(self.idx.cmp(&o.idx)).reverse(),
                (true, false) => Ordering::Greater,   // valid 上浮(大顶堆顶)
                (false, true) => Ordering::Less,
                (false, false)=> Ordering::Equal,
            }
        }
    }
    impl<'a> PartialOrd for HeapItem<'a> {
        fn partial_cmp(&self, o: &Self) -> Option<Ordering> { Some(self.cmp(o)) }
    }
    impl<'a> PartialEq for HeapItem<'a> {
        fn eq(&self, o: &Self) -> bool { self.cmp(o) == Ordering::Equal }
    }
    impl<'a> Eq for HeapItem<'a> {}

    pub struct MergeIterator<'a> {
        heap: BinaryHeap<HeapItem<'a>>,   // 唯一字段:堆拥有全部源(含已耗尽的,沉底待 seek 复用)
    }
    impl<'a> MergeIterator<'a> {
        /// sources 按优先级降序:第 0 个最高(同 key 时它赢)。idx 即优先级。
        pub fn new(sources: Vec<Box<dyn LsmIterator + 'a>>) -> Self {
            let heap = sources.into_iter().enumerate()
                .map(|(idx, iter)| HeapItem { idx, iter })
                .collect();                   // collect 进 BinaryHeap 即建堆
            Self { heap }
        }
    }

代码 2 — 实现 LsmIterator(merge.rs)

    impl LsmIterator for MergeIterator<'_> {
        fn seek_to_first(&mut self) {
            // 堆是唯一所有者:全员 drain 出来 → 各自 seek → 重建堆(Ord 按新位置重排)。
            let mut items: Vec<_> = self.heap.drain().collect();
            for it in &mut items { it.iter.seek_to_first(); }
            self.heap = items.into_iter().collect();
        }
        fn seek(&mut self, target: &[u8]) {
            let mut items: Vec<_> = self.heap.drain().collect();
            for it in &mut items { it.iter.seek(target); }
            self.heap = items.into_iter().collect();
        }
        fn next(&mut self) {
            // 弹出胜者但先不推进 —— 它当"活参照锚",免去 cur 的 to_vec()。
            let mut top = match self.heap.pop() {
                Some(t) if t.iter.valid() => t,
                other => { if let Some(t) = other { self.heap.push(t); } return; }
            };
            // 推进所有"== 锚 key"的低优先级源(被遮蔽的旧值),否则下一轮重复冒出。
            loop {
                let same = match self.heap.peek() {
                    Some(p) => p.iter.valid() && p.iter.key() == top.iter.key(),
                    None => false,
                };
                if !same { break; }
                let mut loser = self.heap.pop().unwrap();
                loser.iter.next();
                self.heap.push(loser);          // 失效也压回:Ord 让它沉底,供后续 seek 复用
            }
            top.iter.next();
            self.heap.push(top);                // 锚最后推进、压回
        }
        fn valid(&self) -> bool {
            self.heap.peek().map_or(false, |h| h.iter.valid())   // 非空≠有效:全沉底也算 invalid
        }
        fn key(&self)   -> &[u8] { self.heap.peek().unwrap().iter.key() }
        fn value(&self) -> &[u8] { self.heap.peek().unwrap().iter.value() }
    }

key()/value() 走 heap.peek() 取堆顶迭代器的活值,零拷贝。next 里 top 一旦 pop 出堆就不再借 heap,所以循环中能自由 pop/push 改堆;且 top 全程持有、最后才推进 → top.iter.key() 始终是合法参照,连旧版那次 cur.clone() 都省了。same 用 match 算出布尔后 peek 的借用立即结束(NLL),才轮到 pop——避开"持 peek 借用又改堆"的冲突。

去重的关键:next 要推进所有同 key 源,不只堆顶。 若只推堆顶(胜者),同 key 的低优先级旧值还赖在堆里,下一轮就会作为"新的最小"被吐出来——同一个逻辑 key 吐两次。所以 while heap.peek().key == cur 把它们全 pop+推进+回填。这正是"覆盖写的旧值被新值遮蔽"在归并层的实现。

🧰 Coding style:堆直接拥有源 · 自定义 Ord(活 key + invalid 沉底)· pop+push 而非 peek_mut

  • 让排序结构直接拥有元素 —— 破自引用第四招:前三招都在"不让排序结构碰所有权"(F1 Arc<Block> 引用计数、F3 借外部 map、旧版"存副本")。这里反过来:把迭代器 move 进 HeapItem,堆就是源的唯一所有者——没有兄弟字段可借,自引用从根上不存在,to_vec() 也随之消失。Rust 破自引用全集:借外部 / 引用计数 / 存副本 / 直接拥有。
  • 自定义 Ord 把大顶堆当小顶 + invalid 沉底:std 只有大顶 BinaryHeap;给 HeapItem 写 Ord,主序 key、次序 idx、末尾 .reverse() → 小顶语义,"同 key 谁赢"顺手解决。额外一笔:invalid 源没 key,用 valid() 分支让它恒沉底——这是"堆独占所有权"的必然代价(耗尽的源不能丢,得留着给 seek 捞回)。
  • pop+push 而非 peek_mut:弹出胜者→推进→压回。比 peek_mut 多一次 sift(可忽略),但躲掉 PeekMut 的 Drop-resift 访问失效迭代器的坑(skyzh 文档专门警告"别在 PeekMut 作用域里用 ?")。所有权故事更直白:弹出后 top 不借堆,改堆随意。
  • idx 即优先级:不另存优先级字段,HeapItem.idx(传入顺序)直接承载;Ord 的次序把"同 key 选最高优先级"编进堆序,next 只管"推进所有同 key 源"。把约定编进数据布局,省掉一段判断逻辑。

免这次 copy:三种真实做法对照

系统堆里放什么key 副本比较代价
本课(dyn)HeapItem 拥有 Box<dyn>,Ord 读活 key零每次比较走 vtable(O(log N)/步)
RocksDBHeapItem* 指针,比较器 deref 活 key每个子迭代器缓存 1 份(IteratorWrapper,每 advance 刷新)读连续内存,免 vtable + cache 友好
skyzh泛型 HeapWrapper<I> 拥有迭代器零单态化 inline,无 vtable

反直觉点:在 dyn 下"免 copy"不是纯赚。copy 没了,但每次比较多一次 vtable 调 key()。RocksDB 实测后故意把 copy 当缓存请回来——每个子迭代器存一份 key(注意:是 per-child、每 advance 刷新一次,不是 per-heap-node),为的是免 vtable + cache 局部性。真正的零 copy 要靠静态分发(skyzh 的泛型塔,单态化让 key() inline)——那是 Phase F→G 的大重构,本课用 Box<dyn> 拿"零 copy + 一点 vtable"的折中,N=几层时 vtable 可忽略。

🧠 System mindset:读时归并 = LSM 的灵魂 · k 路归并无处不在

LSM 树的核心权衡:写时只追加(快),读时才归并(把代价后移)。数据按时间分层散落(memtable→L0→L1…),从不原地改;读一个范围 = 把所有层的有序片段归并成一条逻辑流,新值遮旧值。MergeIterator 就是这台归并机。它和一大票系统同构:

领域k 路归并长这样
外部排序数据大于内存 → 切成有序 run 落盘 → k 路归并成最终序。归并排序的 merge 步就是 2 路特例。
MapReduce / Sparkshuffle 后每个 reducer 把多个 mapper 的有序分片归并成一条 key 有序流再 reduce。
日志/时序系统多个有序日志段(按时间)归并成全局时间线;Kafka compaction、Prometheus TSDB block merge 同理。
LSM(本课 ★)读时归并 active+imm+SST;compaction(Phase O)也是归并——只是它把结果写回盘、顺手丢被遮蔽/墓碑数据。

迁移收获:只要你有"多个各自有序的源,要合成一条全局有序流",答案几乎总是 min-heap k 路归并:堆顶取最小、推进它、回填。难点从来不在堆,而在同 key 怎么定胜负——本课用源优先级,数据库用版本号(seq/MVCC),CRDT 用因果序。"归并"解决顺序,"优先级/版本"解决冲突,两件事要分开想。

🧪 Harness 系统:归并差分 vs 合并后的 BTreeMap(Tier 2 核心回归门)

为什么用 BTreeMap 当 oracle

归并的正确性 = "full scan 吐出的序列,等于把所有源按优先级合并进一个 BTreeMap 后的有序遍历"。oracle 怎么建:按优先级从高到低把每个源灌进 BTreeMap,entry().or_insert()——已存在就不覆盖(高优先级先写、占坑),完美复刻"源优先级去重"。BTreeMap 显然正确、有序,拿它差分能抓:漏 key、重复 key(去重没推全)、错序(堆比较错)、选错胜者(优先级反了)。这条差分到 Phase O(compaction)、Phase K(对外 scan)会一路复用,是 Tier 2 最核心的回归门。

代码 3 — harness(merge.rs tests)

    // 测试用:有序 Vec 上的 LsmIterator(隔离归并逻辑,不牵扯 memtable/SST 生命周期)。
    struct VecIter { data: Vec<(Vec<u8>, Vec<u8>)>, idx: usize }
    impl VecIter {
        fn boxed(pairs: &[(&str, &str)]) -> Box<dyn LsmIterator> {
            let mut data: Vec<_> = pairs.iter()
                .map(|(k,v)| (k.as_bytes().to_vec(), v.as_bytes().to_vec())).collect();
            data.sort();
            Box::new(VecIter { data, idx: 0 })
        }
    }
    impl LsmIterator for VecIter {
        fn seek_to_first(&mut self) { self.idx = 0; }
        fn seek(&mut self, t: &[u8]) { self.idx = self.data.partition_point(|(k,_)| k.as_slice() < t); }
        fn next(&mut self) { self.idx += 1; }
        fn valid(&self) -> bool { self.idx < self.data.len() }
        fn key(&self) -> &[u8] { &self.data[self.idx].0 }
        fn value(&self) -> &[u8] { &self.data[self.idx].1 }
    }

    // oracle:按优先级把源灌进 BTreeMap,低 idx 先写、or_insert 不覆盖(高优先级赢)。
    fn oracle(sources: &[&[(&str, &str)]]) -> Vec<(Vec<u8>, Vec<u8>)> {
        let mut m = BTreeMap::new();
        for src in sources {
            for (k, v) in *src {
                m.entry(k.as_bytes().to_vec()).or_insert_with(|| v.as_bytes().to_vec());
            }
        }
        m.into_iter().collect()
    }

    #[test] // 归并差分:full scan == 按优先级合并后的 BTreeMap
    fn merge_full_scan_matches_oracle() {
        let s0: &[(&str,&str)] = &[("k01","a1"),("k03","a3"),("k05","a5")];
        let s1: &[(&str,&str)] = &[("k02","b2"),("k03","b3"),("k06","b6")]; // k03 撞 s0
        let s2: &[(&str,&str)] = &[("k00","c0"),("k05","c5"),("k07","c7")]; // k05 撞 s0
        let mut it = MergeIterator::new(vec![VecIter::boxed(s0), VecIter::boxed(s1), VecIter::boxed(s2)]);
        it.seek_to_first();
        assert_eq!(collect(&mut it), oracle(&[s0, s1, s2]));
    }

    #[test] // 源优先级:k03 在 s0/s1 都有 → 吐 s0 的 WIN,只一次
    fn source_priority_wins_and_dedups() {
        let s0: &[(&str,&str)] = &[("k03","WIN")];
        let s1: &[(&str,&str)] = &[("k03","lose"),("k04","b4")];
        let mut it = MergeIterator::new(vec![VecIter::boxed(s0), VecIter::boxed(s1)]);
        it.seek_to_first();
        assert_eq!(collect(&mut it),
            vec![(b"k03".to_vec(), b"WIN".to_vec()), (b"k04".to_vec(), b"b4".to_vec())]);
    }

    #[test] // seek 跨源:命中 / 缝隙 / 超界
    fn merge_seek_lower_bound_across_sources() {
        let s0: &[(&str,&str)] = &[("k01","a1"),("k04","a4")];
        let s1: &[(&str,&str)] = &[("k02","b2"),("k05","b5")];
        let mut it = MergeIterator::new(vec![VecIter::boxed(s0), VecIter::boxed(s1)]);
        it.seek(b"k02"); assert_eq!(it.key(), b"k02");   // 命中(s1)
        it.seek(b"k03"); assert_eq!(it.key(), b"k04");   // 缝隙 → 下一个(s0)
        it.seek(b"k99"); assert!(!it.valid());          // 超界
    }

核心问题(自己答)

G1 想透这几个
  1. 为什么 next 要推进所有同 key 源,不只堆顶?(若只推堆顶胜者,同 key 的低优先级旧值还在堆里 → 下一轮作为"新最小"被吐 → 同一逻辑 key 吐两次。去重的本质 = 把被遮蔽的旧值一并跳过。)
  2. 堆为什么直接拥有迭代器,而不是"存 key 副本 + sources Vec"?代价又是什么?(让 HeapItem move-own 迭代器,堆成唯一所有者 → 无兄弟字段可借 → 自引用从根上消失、to_vec() 也消失。这是破自引用第四招"直接拥有"(前三招:F1 Arc、F3 借外部 map、旧版存副本)。代价两笔:① 失效源不能丢,得靠 Ord 让它沉底留给 seek 复用;② 每次比较走 vtable 调 key()。后者正是 RocksDB 宁可给每个子迭代器缓存一份 key(IteratorWrapper,per-child、每 advance 刷新)来免掉的——所以"免 copy"在 dyn 下是拿 copy 换了 vtable;真零 copy 要静态分发,如 skyzh 的泛型塔。)
  3. plain key 阶段,凭什么断定"第 0 个源(idx=0)的值更新"?(断不出来——plain key 字节一样、无新旧信息。只能靠调用方按约定排好源顺序(引擎的 mem→imm→ssts)。这是个脆弱的外部约定:排错了就读到旧值。Phase H3 把 seq 编进 key,新旧变成 key 自带的、比较器能读的信息,约定就长进数据里、不再靠调用方自觉。想清楚:这正是 A1 埋下的"plain key 不够"张力的兑现。)
  4. heap vs 线性扫,差在哪?(线性扫每步 O(N) 找最小;堆每步 O(log N)。N 小(几层)时绝对差距小,但 compaction 归并几十上百个 SST 时堆才显威。我们选堆是为贴 RocksDB 且让 Phase O 直接复用。反过来想:堆的代价是每步 O(log N) 次 Ord 比较(本课每次还是一次 vtable key())+ pop/push 维护——N 极小时,线性扫可能更快(无堆维护、cache 友好)。没有绝对优解,看 N 的规模。)
  5. 失效的源为什么不能直接丢、要沉底?(堆是源的唯一所有者,而 seek/seek_to_first 要重定位所有源。若 next 把耗尽的源丢出堆,后续 seek 就再也捞不回它 → 漏源。所以失效也 push 回,靠 Ord 的 valid() 分支让它沉到堆底、绝不上顶污染比较。这也逼着 valid() 改判"堆顶是否 valid"而非"堆是否非空"。)
