Phase S — block cache(LRU)

Phase E 的基准反复点名同一个洞:热块被重复读盘、重复解压。E3 看到块化在热点查上输给"整文件常驻"的扁平实现;E5 加了 lz4 后,热点查更慢(每次 get 都解一整块)。Phase S 把这个洞补上——一个用户态 LRU block cache,缓存解析好的 Block:命中则跳过 pread + 解压 + 解块,直接复用。压缩省的磁盘(E5)+ 缓存抹平的解压(S)= 既小又快。对标 RocksDB lru_cache.cc / block_based_table_reader 的 block_cache。

这是被基准牵引出来的 phase. 我们没有"按路线图顺序"提前做它,而是 E3/E5 的诚实数字逼着我们做:Q=100k 冷读里 block 搬了 410MB(>文件本身)、热读因 lz4 涨到 2867ns——两处都写着"缺一个 cache"。这正是测量驱动优化的样子:先量化痛,再针对性补。
GOAL

新建 block_cache.rs:一个 O(1) 的 LRU(HashMap + 侵入式双向链表),key = (file_id, offset) 打包成 u128,value = Arc<Block>;带 hits/misses 计数。改 SstReader:加 open_with_cache(path, file_id, cache),read_block 命中即复用 Arc<Block>、未命中才读盘装填。引擎 / 既有 open 路径不变(cache 可选)。

Done when: hit_miss_counting + lru_evicts_least_recently_used(触碰改变淘汰对象)绿;引擎 + Phase E 既有测试全过;跑 bench 的 Phase S 段:满驻 cache 命中率 >95%、热读延迟掉到 flat 之下,受限 cache 命中率坍塌(诚实显示淘汰开销)。

read_block:cache 命中跳过整条重活(数据流)

为什么缓存"解析好的 Block"而不是"原始字节"? OS page cache 已经替我们缓存了磁盘字节(热块第二次 pread 往往不真打盘)。我们要省的是字节之上的工:lz4 解压 + 解 restarts/offsets 数组——这部分 OS 帮不了,只能在用户态缓存解码产物。这就是 RocksDB block_cache 与 OS page cache 的分工:前者缓存"可直接查的 Block 对象",后者缓存"裸文件页"。E5 加了压缩后,这层用户态缓存的价值陡增(解压很贵)。

cache/lru_cache.cc(LRUCacheShard:侵入链表 + 引用计数)· table/block_based/block_based_table_reader.cc(MaybeReadBlockAndLoadToCache · key = cache_key_prefix + offset)

代码 1 — block_cache.rs(O(1) LRU)

// key = (file_id, offset) 打包成 Copy 的 u128;value = Arc<Block>。
pub fn cache_key(file_id: u64, offset: u64) -> u128 {
    ((file_id as u128) << 64) | (offset as u128)
}

struct Entry { block: Arc<Block>, prev: Option<u128>, next: Option<u128> }
struct Inner { map: HashMap<u128, Entry>, head: Option<u128>, tail: Option<u128>, cap: usize }

impl Inner {
    fn unlink(&mut self, k: u128) {           // 把 k 从链表摘下
        let (prev, next) = { let e = &self.map[&k]; (e.prev, e.next) };
        match prev { Some(p) => self.map.get_mut(&p).unwrap().next = next, None => self.head = next }
        match next { Some(n) => self.map.get_mut(&n).unwrap().prev = prev, None => self.tail = prev }
    }
    fn push_front(&mut self, k: u128) {       // 接到链表头(最近用)
        let old = self.head;
        if let Some(h) = old { self.map.get_mut(&h).unwrap().prev = Some(k); }
        { let e = self.map.get_mut(&k).unwrap(); e.prev = None; e.next = old; }
        self.head = Some(k);
        if self.tail.is_none() { self.tail = Some(k); }
    }
}

pub struct BlockCache { inner: Mutex<Inner>, hits: AtomicU64, misses: AtomicU64 }

impl BlockCache {
    pub fn get(&self, k: u128) -> Option<Arc<Block>> {
        let mut g = self.inner.lock().unwrap();
        if !g.map.contains_key(&k) { self.misses.fetch_add(1, Relaxed); return None; }
        g.unlink(k); g.push_front(k);          // 命中 → 提到最近用
        self.hits.fetch_add(1, Relaxed);
        Some(g.map[&k].block.clone())            // Arc 克隆,廉价
    }
    pub fn put(&self, k: u128, block: Arc<Block>) {
        let mut g = self.inner.lock().unwrap();
        if g.map.contains_key(&k) { /* 刷新位置 + 替换 */ return; }
        g.map.insert(k, Entry { block, prev: None, next: None });
        g.push_front(k);
        if g.map.len() > g.cap {                  // 超容量 → 淘汰 tail(最久未用)
            if let Some(victim) = g.tail { g.unlink(victim); g.map.remove(&victim); }
        }
    }
}
侵入式链表:key 当指针。 Rust 里把节点用 Box/裸指针串成链表会跟借用检查器硬碰硬。技巧:不存指针,存 key——prev/next: Option<u128>,要"跟着指针走"就拿 key 回 HashMap 查。unlink/push_front 全程只改 HashMap 里的字段,无裸指针、无 unsafe,却保住 O(1)。这是 Rust 写链式结构的常用绕法(slotmap/index-as-pointer 同源)。代价:每次"跟指针"多一次哈希查找——可接受。

pub struct SstReader {
    // …file, index, blocks_read, bytes_read…
    file_id: u64,                     // cache key 前半段
    cache: Option<Arc<BlockCache>>,   // None = 直读(引擎/既有路径)

impl SstReader {
    pub fn open(path: &Path) -> Result<Self> { Self::open_inner(path, false, 0, None) }
    pub fn open_with_cache(path: &Path, file_id: u64, cache: Arc<BlockCache>) -> Result<Self> {
        Self::open_inner(path, false, file_id, Some(cache))
    }

    fn read_block(&self, h: BlockHandle) -> Result<Arc<Block>> {
        if let Some(cache) = &self.cache {
            let key = cache_key(self.file_id, h.offset);
            if let Some(b) = cache.get(key) { return Ok(b); }  // 命中:0 IO 0 解压
            let block = Arc::new(self.read_block_uncached(h)?);
            cache.put(key, block.clone());
            return Ok(block);
        }
        Ok(Arc::new(self.read_block_uncached(h)?))
    }

blocks_read/bytes_read 只在 read_block_uncached(真读盘)里自增。所以命中不计入读放大——cache 生效时这俩计数器停止增长,正是它有效的铁证。

实测:cache 让块化在热点查上反超 flat

同一组 10 万次热点 readrandom,先 warmup 一遍再计时:

配置热读延迟命中率读这一行
flat(整文件常驻,参照)~1260 ns/op—纯内存二分 + 偏移,无 IO 无解压
block 无 cache~2639 ns/op—每次 pread + lz4 解压 + 解块
block + cache 满驻~975 ns/op97.3%反超 flat:热块只解一次,后续只剩"块内二分"
block + cache 受限(64 块)~2854 ns/op0.2%27778 块里只缓 64 个 → 几乎全 miss + 淘汰开销

这是整条 Phase E 弧线的收束。把三个数字连起来看:

  • E3:block 热读 ~1440ns,输给 flat 的 ~1190ns(块化的"按需解块"代价)。
  • E5:加 lz4,block 热读涨到 ~2639ns(解压更贵),但磁盘 −63%。
  • S:满驻 cache,block 热读掉到 ~975ns,反超 flat——而磁盘 −63% 的赢面一分没还。

压缩(省空间、费 CPU)与缓存(费内存、省 CPU)是天生一对:cache 把压缩的 CPU 代价摊成一次性(每块只解一次),压缩的磁盘/内存收益却持续。而受限 cache 那一行同样重要——它诚实地说:cache 不是魔法,工作集装不下时命中率坍塌(0.2%),还白搭淘汰开销。cache 大小该按热数据工作集定,不是越大越好(那就是 Phase T 的 table cache、以及容量 vs 命中率的调优课)。

🧰 Coding style:Arc 共享只读 · index-as-pointer · 可选依赖注入

  • Arc<Block> 共享只读:cache 和调用方同时持有同一个解码块,靠 Arc 引用计数共享、克隆廉价(只加计数)。"不可变 + 引用计数 = 安全共享只读"是 Rust 缓存/快照类结构的主力范式(承 H 阶段 MVCC 的同一招)。
  • index-as-pointer 写链表:用 key 当链接、回哈希表解引用,绕开"Rust 裸指针链表"的借用地狱,还保 O(1)。当你需要"侵入式数据结构 + 借用检查器友好",这是标准答案。
  • 可选依赖注入(Option<Arc<Cache>>):cache 作为可选能力注入 reader,None 时行为与从前一字不差。新功能不破坏旧路径、按需开启——引擎、既有测试全程无感。这是给深模块"加 cache 一类横切能力"的干净接法。

🧠 System mindset:缓存层次 · 局部性 · 容量/命中率 · 压缩×缓存对偶

概念本阶段怎么体现 / 跨域同构
缓存层次分工用户态 block_cache(缓存解码产物)叠在 OS page cache(缓存裸字节)之上,各缓各的、互不替代。同构:CPU L1/L2/L3、CDN edge/origin、DB buffer pool + OS cache、浏览器 memory/disk cache。每层缓存离"用"越近,缓的东西越"熟"(越接近可直接消费的形态)。
局部性是命中率的前提满驻 97.3% vs 受限 0.2%,差距全在"工作集装不装得下"。LRU 赌的是时间局部性(刚用的还会用)。无局部性的负载(纯随机扫一遍)cache 必然失效——这时该上 bloom(跳过)或干脆别缓。
容量 vs 命中率cache 越大命中率越高,但内存有限、且大 cache 自身有管理开销。这条曲线(命中率 vs 容量)是所有缓存系统的核心调优面——Phase T 的 table cache、Redis maxmemory、CDN 容量规划都在这条曲线上找点。
压缩 × 缓存对偶压缩省空间/费 CPU,缓存费内存/省 CPU——两者在 CPU 维度互补:cache 命中后压缩的解压成本归零,只剩压缩的空间收益。"压了就缓、缓了敢压"是存储引擎的标准组合拳。

迁移收获:cache 不是"加上就快"。它的收益 = 命中率 × 单次省下的工。Phase E 把"单次省下的工"做大了(lz4 解压很贵),于是同样命中率下 cache 的边际价值飙升。设计任何缓存前,先量这两个量;受限 cache 那一行提醒你:命中率上不去时,再贵的单次节省也乘出 0。

🧪 Harness 系统:LRU 淘汰序 + 命中率断言

为什么这样验

LRU 最容易错在淘汰顺序:链表指针接反、get 忘了把命中项提到头、淘汰挑错端。功能差分测不出来(返回值都对),得专门构造淘汰序断言:cap=2,放 k1、k2,触碰 k1,再放 k3 —— 正确的 LRU 必须淘汰 k2(最久未用),而非 k1。这条用例同时验证了"get 会刷新最近用"和"淘汰挑 tail"两件事。其次是命中率作为可断言指标:bench 里满驻 cache 命中率必须 >95%、受限必须坍塌——把"cache 有没有生效"从感觉变成数字门槛。这类"断言一个统计量落在预期区间"的 harness(命中率、p99、压缩率、QPS),是性能组件区别于功能组件的测试方式:功能测对不对,性能组件还要测有没有起到作用。

    #[test] fn hit_miss_counting() {
        let c = BlockCache::new(2);
        let k1 = cache_key(0, 0);
        assert!(c.get(k1).is_none());        // miss
        c.put(k1, one_block(b"a", b"1"));
        assert!(c.get(k1).is_some());        // hit
        assert_eq!(c.hits(), 1); assert_eq!(c.misses(), 1);
    }

    // 淘汰序:cap=2,触碰 k1 后插 k3 → 必须淘汰 k2(最久未用),不是 k1。
    #[test] fn lru_evicts_least_recently_used() {
        let c = BlockCache::new(2);
        let (k1, k2, k3) = (cache_key(0,0), cache_key(0,1), cache_key(0,2));
        c.put(k1, ..); c.put(k2, ..);
        assert!(c.get(k1).is_some());        // 触碰 k1 → k2 变最久未用
        c.put(k3, ..);                       // 淘汰 k2
        assert!(c.get(k2).is_none(), "k2 应被淘汰");
        assert!(c.get(k1).is_some() && c.get(k3).is_some());
    }

核心问题(自己答)

Phase S 想透这几个
  1. 缓存解码块 vs 缓存裸字节,差在哪? OS page cache 已经缓了文件页,我们为什么还要自建 cache?(提示:OS 缓的是压缩态裸字节,每次 get 仍要 lz4 解压 + 解 restarts 数组。我们缓解码后的 Block,把这部分用户态的工也省掉。E5 加压缩后两者价值差距拉满。)
  2. cap 该按什么定? 满驻 97% vs 受限 0.2%。生产里 cache 该开多大?(提示:按热数据工作集,不是按总数据。受限那行说明:小于工作集就坍塌。但也不能无限大——内存有限、且要给 memtable/索引留地方。这是 Phase T、以及"内存预算分配"的调优题。)
  3. 单锁会成瓶颈吗? 我们的 Mutex<Inner> 全局一把锁。多线程高并发读时会怎样?(提示:所有 get 抢同一把锁 → 串行化。RocksDB 的 LRUCache 分片(按 key 哈希分到 N 个独立 shard,各自一把锁)把锁竞争降到 1/N。这是你"并发 backlog"在缓存层的一道题:分片是降低锁竞争的通用招。)
  4. cache 会不会让 blocks_read 说谎? 命中时我们不增 blocks_read。那 E2 的"一次点查最多读一块"不变量在带 cache 时还成立吗?(提示:那条不变量测的是无 cache 的 open 路径,语义是"读盘放大";带 cache 后"读盘"次数只会更少。两个计数器各测一件事:blocks_read=真读盘,cache.hits=免读盘。别让一个指标背两个含义。)

承上. 这一步直接回收了 E3 与 E5 反复点名的两笔账:热块重复读盘、重复解压。块化 SST 至此"既小又快"的故事才真正闭合。

启下. Phase T — table cache(复用打开的文件句柄,省 open + 索引载入)是同一思路的下一层缓存;Phase R — bloom filter 则从另一端发力:让不存在的 key 连块都不读(cache 救热块,bloom 救 miss)。两者 + 本 cache 合起来,就是 RocksDB 读路径"少读盘"的三板斧。性能回归门(接 storage-benchmark-lab)在 R 正式落地,本课的命中率指标是它的第一块拼图。
