STEP D1 · Phase D 读路径

读路径接 SST + 重启发现

补上 Phase C 末尾那个故意的空洞:让 Engine::get 在 active MemTable 落空后,顺着新到旧的 SST 列表继续找;并让 Engine::open 在重启时从磁盘文件名重建那个 SST 列表。这两步合起来,数据第一次能"写进 SST → 重启 → 读回来"。

GOAL

给 Engine 加 ssts: Vec<SstReader>(新到旧);get 落空查 SST;flush 把新写的 reader 插到列表头;open 调 discover_ssts 扫盘装载并续 next_sst。

Done when: flush 后 get 能从 SST 读到(不再是 None);重开引擎后老 SST 仍可读、且新 flush 写的是 000001.sst 而非覆盖 000000.sst。

为什么是"新到旧"(查找链锚)

LSM 的每次覆盖写,不是原地改,而是追加一个更新的版本到更上层(memtable / 更新的 SST)。所以同一个 key 可能在多层都有副本,越上层越新。读路径必须按从新到旧扫,第一个命中就停——这样上层的新值自然把下层的旧值挡住(shadow)。我们现在没有删除/墓碑(Phase I 才有),所以"第一个 Some 即最新值"成立。对标 RocksDB DBImpl::GetImpl:先 sv->mem->Get,落空再 sv->imm->Get,再 sv->current->Get(查 SST);任一命中即 done = true。

use crate::sst::{SstBuilder, SstReader};
use std::path::{Path, PathBuf};

pub struct Engine {
    dir: PathBuf,
    mem: MemTable,
+   ssts: Vec<SstReader>,   // 已落盘 SST;index 0 = 最新
    wal: Wal,
    next_sst: u64,
}

代码 2 — open 扫盘发现 SST

    pub fn open(dir: &Path) -> Result<Self> {
        std::fs::create_dir_all(dir)?;
        let wal_path = dir.join("wal.log");
        let mem = MemTable::new();
        for (k, v) in Wal::replay(&wal_path)? { mem.put(k, v); }
        let wal = Wal::open_append(&wal_path)?;
+       let (ssts, next_sst) = Self::discover_ssts(dir)?;   // 扫盘 + 续号
        Ok(Self { dir: dir.to_path_buf(), mem, ssts, wal, next_sst })
    }

+   /// 扫描 NNNNNN.sst,按编号升序解析;返回(新到旧的 reader, 下一个可用编号)。
+   /// 重开不读 manifest,纯靠文件名约定。RocksDB 靠 MANIFEST 精确登记(Phase L)。
+   fn discover_ssts(dir: &Path) -> Result<(Vec<SstReader>, u64)> {
+       let mut nums: Vec<u64> = Vec::new();
+       for entry in std::fs::read_dir(dir)? {
+           let path = entry?.path();
+           if path.extension().and_then(|e| e.to_str()) == Some("sst")
+               && let Some(stem) = path.file_stem().and_then(|s| s.to_str())
+               && let Ok(n) = stem.parse::<u64>()
+           { nums.push(n); }
+       }
+       nums.sort_unstable();
+       let next = nums.last().map_or(0, |n| n + 1);     // 续到 max+1,绝不覆盖
+       let mut ssts = Vec::with_capacity(nums.len());
+       for n in nums.into_iter().rev() {                // 编号大(新)的排前面
+           ssts.push(SstReader::open(&dir.join(format!("{n:06}.sst")))?);
+       }
+       Ok((ssts, next))
+   }
续号是个真 bug 防线. Phase C 的 open 永远把 next_sst 设 0。一旦你重开后再 flush,就会写 000000.sst 覆盖掉上次的——静默丢一整段数据。discover_ssts 把 next_sst 续到 max+1,这是 Phase D 顺手堵上的洞。

代码 3 — get 接 SST 查找链

    /// active MemTable → SST(新到旧),第一个命中即返回。
    /// 暂无墓碑(Phase I 才有)→ "第一个 Some 即最新值" 成立。
    pub fn get(&self, key: &[u8]) -> Option<Value> {
        self.mem.get(key)
            .or_else(|| self.ssts.iter().find_map(|r| r.get(key)))
    }

ssts 是新到旧排的,find_map 返回第一个(=最新)命中。一行就是整条"新到旧、命中即停"的语义。

代码 4 — flush 注册新 reader

    pub fn flush(&mut self) -> Result<PathBuf> {
        let path = self.dir.join(format!("{:06}.sst", self.next_sst));
        self.next_sst += 1;
        let mut b = SstBuilder::new();
        for (k, v) in self.mem.iter() { b.add(&k, &v)?; }
        b.finish(&path)?;
+       self.ssts.insert(0, SstReader::open(&path)?);  // 刚写的最新,插到头部
        self.mem = MemTable::new();
        self.wal = Wal::reset(&self.dir.join("wal.log"))?;
        Ok(path)
    }

注意顺序仍是 Phase C 的铁律:finish 落盘成功,才 open 注册、才截 WAL。insert(0, ..) 维持"新到旧"不变量。(D2 会把 flush 推广到排空多个 immutable;这里先保持 Phase C 的单表签名。)

🧰 Coding style:Option 组合子串成查找链 · 不变量编码进数据结构序

  • or_else 短路链:mem.get(k).or_else(|| ssts...) 把"逐层 fallback、命中即停"写成一条声明式表达式,而不是一串 if let Some(..) return。or_else 惰性——前一层命中,后面的闭包根本不执行。等价于 RocksDB 那段 if mem->Get(..) done=true; else if imm->Get(..) ... 的命令式短路,但更难写错。可迁移到任意有 Option/Maybe/?: 的语言。
  • find_map = "找第一个能映射成功的":对新到旧的 reader 列表,find_map(|r| r.get(k)) 一次表达"扫到第一个 Some 就停并返回它"。比手写下标循环 + break 更不易出错。
  • 把不变量编码进数据结构的"序":整个读路径的正确性,押在 ssts 永远新到旧排列这一条不变量上。我们用两个动作守住它:flush 时 insert(0, ..)(新的进头),discover 时 .rev()(大编号在前)。不变量一旦由结构保证,查找代码就能极简(一行 find_map)。这是"让数据结构替你记住规则"的范式。

🧠 System mindset:分层查找 · 新到旧覆盖 · "最近写在最前"

"维护多个版本、查询时从新到旧扫、第一个命中即停"——这不是 LSM 专利,是一类极通用的覆盖语义。哪里有"新配置/新定义覆盖旧的、但旧的还留着",哪里就是这个形状:

领域"新到旧、第一个命中即停"长这样
操作系统$PATH 查找:左边的目录先命中即用;进程页表多级从顶层往下走第一个有效映射
编程语言作用域链 / 原型链:内层变量遮蔽外层;JS 原型链、Python MRO 找第一个有该属性的类
网络路由表最长前缀匹配、防火墙规则"首条匹配即生效"、DNS 解析器缓存优先于权威
版本控制Git:HEAD → 父提交链;一个文件的"当前内容"是从最新提交往回找的第一个版本
构建/容器OverlayFS / Docker 镜像分层:上层文件遮蔽下层同名文件(和 LSM 几乎同构)

迁移收获:见到"多版本叠放 + 覆盖"语义,立刻问三件事——(1) 层序是什么(谁更新)?(2) 查找是命中即停(覆盖)还是全收集再合并(如 merge operator / CSS 层叠)?(3) 删除怎么表达?LSM 这里"命中即停 + 暂无删除",所以一行 find_map 就够;等 Phase I 加了墓碑,"删除"就得是一个能命中并返回 None 语义的特殊版本——届时这条链要升级。OverlayFS 用 .wh. whiteout 文件解决同一问题,正是同构的设计点。

🧪 Harness 系统:重启重发现(跨进程持久)

为什么这样验

读路径的持久性 bug 只有跨一次进程重启才暴露:内存里的 ssts 列表重启就没了,必须纯从磁盘文件名重建。这个 harness 用一对花括号 { } 制造"第一段生命周期写盘 → drop → 第二个 Engine::open 冷启动"的边界,逼着 discover_ssts 真干活。它还顺带验续号:重开后再 flush 必须写新编号,不能覆盖。这类"建立状态 → 销毁运行时 → 冷重建 → 校验"的 harness,在数据库恢复、配置热加载、缓存预热里都通用。

代码 5 — 两个 harness(接 engine.rs 的 tests)

    // D1 核心:flush 后 get 能从 SST 读到(Phase C 那个"故意空洞"补上)。
    #[test]
    fn get_reads_from_sst_after_flush() {
        let dir = tempdir().unwrap();
        let mut e = Engine::open(dir.path()).unwrap();
        e.put(bytes("apple"),  bytes("red")).unwrap();
        e.put(bytes("banana"), bytes("yellow")).unwrap();
        e.put(bytes("apple"),  bytes("green")).unwrap(); // 覆盖
        e.flush().unwrap();
        assert_eq!(e.get(b"apple").as_deref(),  Some(&b"green"[..])); // 从 SST 读,最新值
        assert_eq!(e.get(b"banana").as_deref(), Some(&b"yellow"[..]));
        assert_eq!(e.get(b"missing"), None);
    }

    // D1 重启:open 重新发现 *.sst 并能读回;next_sst 续号不覆盖旧文件。
    #[test]
    fn reopen_rediscovers_ssts() {
        let dir = tempdir().unwrap();
        { let mut e = Engine::open(dir.path()).unwrap();
          e.put(bytes("x"), bytes("1")).unwrap();
          e.flush().unwrap(); }                       // 000000.sst,然后 drop
        let mut e2 = Engine::open(dir.path()).unwrap(); // 冷启动,扫盘重建
        assert_eq!(e2.get(b"x").as_deref(), Some(&b"1"[..]));  // 不再是 None!
        e2.put(bytes("y"), bytes("2")).unwrap();
        e2.flush().unwrap();                              // 写 000001.sst,不覆盖
        assert!(dir.path().join("000000.sst").exists());
        assert!(dir.path().join("000001.sst").exists());
        assert_eq!(e2.get(b"x").as_deref(), Some(&b"1"[..]));
        assert_eq!(e2.get(b"y").as_deref(), Some(&b"2"[..]));
    }

核心问题(自己答)

D1 想透这几个
  1. 读放大:现在 get 一个不存在的 key,最坏要扫几个 SST?每个 SST 的 get 是 O(log n) 二分。10 个 SST、查一个 miss,要做多少次二分?这就是 LSM 的读放大。哪两个后续 phase 专治它?(提示:每 SST 一个 bloom 先否定 / 把 SST 按 key range 分层只查相关的。)
  2. 线性扫 vs 按 range 二分:我们对 SST 列表线性扫(find_map)。RocksDB 在 L1+ 是"每层 SST 按 key range 不重叠,二分定位到唯一可能的那个 SST"。为什么 L0 仍然必须线性扫所有文件,而 L1+ 不用?(想想 L0 的文件是 flush 直出、range 互相重叠。)
  3. discover 的脆弱点:discover_ssts 信任"凡是 NNNNNN.sst 就是有效 SST"。如果上次 flush 在 finish 写到一半崩了,留下半截 000003.sst,这次 open 会怎样?(提示:SstReader::open 读 footer magic——半截文件没有合法 footer。这一步会 Err 还是被静默跳过?当前实现是哪种?这正是 RocksDB 要 MANIFEST "只认登记过的 SST" 的原因——Phase L。)
  4. 句柄成本:每个 SstReader 现在把整个文件读进内存(raw: Vec<u8>)。100 个 SST 各 64MB 会怎样?真引擎用 table cache(Phase T)+ block cache(Phase S)按需打开/缓存,而非全载。

下一步:D2 — immutable 表层 + 分层差分. 现在查找链只有两层(active → SST)。D2 在中间塞进 immutable memtable 列表:flush 不再瞬间完成,而是先 freeze(active 转 immutable,仍可读)、再慢慢落盘。get 要学会夹查这一层。届时上线本 Lab 的旗舰 harness:分层查找差分。
