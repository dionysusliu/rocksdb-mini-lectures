STEP D2 · Phase D 读路径

immutable 表层 + 分层查找差分

D1 的查找链只有两层(active → SST)。但真实引擎里,flush 不是瞬间完成的——它是慢 IO。这中间有个关键中间态:active 表满了,先"冻结"成只读的 immutable,换上新空表继续接写;旧表在后台慢慢落盘,期间仍然可读。 这一步把这层 immutable 塞进查找链,并上线本 Lab 的旗舰测试系统:分层查找差分。

GOAL

给 Engine 加 imm: Vec<Arc<MemTable>>(新到旧);加 freeze(active 整表移入 imm 头);get 在 active 与 SST 之间夹查 imm;flush 推广为"排空内存(active + 所有 imm)→ 多个 SST"(截 WAL 留到 D3接上)。

Done when: freeze 后数据仍可从 imm 读到;分层差分 harness 在 3000 步随机 put/freeze/flush 后,每步 get 都 == 扁平 BTreeMap oracle。

为什么 active 满了要冻结而非就地清空(SuperVersion 锚)

这正是 Phase C 核心问题 #3 埋的伏笔。如果直接清空 active,任何正拿着它的读线程会读到空表。RocksDB 的做法是 switch memtable:满表整个转为 immutable(只读、还在内存、仍服务读),装上一张新空表接写。一个读请求拿到的 SuperVersion = { mem(活跃) + imm(若干只读) + current(SST 版本) } 的一致快照,整段 get 都基于这个快照查,不受并发 flush 影响。我们 MVP 是单线程,但用 Arc<MemTable> 把"immutable 表可被多方共享、不可变"这个语义先立住——这就是无锁读历史的地基(Phase H/J MVCC)。

代码 1 — 加 imm 字段 + 引入 Arc

+ use std::sync::Arc;

pub struct Engine {
    dir: PathBuf,
    mem: MemTable,
+   imm: Vec<Arc<MemTable>>,   // 冻结待 flush 的不可变表;index 0 = 最新
    ssts: Vec<SstReader>,
    wal: Wal,

代码 2 — freeze:整表移入 imm

    /// 冻结 active:整表移入 imm 头部,换上新空表。
    /// WAL 不动——数据仍只在内存(imm),WAL 仍需保护它。只有 flush 落盘后 WAL 才可截断。
    pub fn freeze(&mut self) {
        if self.mem.is_empty() { return; }
        let old = std::mem::take(&mut self.mem);  // Default = MemTable::new()
        self.imm.insert(0, Arc::new(old));         // 新冻结的排头部
    }

std::mem::take 把 self.mem 换成默认值(空表)并交出旧表的所有权——一步完成"取出旧的、装上新的"。比 replace(&mut self.mem, MemTable::new()) 更简洁(clippy 也会这么提示,因为 MemTable: Default)。

代码 3 — get 夹查 imm(三层链)

    /// active → immutable(新到旧)→ SST(新到旧),第一个命中即返回。
    pub fn get(&self, key: &[u8]) -> Option<Value> {
        self.mem.get(key)
            .or_else(|| self.imm.iter().find_map(|m| m.get(key)))   // ← 新增中间层
            .or_else(|| self.ssts.iter().find_map(|r| r.get(key)))
    }

三层全是同一个 or_else + find_map 模式串起来。查找链的形状没变,只是中间多接了一节——这正是 D1 把不变量编码进"序"换来的扩展性。

代码 4 — flush 推广:排空内存 → 多个 SST

    /// 把内存里全部数据(active + 所有 imm)倒成 SST。返回写出的路径们。
    /// 注:截断 WAL 留到 D3 接上(见末尾 TODO),D2 先把"排空内存→多 SST"这条主链打通。
    /// 逐个 imm 落盘,旧→新:越新的表得到越大的编号 → 落到 ssts 头部(新到旧)。
    pub fn flush(&mut self) -> Result<Vec<PathBuf>> {        // 签名 PathBuf → Vec<PathBuf>
        self.freeze();                                   // 把 active 也纳入本次 flush
        let mut paths = Vec::new();
        while let Some(imm) = self.imm.pop() {           // pop = 末尾 = 最旧,先落盘
            let path = self.dir.join(format!("{:06}.sst", self.next_sst));
            self.next_sst += 1;
            let mut b = SstBuilder::new();
            for (k, v) in imm.iter() { b.add(&k, &v)?; }
            b.finish(&path)?;
            self.ssts.insert(0, SstReader::open(&path)?);  // 刚写的最新,排头部
            paths.push(path);
        }
        // TODO(D3):内存已排空 → 此处整截 WAL 才安全。self.wal = Wal::reset(&self.dir.join("wal.log"))?;
        // D2 暂不截:重开时 WAL 重放仍能恢复(值正确,只是 WAL 不回收)。截 WAL 的代码与"为什么安全"在 D3 讲透。
        Ok(paths)
    }
为什么先排空内存才敢整截 WAL(截 WAL 的代码 D3 接上,这里先把不变量讲清). WAL 保护的是所有还在内存的数据(active + 每个 imm)。若只 flush 一个 imm 就截 WAL,其它 imm + active 的数据会从 WAL 里被抹掉、却还没落盘 → 崩了就丢。我们 MVP 用最简单的安全策略:flush 一次性把内存全倒空,内存里啥都不剩了,才整截 WAL。RocksDB 能"只 flush 最旧那张表、只回收对应 WAL 段",靠的是给每张表/每段 WAL 编 log number 并在 MANIFEST 里记账——那是 Phase L + X。我们诚实地把它简化掉。

flush 的"新到旧"怎么保住 — 走一遍

设依次:put a=1 → freeze → put a=2 → flush。

动作activeimm(新→旧)ssts(新→旧)
put a=1{a=1}[][]
freeze{}[{a=1}][]
put a=2{a=2}[{a=1}][]
flush → freeze{}[{a=2},{a=1}][]
pop 最旧 {a=1} → SST#0{}[{a=2}][#0:{a=1}]
pop {a=2} → SST#1,insert(0){}[][#1:{a=2}, #0:{a=1}]

最终 get a → active 空 → imm 空 → ssts[0]=#1 → a=2。最新值胜出。 旧→新落盘 + insert(0) 这对组合,让"内存里的新旧"无损翻译成"SST 编号的大小",新到旧的不变量一路守住。

🧰 Coding style:Arc 共享不可变 · mem::take 原子换出 · 同构链的可组合

  • Arc<MemTable> = 共享的不可变所有权:immutable 表一旦冻结就只读,可能被多个读者(未来:多个 SuperVersion 快照)同时持有。Arc 让"同一份不可变数据多方共享、最后一个放手才析构"成为类型层面的保证。这是不可变 + 引用计数 = 安全共享的范式,见于 Swift class、C++ shared_ptr、Java GC 对象图。
  • std::mem::take 原子换出:freeze 要"把旧表拿走、立刻放一张新空表",中间一刻都不能让 self.mem 处于无效态。take 用 Default 一步换出,无 panic 间隙。这是 Rust 处理"移动一个 &mut 后面的值"的惯用法(对应 C++ move + 默认构造)。
  • 同构链可组合:三层查找全是 X.or_else(|| Y).or_else(|| Z),每层内部全是 iter().find_map(..)。因为每层都满足同一个"返回 Option、内部新到旧"的契约,新增一层就是再接一节 or_else,零额外脑力。这预告 Phase G 的 MergingIterator:那里把"点查命中即停"升级成"范围扫描多路归并",但"每个数据源是一个统一契约的迭代器、按新旧合并"的骨架,从这里就立住了。

🧠 System mindset:读时合并多个有序源 · 写快读偿 · 不可变 + 版本

LSM 的核心权衡在这一步显形:写极快(只追加到内存表),代价是读要去多个地方合并。这种"把整理工作从写时推迟到读时(或后台)"的思路,是一大类系统的共同心法:

领域"读时/惰性合并多个有序源"长这样
算法外部排序 / 多路归并:内存放不下,先切成多个有序 run,再 k 路归并。LSM 的 imm+SST 就是一堆 run
分布式MapReduce shuffle / Spark:各 mapper 出有序分片,reducer 归并。LSM compaction 是单机版的同一动作
数据库多版本读:活跃事务的改动 + 已提交快照,读时按可见性合并(MVCC)
内核(★)RCU:写者造一份新副本、原子换指针、旧副本等所有读者放手才回收。我们的 Arc+freeze 是它的用户态雏形
HPClog-structured / append-only 写聚合,后台再 compaction;摊薄随机写为顺序写

迁移收获:当"写路径是瓶颈"时,问一句——能不能把整理推迟? 把写做成"只追加 + 记个版本",让读/后台去合并。代价是读放大与空间放大(读/写/空间三角,Phase N/O/AF 会实测调)。反向也成立:当读是瓶颈、写很闲,就该把整理提前到写时(B-tree 原地更新)。LSM vs B-tree 的全部分歧,就是"整理放写时还是读时"这一个选择。这一框架能帮你看穿存储引擎、日志系统、甚至 CI 缓存策略的设计取舍。

🧪 Harness 系统:分层查找差分(本 Lab 旗舰)

为什么这是最强的一种验法

读路径的正确性,押在一个横跨多个组件、随时间演化的不变量上:无论引擎把数据怎么摊、怎么重排(active↔imm↔SST 不停搬家),get(k) 永远等于"k 的最新写入值"。这种不变量,用几个手写断言只能抽样几个点,抓不住"某条特定 put/freeze/flush 交织序列下,第 873 步层序错乱"这类 bug。

差分(differential)的威力:拿一个显然正确但慢的参照实现(扁平 BTreeMap,一个 key 只存一份最新值)当 oracle,让它和被测引擎跑同一串随机操作,每步比对输出。引擎内部多复杂都没关系——只要它对外行为和那张平表无法区分,就对了。随机操作流会自动撞出你想不到的交织顺序;固定种子的伪随机让失败可复现(打印 step N 即可定位)。这是 property-based / model-based testing 的核心,SQLite、TiDB、FoundationDB 都靠它的重型版本守命。

⏱ 跑得慢是正常的(~13s)。 Engine::put 每写一条都 wal.sync()(fsync 落盘),3000 步 × 每步多次写 = 数千次真 fsync,这是磁盘往返、不是 CPU 慢。所以这条长跑差分被隔离进 CI 的 integration 档(单线程跑),不拖慢秒级反馈的单元 test 档。

代码 5 — freeze 可读性(接 tests)

    // D2:freeze 把 active 移入 imm,数据仍可读;新 active 覆盖同名 key 时影子 imm。
    #[test]
    fn frozen_immutable_still_readable() {
        let dir = tempdir().unwrap();
        let mut e = Engine::open(dir.path()).unwrap();
        e.put(bytes("a"), bytes("1")).unwrap();
        e.freeze();                                       // a=1 移入 imm,active 空
        assert_eq!(e.get(b"a").as_deref(), Some(&b"1"[..]));  // 从 imm 读
        e.put(bytes("b"), bytes("2")).unwrap();
        assert_eq!(e.get(b"b").as_deref(), Some(&b"2"[..]));  // active
        assert_eq!(e.get(b"a").as_deref(), Some(&b"1"[..]));  // 仍从 imm
        e.put(bytes("a"), bytes("9")).unwrap();             // active 覆盖
        assert_eq!(e.get(b"a").as_deref(), Some(&b"9"[..]));  // active 影子 imm
    }

代码 6 — 旗舰:分层查找差分(接 tests)

    // 随机交织 put/freeze/flush;每步 get 必须 == 扁平 BTreeMap oracle。
    // 暂无删除 → oracle 只增/覆盖,"第一个 Some 即最新值" 成立。
    #[test]
    fn layered_lookup_differential() {
        fn xorshift(state: &mut u64) -> u64 {           // 固定种子伪随机,失败可复现
            let mut x = *state;
            x ^= x << 13; x ^= x >> 7; x ^= x << 17;
            *state = x; x
        }
        let dir = tempdir().unwrap();
        let mut e = Engine::open(dir.path()).unwrap();
        let mut model: BTreeMap<Vec<u8>, Vec<u8>> = BTreeMap::new();
        let mut s = 0x1234_5678_9abc_def0u64;
        for step in 0..3000u32 {
            match xorshift(&mut s) % 100 {
                0..=79 => {                                  // 80% put(~50 个热点 key,频繁覆盖)
                    let k = format!("k{:03}", xorshift(&mut s) % 50);
                    let v = format!("v{step}");
                    e.put(bytes(&k), bytes(&v)).unwrap();
                    model.insert(k.into_bytes(), v.into_bytes());
                }
                80..=91 => e.freeze(),                          // 12% freeze
                _ => { e.flush().unwrap(); }                     // 8% flush
            }
            for probe in 0..50u32 {                          // 每步全量对拍
                let k = format!("k{probe:03}");
                assert_eq!(
                    e.get(k.as_bytes()).as_deref(),
                    model.get(k.as_bytes()).map(|v| v.as_slice()),
                    "mismatch at step {step}, key {k}");
            }
        }
        e.flush().unwrap();                                // 收尾再重开,证明持久化也对(D2 未截 WAL → 重开走 WAL 重放恢复;D3 截 WAL 后才纯靠 SST)
        drop(e);
        let e2 = Engine::open(dir.path()).unwrap();
        for (k, v) in &model {
            assert_eq!(e2.get(k).as_deref(), Some(v.as_slice()));
        }
    }

核心问题(自己答)

D2 想透这几个
  1. 为什么 oracle 必须比被测实现简单:我们的 oracle 是扁平 BTreeMap——一个 key 只存一份最新值,没有分层、没有新旧覆盖逻辑。如果 oracle 也照着"分层 + 新到旧"实现,这个差分还能抓 bug 吗?(差分的铁律:oracle 要用和被测实现不同的、更显然正确的方式算出答案。同源 = 同错。)
  2. 无墓碑的红利与债:现在"第一个 Some 即答案"。等 Phase I 加了删除(墓碑),查找链撞到一个"删除标记"该返回什么?or_else 这条链还够用吗?(提示:删除是个会命中、但语义是"到此为止、返回不存在"的特殊版本。or_else 只认 None 才继续——墓碑是 Some(删除),会卡在这层。这条链届时必须改造。)
  3. Arc 现在是不是多余:单线程 MVP 里 imm 没人并发读,用 Box 甚至 MemTable 直接存都行。为什么仍坚持 Arc?(为 Phase H/J/V:读者要能持快照、跨锁、并发读 immutable,而 flush 同时在背后消费它。Arc = "最后一个放手才回收",正是 SuperVersion 引用计数的雏形。提前立类型,后面不返工。)
  4. imm 列表会无限长吗:freeze 不断把表塞进 imm,flush 才清空。如果写得比 flush 快,imm 越堆越多 → 内存爆。RocksDB 怎么挡?(提示:max_write_buffer_number 到了就挡住写——write stall,Phase W。D3 先把"满了自动 flush"接上,W 再补"flush 跟不上就限速"。)
