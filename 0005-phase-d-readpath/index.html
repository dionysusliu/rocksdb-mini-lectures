Phase D — 读路径合并(触达 M1 MVP)

Phase C 把数据倒进了 SST,却留了个故意的空洞:Engine::get 只查 active MemTable,flush 进 SST 的数据读不回来。这一阶段补上完整的 LSM 查找链——get 顺着 active → immutable → SST,新到旧找,第一个命中即返回。补完这条链,引擎第一次 Put / Get / 重启 / flush 全闭环:M1 MVP 达成。

Mission 锚点. LSM 把一次写"摊"在多个有序结构里(内存表 + 一堆 SST),换来极快的写。代价转嫁给读:一个 key 可能在任意一层,且同名 key 的新版本影子旧版本。读路径的职责就是——按从新到旧的顺序合并这些层,让上层把下层挡住。对标 RocksDB 的 SuperVersion(mem + imm + version)与 DBImpl::GetImpl 的三段式查找。
本 Lab 宣称的 GOAL
  1. 读穿 SST:Engine 持有 Vec<SstReader>(新到旧),get 在 active 落空后顺着 SST 列表新到旧找;open 发现磁盘上的 *.sst 并装载,next_sst 续号不覆盖旧文件。
  2. 读穿 immutable:引入 Vec<Arc<MemTable>> 不可变表层,freeze 把 active 整表移入;get 在 active 与 SST 之间夹查 immutable。flush 排空内存(active+imm)成 SST 再截 WAL。
  3. 自动落盘:put 检测 is_full 自动 flush(A3 那个"满了"信号终于接上闭环)。

Done when: cargo test 全绿,含分层查找差分(随机交织 put/freeze/flush 对拍扁平 BTreeMap)与重启重发现 SST 两类 harness。

🏁 本 Lab 收尾即 M1 — MVP 引擎:Put / Get / 重启持久 / flush 全通。路线图开头说的"先跑通",到这里第一次成真。

harness 系统:自触发不变量

本 Lab 的 harness 主题:合并语义的可信

读路径的 bug 是语义级的:数据没丢、文件没坏,但读到了旧版本(层序搞反)、或该有的没读到(漏了某层)。这类 bug 用单点断言很难抓,必须用差分把引擎的复杂内部布局对拍一个简单 oracle。

Harness 系统抓什么用在
重启重发现(跨进程持久)open 必须从磁盘文件名重建 SST 列表 + 续号;重开后老 SST 仍可读、新 flush 不覆盖D1
分层查找差分随机交织 put/freeze/flush,数据被摊在 active+imm+多个 SST、内部反复重排;每步 get 仍须 == 扁平 BTreeMapD2
自触发不变量狂写不手动 flush,引擎应自己落盘且全程可读D3
带着这个问题进 D1:同一个 key "a" 既在一个旧 SST 里(值 "1"),又在 active 里(值 "2")。get("a") 要返回哪个?为什么?如果反过来——SST 比 memtable"新",查找顺序该怎么变?(这就是"新到旧"四个字的全部分量。想清楚再写代码。)
