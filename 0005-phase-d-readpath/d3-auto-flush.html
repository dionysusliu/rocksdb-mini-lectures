STEP D3 · Phase D 读路径

满了自动 flush — M1 MVP 达成

到这里读写两条路都通了,只剩一件事手动:什么时候落盘。还记得 Phase A3 给 MemTable 加的 is_full(threshold) 信号吗?它一直在那儿空等。这一步把它接上动作:put 后一旦 active 满了,自动 flush。 引擎第一次能"自己照顾自己"——不再需要外部手动调 flush。

但自动 flush 上线前,要先补 D2 留的那个 TODO:flush 落盘后截断 WAL。D2 手动 flush 时不截还能将就(WAL 只是不回收);可一旦 put 自动触发 flush,不截的 WAL 会随每次写无限增长——重启重放越来越慢、磁盘越占越多。所以本课分两半:① 补上 flush 的 WAL 截断;② 接上自动触发。

GOAL

① 在 flush 末尾补上 self.wal = Wal::reset(...),落盘后截断 WAL(填掉 D2 的 TODO)。② 给 Engine 加 flush_threshold: usize 字段;put 末尾检测 self.mem.is_full(threshold),满则调 flush()。加 set_flush_threshold 便于测试调小阈值。

Done when: 把阈值调小后狂写、不手动 flush,磁盘上自己冒出 SST 且全部 key 仍可读;flush 后 WAL 被截断(重启不再重放已落盘的 key,WAL 不随写无限增长)。

第一半 — 补上 flush 的 WAL 截断(D2 的 TODO)

D2 的 flush 把内存全排空成 SST,却没截 WAL——留了行 TODO。现在补上:落盘成功后,内存里啥都不剩了,WAL 保护的那批数据已全在 SST 上,整截 WAL 才安全。

代码 0 — 在 flush 末尾填掉 TODO

    pub fn flush(&mut self) -> Result<Vec<PathBuf>> {
        self.freeze();
        let mut paths = Vec::new();
        while let Some(imm) = self.imm.pop() {
            // …逐个 imm 落盘(D2 已写)…
        }
+       // D2 的 TODO 在此填上:内存已排空 → 整截 WAL 安全。
+       self.wal = Wal::reset(&self.dir.join("wal.log"))?;
        Ok(paths)
    }
为什么"内存全空才敢整截 WAL"是铁律. WAL 保护的是所有还在内存的数据(active + 每个 imm)。若只 flush 一个 imm 就截 WAL,其它 imm + active 的数据会被从 WAL 抹掉、却还没落盘 → 崩了就丢。MVP 用最简单的安全策略:flush 一次性把内存全倒空,才整截 WAL。RocksDB 能"只 flush 最旧那张表、只回收对应 WAL 段",靠的是给每张表/每段 WAL 编 log number 并在 MANIFEST 记账——那是 Phase L + X。这也正是 D2 核心问题 #4、本课核心问题 #3 指向的同一处简化:"满了就排空全部内存"和"内存全空才截 WAL"是一对绑定的 MVP 取舍。
为什么这件事拖到自动 flush 才必须做. D2 手动 flush 时不截 WAL 顶多是"WAL 不回收、重启多重放一遍",将就得过去。可 D3 让 put 自动触发 flush 后,每攒满一批就 flush 一次——若仍不截,WAL 会随写入无限增长:重启重放越来越慢、磁盘越占越多。自动化把一个"能将就的瑕疵"放大成"必须修的 bug"——这正是该在本课补它的原因。

为什么 flush 该是后台、而非写路径同步(诚实标注锚)

我们 MVP 在 put 里同步调 flush——简单,但有代价:flush 是慢磁盘 IO,把它放在写路径上,触发那一刻的 put 会被卡住等落盘完成(尾延迟尖峰)。真 RocksDB 的写路径只做一件快事——switch memtable(active 转 imm,装新表,几乎瞬时),然后给后台线程派一个 flush job 就立刻返回;落盘在后台慢慢做。只有当imm 堆积过多(写比 flush 快)才反过来挡住写(write stall)。我们把"后台线程 + 背压"整体简化成"同步 flush",并诚实标注:这是 Phase W(write stall / 限速)要补的洞。

对标 DBImpl::SwitchMemtable + 后台 flush 调度 · 背压:write_controller.cc(Phase W)

第二半 — 接上自动触发

  1. 给 Engine 加 flush_threshold 字段,open 里给个默认值(如 4 MiB)(「代码 1」)。
  2. put 末尾接 is_full 自动 flush;加 set_flush_threshold(「代码 2」)。
  3. 敲入「代码 3」自触发 harness,跑 cargo test 全绿。

代码 1 — 加阈值字段

pub struct Engine {
    dir: PathBuf, mem: MemTable, imm: Vec<Arc<MemTable>>,
    ssts: Vec<SstReader>, wal: Wal, next_sst: u64,
+   flush_threshold: usize,   // active 超过此字节数 → 自动冻结+flush
}
// open(...) 构造 Self 时加上:
+   flush_threshold: 4 * 1024 * 1024,   // 4 MiB 默认

代码 2 — put 自动触发 + 阈值 setter

    pub fn put(&mut self, key: Key, value: Value) -> Result<()> {
        self.wal.append(&key, &value)?;
        self.wal.sync()?;
        self.mem.put(key, value);
+       // active 满了就自动落盘。MVP 同步(会堵写路径=write stall,Phase W);
+       // 真 RocksDB 是写路径只切表、后台线程 flush。
+       if self.mem.is_full(self.flush_threshold) {
+           self.flush()?;
+       }
        Ok(())
    }

+   /// 测试/调参用:把自动 flush 阈值调小,演示自动落盘。
+   pub fn set_flush_threshold(&mut self, bytes: usize) { self.flush_threshold = bytes; }
顺序仍是铁律. 自动 flush 放在 self.mem.put(..) 之后:先把这条写真正落进 active(且 WAL 已 sync),再判断要不要 flush。反过来先判断后 put,会漏掉"正是这条 put 让表变满"的情况。

🧰 Coding style:策略与机制分离 · 阈值即策略

  • policy / mechanism 分离:flush() 是机制(怎么落盘),is_full(threshold) 是策略(何时该落盘)。把"多大算满"做成一个可注入的字段(flush_threshold + set_flush_threshold),而非硬编码,就能让测试用 64 字节、生产用 64 MB,机制一行不改。这是"把会变的参数从会稳定的逻辑里拎出来"的通用范式,见于内核调度策略、GC 触发阈值、限流器配置。
  • 触发器贴着状态变化点:满判定紧跟在唯一会让表变大的操作(put)之后——状态一变就立刻检查不变量/触发条件,而不是另起一个轮询循环去问"满了没"。事件驱动胜过轮询,延迟更低、逻辑更局部。

🧠 System mindset:背压与流量控制 · 同步 vs 异步落盘

"生产快于消费,中间缓冲会涨;涨到某个水位就得采取行动"——这是背压(backpressure),贯穿所有有生产/消费速差的系统:

领域"缓冲涨到水位就行动"长这样
网络(★)TCP 流量控制:接收窗口满 → 通告 zero window 让发送方停;拥塞控制按丢包/RTT 退避
操作系统脏页回写:dirty page 比例过高 → pdflush/回写线程落盘;再高则阻塞写者(我们的同步 flush 正是这种"阻塞写者")
消息/流Kafka/Reactive Streams:消费跟不上 → 缓冲、丢弃、或反压上游限速
HPC双缓冲 / 流水线:一块算、一块 IO;若 IO 慢于计算,计算就得等(=stall)

迁移收获:我们当前是最朴素的背压——直接在写者线程里同步消费(flush),等于"缓冲一满就让生产者亲自去倒"。它简单、绝不丢数据,但把慢 IO 的延迟直接传导给写延迟(脏页回写阻塞写者、TCP 发送阻塞,都是这个味道)。进阶做法是解耦:后台线程消费(异步 flush),写者只在缓冲逼近上限时才被限速(RocksDB 的 slowdown/stop 双水位,Phase W)。"同步消费 vs 异步消费 + 水位限速"这对取舍,你会在写存储、调网络、设计任何队列时反复遇到——朴素同步先保正确,异步 + 背压再要吞吐。

🧪 Harness 系统:自触发不变量

为什么这样验

前面的 harness 都显式调 flush。这个要验的是引擎自己会不会动:把阈值调到极小(64 字节),只管狂 put、一次都不手动 flush,然后断言两件事——(1) 磁盘上确实自发冒出了 SST 文件(000000.sst 存在);(2) 数据散落在 active/imm/SST 各处,但全部仍可读。这是"自触发不变量":系统在无人干预下,既要主动达成某个目标态(落盘),又不能在自动化过程中破坏对外契约(可读性)。同类验法用于自动伸缩、自动 GC、自愈系统——验"它该动时动了,且动的过程没搞坏别的"。

代码 3 — 自触发 harness(接 tests)

    // D3:阈值调小后,put 自动触发 flush——无需手动调,数据自己落盘且可读。
    #[test]
    fn auto_flush_on_full() {
        let dir = tempdir().unwrap();
        let mut e = Engine::open(dir.path()).unwrap();
        e.set_flush_threshold(64);                  // 极小阈值,几次 put 就满
        for i in 0..20u32 {
            let k = format!("key{i:02}");
            e.put(bytes(&k), bytes("payload-bytes")).unwrap();
        }
        // 没手动 flush,但应已有 SST 落盘:
        assert!(dir.path().join("000000.sst").exists(),
                "auto-flush 应已写出至少一个 SST");
        // 全部 key 仍可读(分布在 active/imm/SST):
        for i in 0..20u32 {
            let k = format!("key{i:02}");
            assert_eq!(e.get(k.as_bytes()).as_deref(), Some(&b"payload-bytes"[..]));
        }
        // WAL 已截断也不丢数据:重开后全部 key 仍在(已落盘的来自 SST,未满那批来自 WAL 尾)。
        drop(e);
        let e2 = Engine::open(dir.path()).unwrap();
        for i in 0..20u32 {
            let k = format!("key{i:02}");
            assert_eq!(e2.get(k.as_bytes()).as_deref(), Some(&b"payload-bytes"[..]));
        }
    }

🏁 MILESTONE M1 — MVP 引擎达成

Put / Get / 重启持久 / flush 全闭环。你手上第一次有了一个最小但完整的 LSM 引擎:写进内存表(WAL 保命)→ 满了倒成有序 SST → 读穿 active+imm+SST 新到旧 → 重启从盘上重建。路线图开头说的"先跑通 MVP",到此成真。

核心问题(自己答)

D3 / Phase D 收尾
  1. 同步 flush 的延迟代价:用 cargo test -- --nocapture 或加个计时,观察触发 flush 的那次 put 比普通 put 慢多少。如果一个客户端正等这次 put 返回,它体验到的尾延迟是多少?这就是 write stall 的雏形——为什么生产引擎必须把 flush 推到后台?
  2. 阈值定多大:阈值太小 → SST 又多又小(读放大、文件句柄爆、compaction 压力大);太大 → 内存占用高、单次 flush 卡顿久、崩溃丢的未落盘数据多。RocksDB 默认 write_buffer_size 64MB,凭什么?(想想:它要平衡内存、单次 flush 时长、L0 文件数。没有银弹,是个可调旋钮。)
  3. auto-flush 触发的是排空全部内存:我们的 flush 会 freeze + 排空所有 imm。但"满了"只是 active 满,为什么要顺手把之前 freeze 的 imm 也一起落?(MVP 的 WAL 整截策略要求内存全空才安全——回顾 D2 的 warnbox。这也暴露了 MVP 的粗糙:真引擎该只 flush 该 flush 的那张表。Phase L 的 log number 解锁精细控制。)
  4. 崩溃窗口:自动 flush 在后台(未来)进行时崩溃,会怎样?哪些数据在 WAL、哪些在半截 SST、哪些已安全?把 Phase B 的崩溃恢复 + Phase C 核心问题 #1(半截 SST 识别)+ 本课串起来想一遍——这就是 LSM 的崩溃一致性全景,Phase L(MANIFEST)是收口的关键。
