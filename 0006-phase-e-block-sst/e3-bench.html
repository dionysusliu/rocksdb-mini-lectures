STEP E3 · Phase E block-based SST

基准:flat(Phase D) vs block(Phase E) 的 A/B

E1/E2 把 SST 块化了,但"快了多少、省了多少"还只是口头声称。这一步造一个最小的基准台,把 Phase D 的扁平 SST 和 Phase E 的块化 SST 放进同一个二进制,喂同一份数据、同一组随机查询(同种子),实测三件事:磁盘大小、索引内存、读延迟(冷/热)。对标 RocksDB 的 db_bench(fillseq / readrandom / readmissing)。

GOAL

冻结 Phase D 的扁平 SST 为 sst_flat.rs 基线(不再改),新建 nocache.rs(手写 F_NOCACHE 冷读)和 bin/bench.rs(手写 SplitMix64 + 正确性闸门 + 指标表)。cargo run --release --bin bench 一条命令出对照表。

Done when: 正确性闸门绿(flat.get == block.get == oracle,不绿不出数);表里 索引内存、读放大、冷读交叉点 三项都能读出块化的代价与收益——并能诚实说出块化在热点查上反而略慢。

诚实是这节课的主轴. 块化不是"全面更快"。它在磁盘 / 索引内存 / 冷读选择性查上赢,在热点查上输给"整文件常驻 + 直接偏移"的扁平实现。一个只报喜不报忧的基准是骗自己。这节课的价值,一半在数字,一半在敢把不利的那一栏也印出来。

为什么是"两个版本同进一个二进制"?(方法学锚)

对比两版代码,最容易翻车的是负载漂移——两次 run 的数据/查询序列稍有不同,数字就没法比。我们把扁平版冻结成 sst_flat.rs、块化版就是现在的 sst.rs,二者签名完全一致(new/add/finish/open/get),bin/bench.rs 同时 import 两者,用同一个 SplitMix64 种子生成数据与查询。这样"唯一变量"就只剩格式本身。git checkout 老版本再跑是另一条路,但两个二进制、两次环境,极易引入偏差——A/B 要可信,就把变量锁到只剩一个。

RocksDB 对标:tools/db_bench_tool.cc(fillseq / readrandom / readmissing)· 我们只取点查相关子集;readseq/seekrandom 等迭代器负载留到 Phase F 有了 iterator 再补。

步骤

  1. 把 Phase D 的扁平 sst.rs 整体复制为 src/sst_flat.rs(改模块名/报错字样即可),再给它的 SstReader 补三个测量方法:index_bytes()、bytes_read()、open_cold()(「代码 1」)。block 版 sst.rs 同样补这三个(read_block 里累加 bytes_read)。
  2. 新建 src/nocache.rs:手写 extern "C" fcntl + Darwin F_NOCACHE,让某个 fd 绕开 page cache(「代码 2」)。lib.rs 加 pub mod block; pub mod nocache; pub mod sst_flat;。
  3. 新建 src/bin/bench.rs:手写 SplitMix64、生成 1M 条数据、正确性闸门、静态指标 + readrandom(冷/热) + readmissing(「代码 3/4」)。
  4. cargo run --release --bin bench,读那张表。

代码 1 — 给两个 reader 补测量方法

// sst_flat.rs(扁平基线):整文件载入,索引每条一项
impl SstReader {
    pub fn open_cold(path: &Path) -> Result<Self> { Self::open_inner(path, true) }
    fn open_inner(path: &Path, cold: bool) -> Result<Self> {
        let mut f = File::open(path)?;
        if cold { crate::nocache::set_nocache(&f); } // 冷读:先关缓存,再 read_to_end 真打盘
        let mut raw = Vec::new(); f.read_to_end(&mut raw)?;
        let bytes_read = raw.len() as u64; // 扁平读路径:open 就吞下整文件
        /* …footer/index 解析同 Phase D… */
        Ok(Self { raw, index, bytes_read })
    }
    /// 常驻 RAM 索引:每条 entry 一项 = key 字节 + 8(offset)
    pub fn index_bytes(&self) -> usize { self.index.iter().map(|(k,_)| k.len()+8).sum() }
    pub fn bytes_read(&self) -> u64 { self.bytes_read }
}

// sst.rs(块化):稀疏索引,每块一项;read_block 里累加 bytes_read
impl SstReader {
    pub fn open_cold(path: &Path) -> Result<Self> { Self::open_inner(path, true) } // fd 设 F_NOCACHE
    /// 稀疏索引:每块一项 = 末键字节 + 16(BlockHandle)
    pub fn index_bytes(&self) -> usize { self.index.iter().map(|(k,_)| k.len()+16).sum() }
    pub fn bytes_read(&self) -> u64 { self.bytes_read.load(Ordering::Relaxed) }
}

代码 2 — nocache.rs:手写 FFI 关掉 page cache

// 让某个文件 fd 绕开 OS page cache,后续每次读都真打磁盘。仅 bench 用。
// 不引 libc 依赖——自己声明 extern "C" fcntl。
#[cfg(target_os = "macos")]
mod imp {
    use std::os::unix::io::AsRawFd;
    const F_NOCACHE: i32 = 48; // <sys/fcntl.h> Darwin:关闭该 fd 的数据缓存
    unsafe extern "C" { fn fcntl(fd: i32, cmd: i32, arg: i32) -> i32; }
    pub fn set_nocache<F: AsRawFd>(f: &F) {
        unsafe { fcntl(f.as_raw_fd(), F_NOCACHE, 1); } // 失败(-1)就当冷读不可用,不影响正确性
    }
}
#[cfg(not(target_os = "macos"))]
mod imp { /* Linux 用 posix_fadvise(DONTNEED);此处教学只覆盖 macOS,其余 no-op */
    use std::os::unix::io::AsRawFd;
    pub fn set_nocache<F: AsRawFd>(_f: &F) {}
}
pub use imp::set_nocache;
为什么 F_NOCACHE 而不是 purge? 要测"冷读"就得让 get 真打磁盘,而非命中 page cache。macOS 的 purge 会清全系统缓存——又慢又吵(别的进程立刻把缓存填回来)。fcntl(fd, F_NOCACHE) 是外科手术式的:只对这一个文件关缓存,不用 sudo,可复现。Linux 没有等价 per-fd 开关,真做要 posix_fadvise(POSIX_FADV_DONTNEED)。

代码 3 — bin/bench.rs:SplitMix64 + 数据生成 + 正确性闸门

use rocksdb_mini::{sst, sst_flat};

// 手写 SplitMix64:零依赖、可复现。同一种子喂两个格式,A/B 无偏。
struct SplitMix64(u64);
impl SplitMix64 {
    fn next(&mut self) -> u64 {
        self.0 = self.0.wrapping_add(0x9E3779B97F4A7C15);
        let mut z = self.0;
        z = (z ^ (z >> 30)).wrapping_mul(0xBF58476D1CE4E5B9);
        z = (z ^ (z >> 27)).wrapping_mul(0x94D049BB133111EB);
        z ^ (z >> 31)
    }
    fn below(&mut self, n: u64) -> u64 { self.next() % n }
}

const SEED: u64 = 0x6d696e69_6c736d70;  // 打印出来便于复现
const N: u64 = 1_000_000;               // db_bench 默认量级
fn key_of(i: u64) -> [u8; 16] { /* "{i:016}":16B 定宽,字典序==数值序 */ }
fn val_of(i: u64) -> Vec<u8> { /* 定长 100B */ }

// 正确性闸门:不绿不许出数。flat == block == oracle。
let f = sst_flat::SstReader::open(&flat_path)?;
let b = sst::SstReader::open(&block_path)?;
for _ in 0..50_000 {                          // 5 万随机命中
    let i = r.below(N); let k = key_of(i); let want = val_of(i);

}

实测结果(本机 · macOS · 1M×116B · block=4KB)

你的绝对数字会因机器而异,但三条结论的方向应当一致。

① 静态:磁盘 + 索引内存(H1)

指标flat(Phase D)block(Phase E)读这一行
file size on disk152.0 MB129.5 MB块化已小 ~15% ——扁平把全量索引(24MB)也写进文件;稀疏索引只占 ~1MB
index bytes in RAM24.0 MB1.0 MB24× 更小:每条一项 → 每块一项
data blocks—31 2501M 条 / ~4KB 块

意外但关键:"块化省磁盘"这件事现在就成立了,还没上压缩!原因不在数据区,而在索引:扁平格式把"每条 entry 一个索引项"也落进了文件(24MB),块化的稀疏索引只有 ~1MB。所以磁盘 152→129.5MB 的差,基本就是索引瘦身。数据区本身的压缩赢面,要到 E5 才叠加上来。

② 热点查 readrandom / readmissing(page cache 热)

负载flatblock读这一行
readrandom(命中)~1190 ns/op~1440 ns/op块化反而慢:每次 pread+解块 vs 扁平纯内存偏移
readmissing(全不存在)~58 ns/op~45 ns/op块化的越界 miss 一块都不读,稀疏索引上二分极快

这就是要诚实印出来的那一栏。热点查里扁平赢——它整文件常驻 RAM,get 就是一次二分 + 一次偏移解码,没有 IO、没有解块。块化每次 get 都重读+重解一个块。这不是 bug,是块化的固有代价:它把"全载内存"换成了"按需 IO + 有界内存"。补回热点性能正是 Phase S(block cache) 的活——把解析好的热块缓存在用户态,块化才能在热点查上追平甚至反超。

③ 冷读 H2:open + Q 次点查的总墙钟(F_NOCACHE)

Q(随机点查数)flat 总时 / 读盘block 总时 / 读盘谁赢
1 00033 ms / 152 MB1 ms / 4.1 MBblock ~33×
10 00045 ms / 152 MB17 ms / 41 MBblock ~2.6×
100 000161 ms / 152 MB174 ms / 410 MBflat 反超

交叉线是这节课最有信息量的图. flat 每次冷 open 都得吞下整个 152MB 文件(无论你只查 1 个还是 10 万个 key);block 只 pread 命中的那些块。所以:

  • Q 小(选择性读):block 完胜——查 1000 个 key 只搬 4MB,flat 却要搬 152MB。
  • Q 大(逼近全量扫):flat 反超——它整文件读一次就够;block 每个 get 重读一块,同一热块被重复 pread(100k 次查搬了 410MB > 文件本身!)。

那个"410MB > 152MB 文件"正是 没有 block cache 的代价:重复读同一批块。这条交叉线同时给出了后续两个 phase 的动机:bloom(让 miss 连块都不读)、block cache(让热块不重复读盘)。

🧰 Coding style:extern "C" FFI · cfg 平台门 · black_box · median-of-N

  • 手写 extern "C" FFI:不引 libc crate,自己声明 fcntl 的签名 + Darwin 常量 F_NOCACHE=48。这是"只为一个 syscall 付一行 FFI"的范式——任何语言 bind 到 C ABI 都是这套(声明签名、unsafe 调用、自己负责正确性)。
  • #[cfg(target_os)] 平台门:macOS 走 F_NOCACHE,其它平台 no-op。把平台差异收敛在一个模块边界内,调用方(bench)无感。这是跨平台代码的标准切法。
  • std::hint::black_box:基准里最隐蔽的坑是编译器把"没用到结果的循环"整段优化掉,于是你测了个 0ns。black_box 告诉优化器"假装这个值被用了",强制真跑。任何微基准都要它。
  • median-of-N 而非单次/均值:笔记本上有调度抖动、温度墙、后台进程。取中位数比均值抗离群点(一次 GC/抢占就能把均值拉飞)。再配固定种子,结果可复现。

🧠 System mindset:page cache · 冷热分离 · 摊薄 · 诚实测量

概念本阶段怎么体现 / 跨域同构
page cache 冷热同一份代码,缓存热 vs 冷,结论相反(热:flat 赢;冷选择性:block 赢)。不区分冷热的基准是无意义的。同构:CPU cache 命中/失效、JIT 预热、CDN 冷启动、数据库 buffer pool 命中率。
摊薄(amortization)flat 的 open 成本(读整文件)固定,Q 越大越被摊薄;block 的成本随 Q 线性。两条成本曲线一定有交叉。这是"固定开销 vs 边际开销"的经典权衡——批处理、连接池、JIT 编译都在算这笔账。
读放大可量化bytes_read 把"搬了多少字节"变成数字。block 在 Q=100k 搬了 410MB(>文件),一眼看出"重复读同块"的浪费——这就是 cache 的定量动机,不是拍脑袋。

迁移收获:做任何性能声称前,先问"冷还是热?选择性读还是全量扫?固定开销还是边际开销?"。一个数字脱离这三问就是误导。块化 SST 是个完美教具:它在一半场景赢、一半场景输,逼你说清"赢在哪、为什么"。

🧪 Harness 系统:A/B 差分闸门 + 测量方法学

为什么这样验

基准最危险的失败模式是"快但是错"——一个跳过了某些工作的 reader 当然快,但它返回错值。所以 bin/bench 在出任何数字之前先过一道正确性闸门:flat.get == block.get == val_of(i) 在 5 万命中 + 1 万 miss 上全相等,不绿就 panic、一个数字都不印。其次是"what + why"双层指标:延迟(what)告诉你快了多少,bytes_read/blocks_read(why)告诉你为什么——410MB 那个数字让"块化在全量扫上输"从现象变成可解释的机理。光有秒表的基准只能让你惊讶,配上计数器才能让你理解。

核心问题(自己答)

E3 想透这几个
  1. 没压缩为何磁盘已经小了? 块化数据区和扁平几乎一样大(都是 1M 条原始字节),可文件却小了 15%。省在哪?(提示:扁平的全量索引也在文件里。块化把它换成稀疏索引——磁盘和内存同时瘦。压缩省的是数据区,那是 E5 的事。)
  2. 热点查块化为何反而慢? 同样命中,block ~1440ns vs flat ~1190ns。慢在哪一步?(提示:flat 整文件常驻 RAM,get = 二分 + 偏移;block 每次 read_exact_at 重读一块 + Block::decode 重解 offsets。谁能省掉"重读+重解"?→ Phase S 的 block cache 缓存解析好的 Block。)
  3. 冷读交叉点说明什么? Q=1000 时 block 33× 快,Q=100k 时 flat 反超。这条交叉线在工程上意味着什么?(提示:LSM 的真实读负载是高度选择性的——配合 bloom + cache,你几乎只读真正需要的那几块。flat 的"整文件常驻"在小数据集上能赢,但不可扩展:文件 1TB 时它根本载不进内存。块化是为可扩展而生。)
  4. 410MB > 152MB 文件,意味着什么浪费? 10 万次冷点查,block 搬了 410MB,比整个文件还多。为什么?怎么消除?(提示:同一个热块被重复 pread。一个 LRU block cache 让第二次起命中内存——Phase S 正是要把这 410MB 压回去。)

下一步:E4 — restart points + 前缀键压缩. E1 里我们 restart_interval=1、把前缀压缩"留作后续"。E4 把它兑现:块内每隔 K 条设一个 restart 点存完整 key,其余 entry 只存"与前一条共享的前缀长度 + 后缀"。对排好序、前缀相似的 key(如 user:00001…),块更小(数据区第一次缩水),块内查找仍能二分(靠 restart 数组)。对标 block_builder.cc 的 restart 机制。完成后回到 E3 重跑基准,看磁盘那一栏继续下降。
