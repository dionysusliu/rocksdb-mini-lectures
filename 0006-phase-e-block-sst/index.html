Phase E — block-based SST(零拷贝读的地基)

M1 MVP 已闭环,但 Phase C/D 的 SST 有两个粗暴假设:每条 entry 一个索引项(索引和数据一样大)、整文件一次性读进内存(raw: Vec<u8>)。这一阶段把 SST 改造成真正 RocksDB 的形状——数据切成块(block)、稀疏索引每块一条、点查只读那一块。读放大从 O(整文件) 降到 O(一块);并第一次确立 零拷贝:块作为一个 bytes::Bytes 载入,查找返回借入这段缓冲的 &[u8]。这是 Phase F(迭代器)、Phase G(归并迭代器)要踩的地基。

Mission 锚点. "块"是存储引擎与操作系统之间的共同货币:磁盘按扇区、文件系统按 block、OS page cache 按页(4KB)、数据库按 page——读写的最小单位都不是"一条记录"而是"一块"。RocksDB 的 block_size 默认 4KB,正是为了对齐这一层硬件/OS 现实。本阶段让我们的 SST 也长出"块"这个单位,以及定位块的 BlockHandle 与稀疏索引,对标 RocksDB 的 BlockBuilder / BlockHandle / IndexBuilder。
本 Lab 宣称的 GOAL
  1. 块格式:BlockBuilder(升序 add → finish 成 Bytes)+ Block(get(&[u8]) -> Option<&[u8]>,块内二分,返回借入块缓冲的切片)。块布局 = entries + offsets 数组 + 末尾 count。
  2. block-based SST + 稀疏索引:SstBuilder 把 entry 攒进当前块,超 BLOCK_SIZE 就封口成 data block、记一条 (末键, BlockHandle) 稀疏索引;finish 写 [data blocks][index block][footer]。SstReader 只载入稀疏索引,get 二分定位唯一候选块、只读那一块。

Done when: cargo test 全绿,含零拷贝借用(返回切片指向块缓冲)、读放大不变量(一次点查只读一块)、跨块边界键、差分 vs BTreeMap。引擎旧测试不动照过(块化完全封装在 SST 模块内)。

🏁 本 Lab 不改对外行为,改的是 SST 的"骨相":get 结果与 Phase D 一字不差,但读放大和内存占用从此走对了路。这是性能弧线的起点(Phase S block cache、Phase F 零拷贝迭代都建在它上面)。

harness 系统:压缩 round-trip + CRC 故障注入 + 压缩率/读 CPU 权衡

本 Lab 的 harness 主题:不变量(invariant)与读放大

块化改造的危险在于:对外行为必须一字不变,内部布局却天翻地覆。差分(vs BTreeMap)证明"结果没变";而真正新增的检验是读放大不变量——我们给 SstReader 埋一个 blocks_read 计数器,断言"无论文件被切成多少块,一次点查最多读一块"。这把一个性能契约变成了可执行的测试,而不是口头承诺。

Harness 系统抓什么用在
round-tripbuild→finish→decode 后每条可读、顺序不变、空 value 不丢E1
零拷贝借用断言get 返回的切片指针必须落在块缓冲地址区间内(是借用,不是新分配的副本)E1
读放大不变量多块文件,一次点查 blocks_read 增量恰为 1(miss 越界则为 0)E2
差分 vs oracle + 跨块边界强制多块,SST.get == BTreeMap.get;每块首键/末键都命中(partition_point 临界)E2
带着这个问题进 E1:为什么块的 offsets 数组要放在块尾、而且末尾还要再写一个 count?(提示:写的时候 entry 是流式追加、长度未知;读的时候要能反向定位到 offsets 数组的起点。想想为什么 ZIP 的中央目录也在文件尾、为什么 SST footer 在文件尾——"定长尾巴指向变长正文"是个反复出现的格式设计套路。)
