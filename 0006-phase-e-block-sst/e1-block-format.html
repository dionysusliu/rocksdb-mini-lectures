STEP E1 · Phase E block-based SST

块格式 + 零拷贝读

新建 block.rs,造出 SST 的最小 I/O 单位——块(block)。一个块把若干条有序 (key,val) 打包成一段连续字节;读时整块作为一个 bytes::Bytes 载入,块内查找返回借入这段缓冲的 &[u8]。这一步先把块本身做对、做"零拷贝",E2 再用它拼成稀疏索引的 SST。

GOAL

写出 BlockBuilder:add(key,val) 升序追加、finish() -> Bytes 倒出整块;以及 Block:decode(Bytes) 解析、get(&[u8]) -> Option<&[u8]> 块内二分并返回借入块缓冲的切片。

Done when: round-trip(build→decode 全键可读、空 value 不丢)绿;零拷贝断言(返回切片指针落在块缓冲区间内)绿。

块布局:定长尾巴指向变长正文(格式锚)

块的字节布局只有三段,关键在于顺序:

为什么 offsets 在尾部、还要再缀一个 count?因为写是流式的——逐条 add 时整块多大、有几条都还不知道;而读需要随机定位第 i 条。把"目录"(offsets)推到正文之后、再用一个定长的尾巴(count)指向它,读者就能从文件尾反向解析。这正是 RocksDB 块尾 restarts[] + num_restarts 的结构,也是 ZIP 中央目录、SST footer 反复出现的同一套路。

代码 1 — BlockBuilder

use bytes::Bytes;

/// 块构建器:按 key 升序 add,finish 倒出一段可写盘的 Bytes。
pub struct BlockBuilder {
    buf: Vec<u8>,      // entries 区
    offsets: Vec<u32>, // 每条 entry 的起始偏移
}
impl BlockBuilder {
    pub fn new() -> Self { Self { buf: Vec::new(), offsets: Vec::new() } }

    /// 追加一条 entry。调用者保证整体升序(由 SstBuilder 统一把关)。
    pub fn add(&mut self, key: &[u8], val: &[u8]) {
        self.offsets.push(self.buf.len() as u32);
        self.buf.extend_from_slice(&(key.len() as u32).to_le_bytes());
        self.buf.extend_from_slice(key);
        self.buf.extend_from_slice(&(val.len() as u32).to_le_bytes());
        self.buf.extend_from_slice(val);
    }

    pub fn is_empty(&self) -> bool { self.offsets.is_empty() }

    /// 估算 finish 后的字节数:entries + offsets 数组 + 末尾 count。E2 用它判断块满。
    pub fn estimated_size(&self) -> usize { self.buf.len() + self.offsets.len() * 4 + 4 }

    /// 倒出整块:[entries][offsets][count]。消费 self。
    pub fn finish(self) -> Bytes {
        let mut buf = self.buf;
        for off in &self.offsets { buf.extend_from_slice(&off.to_le_bytes()); }
        buf.extend_from_slice(&(self.offsets.len() as u32).to_le_bytes());
        Bytes::from(buf)
    }
}
impl Default for BlockBuilder { fn default() -> Self { Self::new() } }

代码 2 — Block(零拷贝点查的核心)

/// 解析好的只读块。data 持整块字节;查找返回借入 data 的切片(零拷贝)。
pub struct Block {
    data: Bytes,
    offsets: Vec<u32>, // 从尾部 offsets 数组解析出的每条 entry 偏移
}
impl Block {
    pub fn decode(data: Bytes) -> Self {
        let n = data.len();
        let count = u32::from_le_bytes(data[n - 4..n].try_into().unwrap()) as usize;
        let off_start = n - 4 - count * 4;          // 反推 offsets 数组起点
        let mut offsets = Vec::with_capacity(count);
        for i in 0..count {
            let p = off_start + i * 4;
            offsets.push(u32::from_le_bytes(data[p..p + 4].try_into().unwrap()));
        }
        Self { data, offsets }
    }

    pub fn len(&self) -> usize { self.offsets.len() }
    pub fn is_empty(&self) -> bool { self.offsets.is_empty() }

    /// 取第 i 条 entry 的 (key, val),均为借入 data 的切片(零拷贝)。
    pub fn entry_at(&self, i: usize) -> (&[u8], &[u8]) {
        let off = self.offsets[i] as usize;
        let kl = u32::from_le_bytes(self.data[off..off + 4].try_into().unwrap()) as usize;
        let kstart = off + 4;
        let key = &self.data[kstart..kstart + kl];
        let vstart = kstart + kl;
        let vl = u32::from_le_bytes(self.data[vstart..vstart + 4].try_into().unwrap()) as usize;
        let val = &self.data[vstart + 4..vstart + 4 + vl];
        (key, val)
    }

    /// 块内二分点查:命中返回借入本块缓冲的 value 切片(零拷贝)。
    /// 返回值生命周期绑定到 &self —— value 只在 Block 活着时有效。
    pub fn get(&self, key: &[u8]) -> Option<&[u8]> {
        let idx = self.offsets
            .binary_search_by(|&off| {
                let off = off as usize;
                let kl = u32::from_le_bytes(self.data[off..off + 4].try_into().unwrap()) as usize;
                self.data[off + 4..off + 4 + kl].cmp(key)
            })
            .ok()?;
        Some(self.entry_at(idx).1)
    }
}
借用检查的小关卡. get 里 self.offsets.binary_search_by(|&off| ... self.data ...) 同时借了 self.offsets 和 self.data——但都是不可变借用,且是不同字段,编译器放行。如果哪天你想在闭包里改 self 就会爆。这也是为什么"读"路径天然好并行——全是共享只读借用。

🧰 Coding style:Builder 模式 · finish 消费 self · 类型签名即契约

  • Builder 模式(可变累积 → 不可变成品):BlockBuilder 是个一次性、可变的攒数据容器,finish(self) 消费它、吐出不可变的 Bytes。finish 取 self(非 &self)是关键——它在类型层面保证"封口后不能再 add",误用直接编译不过。这套"mutable builder → immutable artifact"在 protobuf、String::from、各种 ...Builder 里反复出现。
  • 用生命周期把"零拷贝"写进签名:fn get(&self, key) -> Option<&[u8]> 里返回的 &[u8] 默认借用 &self。这一行签名就是一份契约:返回值不拥有数据,只是借块的缓冲——块一 drop,切片就失效,编译器替你盯着。对比 Phase D 的 get -> Option<Bytes>(拥有、拷贝)。"返回借用还是拥有",是 API 设计里最影响性能的一个二选一。
  • 偏移数组 = 把"随机访问"外挂到顺序存储上:entries 是顺序流,本身只能从头扫;额外一个 offsets: Vec<u32> 就让它支持 O(log n) 二分。这是"用一个小索引换随机访问能力"的通用手法(等同 CSV + 行偏移索引、mmap 文件 + 偏移表)。

🧠 System mindset:块 = 存储栈的通用货币 · 对齐 page

"为什么是块,不是一条条记录?"——因为从硬件到 OS,数据搬运的最小单位从来不是"一条":

层"块"在这层叫什么 / 多大
硬件磁盘扇区(512B/4KB)、SSD page(4–16KB)/ erase block;读小于一扇区也要搬整扇区
操作系统虚拟内存页(4KB)、page cache 以页为单位换入换出;read() 命中的是页
文件系统ext4/xfs 的 block(默认 4KB);文件按 block 分配
数据库B-tree 的 page(Postgres 8KB、InnoDB 16KB);LSM 的 data block(RocksDB 默认 4KB)
网络TCP MSS / 以太网 MTU(1500B);发小于一个包也占一个包

迁移收获:块大小是一组对冲的旋钮。块越大 → 索引越稀疏、顺序吞吐越高、压缩率越好,但点查的读放大越大(为读一个 key 要搬整块)、cache 粒度越粗。块越小 → 点查省、但索引膨胀、每块的固定开销(checksum/handle)占比升高。RocksDB 默认 block_size=4KB 不是巧合——它对齐 OS page,让一次块读尽量正好是一次 page cache 命中。见到任何"按块/页/包分批"的系统,先问:块多大?对齐了下层的什么单位?读写放大各是多少?

🧪 Harness 系统:零拷贝借用断言(指针落点)

为什么这样验

"零拷贝"是个容易口头声称、实际没做到的性质——把 &[u8] 偷偷 .to_vec() 一下,类型签名没变、测试照过,但你已经多拷了一份。要把它钉死成可执行断言,就检查指针落点:get 返回切片的地址,必须落在块缓冲 data 的 [as_ptr(), as_ptr()+len()) 区间内。落在区间内 = 它确实借的是块的内存,不是新分配的副本。这类"用地址/身份而非值来验证 aliasing"的 harness,在验证 arena 分配、mmap 视图、Cow 是否真的没克隆时都通用。

代码 3 — 两个 harness(block.rs 的 #[cfg(test)] mod tests)

    // round-trip:build → finish → decode,逐条可读,顺序保持。
    #[test]
    fn block_roundtrip() {
        let mut b = BlockBuilder::new();
        b.add(b"apple", b"red");
        b.add(b"banana", b"yellow");
        b.add(b"cherry", b""); // 空 value
        let block = Block::decode(b.finish());
        assert_eq!(block.len(), 3);
        assert_eq!(block.get(b"banana"), Some(&b"yellow"[..]));
        assert_eq!(block.get(b"cherry"), Some(&b""[..]));
        assert_eq!(block.get(b"missing"), None);
        assert_eq!(block.get(b"zebra"), None); // 大于所有
    }

    // 零拷贝不变量:get 返回的切片必须指向块缓冲内部,不是新分配的副本。
    #[test]
    fn get_borrows_into_block_buffer() {
        let mut b = BlockBuilder::new();
        b.add(b"k1", b"value-one");
        b.add(b"k2", b"value-two");
        let block = Block::decode(b.finish());
        let v = block.get(b"k2").unwrap();
        let buf_start = block.data.as_ptr() as usize;     // 同模块测试可读私有字段
        let buf_end = buf_start + block.data.len();
        let v_ptr = v.as_ptr() as usize;
        assert!(v_ptr >= buf_start && v_ptr < buf_end,
            "value 切片必须借入块缓冲(零拷贝),而非独立分配");
        assert_eq!(v, b"value-two");
    }

核心问题(自己答)

E1 想透这几个
  1. offset 用 u32 的隐含上限:offsets 是 u32,意味着单个块最大 ~4GB。这对 4KB 的块绰绰有余。但如果某条 value 本身就 5GB,会发生什么?(提示:它会被塞进一个块,offset as u32 溢出。真引擎怎么处理"超大 value"?搜 RocksDB BlobDB / value separation。)
  2. 为什么 restart_interval=1 是"诚实的简化"而非"错":我们每条 entry 都存整 key、每条一个 offset。RocksDB 默认每 16 条才存一次整 key(restart point),中间用前缀压缩。这只影响空间/速度,不影响 get 的正确性。想想:加上前缀压缩后,块内二分还能直接比 key 吗?为什么二分点必须落在 restart point 上、然后再线性扫?(这是 Phase 后期的性能优化,先理解它为什么能晚做。)
  3. decode 的信任边界:Block::decode 无脑信任 count 和 offsets 合法。如果块字节被磁盘 bit-rot 翻了一位,count 变成天文数字,off_start = n-4-count*4 会下溢 panic。这暴露了什么缺失?(提示:per-block checksum,Phase AC。现在我们假设"写进去的字节读出来不变"——这个假设在真磁盘上不成立。)
  4. 二分 vs 哈希:块内我们用二分(要求 entry 有序)。如果块内改用哈希表,点查 O(1) 但失去范围扫描能力。LSM 为什么宁要二分(有序)也不要哈希?(提示:Phase F 的迭代器、范围查询 scan(a..b) 全押在"块内有序"上。)
