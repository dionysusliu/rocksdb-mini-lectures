STEP E5 · Phase E block-based SST

块尾:per-block CRC + 压缩(lz4)

E4 的前缀压缩吃 key 的结构相似性,对 value 里的统计冗余无能为力。E5 给每块加 RocksDB 式块尾 [type:1][crc32:4]:type 记压缩算法,crc 在读时校验完整性;用 lz4_flex 把整个数据块压一遍。磁盘赢面在此叠满;代价是读路径每块多付解压 + 校验的 CPU——回 E3 把这笔账量化出来。这是 Phase E 的收官:SST 终于长出真 RocksDB 的块尾(BlockTrailer)与可校验、可压缩的骨相。

GOAL

在 sst.rs 加两个函数:seal_block(raw, compress) -> Vec<u8>(压缩 → 加 type → 加 crc32)与 unseal_block(disk) -> Result<Bytes>(校验 crc → 解压)。flush_block / 索引落盘走 seal,read_block / 索引载入走 unseal。BlockHandle.len 改为磁盘上整块长度(含 trailer)。加 set_compress 开关。

Done when: seal_unseal_roundtrip(可压数据 lz4 更小、还原一致)绿;crc_catches_corruption(翻转盘上一字节 → 读该块报错、不返回错值)绿;引擎 + 既有 SST 测试全过;回 E3 重跑,磁盘断崖式下降、热点查变慢(诚实记下解压成本)。

磁盘上的一个块:payload + 块尾(格式锚)

为什么 crc 先验、再解压? 顺序是安全要点。若先解压再校验,一段损坏的压缩数据可能让解压器读出天量长度、疯狂分配、甚至 panic——你在未验真的字节上跑了复杂逻辑。先 crc 校验,等于"验真之后才信任":crc 不符立即拒绝,解压器永远只见到完好数据。这跟"先验签名再解析报文""先校验长度再 memcpy"是同一条铁律——不可信输入,先校验,后处理。

table/format.h:BlockTrailerSize=5(1 type + 4 crc)· table/block_based/block_based_table_builder.cc:WriteRawBlock(compress → trailer)· block_fetcher.cc:VerifyChecksum 先于解压

步骤

  1. Cargo.toml 加 lz4_flex = "0.11"(纯 Rust,无 C 工具链)。crc32fast 早在依赖里、终于用上。
  2. sst.rs 加 seal_block / unseal_block 两个自由函数(「代码 1」),以及压缩 type 常量 + TRAILER_LEN=5。
  3. SstBuilder 加 compress: bool 字段(默认 true)+ set_compress;flush_block 与索引落盘改走 seal_block,handle.len = sealed 长度(「代码 2」)。
  4. SstReader 的 read_block 与 open 载索引改走 unseal_block(「代码 3」)。
  5. 敲入「代码 4」harness(round-trip + CRC 故障注入),cargo test 全绿;cargo run --release --bin bench 看磁盘断崖 + 热读变慢。

代码 1 — seal_block / unseal_block

use anyhow::{anyhow, bail};

const COMP_NONE: u8 = 0;
const COMP_LZ4: u8 = 1;
const TRAILER_LEN: usize = 5; // type(1) + crc32(4)

/// 原始块 → 磁盘形态:可选压缩 → 加 type → 加 crc32(覆盖 payload+type)。
fn seal_block(raw: &[u8], compress: bool) -> Vec<u8> {
    let (mut out, ctype) = if compress {
        (lz4_flex::compress_prepend_size(raw), COMP_LZ4) // 前 4B 记原始长度,解压据此还原
    } else { (raw.to_vec(), COMP_NONE) };
    out.push(ctype);
    let mut h = crc32fast::Hasher::new();
    h.update(&out); // crc 覆盖 payload + type
    out.extend_from_slice(&h.finalize().to_le_bytes());
    out
}

/// 磁盘形态 → 原始块:先验 crc(损坏即报错)→ 按 type 解压。
fn unseal_block(disk: &[u8]) -> Result<Bytes> {
    if disk.len() < TRAILER_LEN { bail!("block 太小,无 trailer"); }
    let body = &disk[..disk.len() - 4];            // payload + type
    let crc_stored = u32::from_le_bytes(disk[disk.len()-4..].try_into()?);
    let mut h = crc32fast::Hasher::new(); h.update(body);
    if h.finalize() != crc_stored { bail!("block crc 不符:数据损坏"); } // 先验真
    let ctype = body[body.len()-1];
    let payload = &body[..body.len()-1];
    let raw = match ctype {                       // 后处理
        COMP_NONE => payload.to_vec(),
        COMP_LZ4  => lz4_flex::decompress_size_prepended(payload).map_err(|e| anyhow!("lz4: {e}"))?,
        other     => bail!("未知压缩 type: {other}"),
    };
    Ok(Bytes::from(raw))
}

代码 2 — 写路径:flush_block + 索引落盘 seal

    fn flush_block(&mut self) {
        if self.block.is_empty() { return; }
        let block = std::mem::take(&mut self.block).finish();
        let sealed = seal_block(&block, self.compress);  // 压缩 + type + crc
        let offset = self.data.len() as u64;
        let len = sealed.len() as u64;             // handle.len = 磁盘整块(含 trailer)
        self.data.extend_from_slice(&sealed);
        let last = self.last_key.clone().unwrap();
        self.index.push((last, BlockHandle { offset, len }));
    }
    // finish 里索引块也 seal:
    let sealed_index = seal_block(&index_block, self.compress);
    f.write_all(&sealed_index)?;

代码 3 — 读路径:read_block + 索引载入 unseal

    fn read_block(&self, h: BlockHandle) -> Result<Block> {
        let mut buf = vec![0u8; h.len as usize];
        self.file.read_exact_at(&mut buf, h.offset)?;
        self.blocks_read.fetch_add(1, Ordering::Relaxed);
        self.bytes_read.fetch_add(h.len, Ordering::Relaxed);
        Ok(Block::decode(unseal_block(&buf)?))  // 校验 crc + 解压 → 解块
    }
    // open 里索引载入:
    let index_block = Block::decode(unseal_block(&ibuf)?);
读路径对损坏的反应是"降级为 miss",不是崩。 get 里 read_block(*handle).ok()? —— unseal 报错(crc 不符)被 .ok()? 转成 None。所以一个损坏块的效果是"这个 key 读不到",而不是返回错值、也不是 panic 整个进程。真 RocksDB 会把 corruption 作为 Status::Corruption 上抛由上层决定(重试别的副本 / 报警),但"宁可没有,不可错有"的原则一致——校验的全部意义就是不让错数据冒充对数据。

实测:磁盘断崖,读 CPU 上账

同一份 1M×116B 数据,data block 开 lz4,再跑 E3 基准:

指标E4(前缀压缩,无块压缩)E5(+lz4 块压缩)读这一行
file size on disk116.1 MB56.9 MB对有冗余的 value,块压缩再砍一半
vs flat(152MB)−24%−63%索引瘦身 + 前缀压缩 + lz4 三层叠加
readrandom 热~1440 ns/op~2867 ns/op每次 get 多付一次 lz4 解压(无 cache → 每块都解)
冷读 Q=1000 字节4.1 MB2.0 MB块更小 → 冷读搬的字节也更少
冷读 Q=100k 总时174 ms294 ms重复读同块时,解压 CPU 盖过省下的 IO

压缩是一笔账,不是免费午餐。磁盘从 152MB 砍到 56.9MB(−63%),但热点查从 1440ns 涨到 2867ns——每次 get 都要 lz4 解一整块。冷读小 Q 仍赢(搬的字节更少),但大 Q(重复读同块)解压 CPU 反客为主,294ms > E4 的 174ms。这条账精确地指向 Phase S block cache:把解压后的块缓存在用户态,热块只解一次,压缩的磁盘收益就能白拿、读 CPU 的代价只付一次。压缩 + 缓存是天生一对。

注:压缩率取决于 value 内容。我们的基准 value 有周期性冗余,lz4 砍得狠;若 value 是随机散列/已压缩的图片,lz4 几乎无效甚至略增(那时 set_compress(false) 更划算——RocksDB 也支持 per-level 关压缩)。诚实地说:−63% 是这份数据的数字,不是普适承诺。

🧰 Coding style:自由函数封装格式 · 先验后用 · 类型字节做版本位

  • seal/unseal 自由函数,不绑进类型:块尾是纯字节变换(&[u8] → Vec<u8>),不依赖 SstBuilder 的任何状态,就该是无副作用的自由函数——好测(直接喂字节)、好复用(data 块、index 块、将来 WAL 都能用)。"能是纯函数就别做成方法"。
  • 先验真后处理(validate-before-use):crc 校验必须在解压之前。把"信任边界"画在校验那一行:之前的字节都不可信,之后才可信。所有解析不可信输入的代码(网络协议、文件格式、反序列化)都该有这条显式边界。
  • type 字节 = 前向兼容的版本位:一个字节区分 none/lz4,未来加 zstd 只需 COMP_ZSTD=2 + 一个 match 分支,旧文件照读(type=0/1 不变)。"在格式里留一个算法标识位"是可演进格式的标配(TLS cipher suite、HTTP Content-Encoding、容器镜像的 media type 全是这招)。

🧠 System mindset:完整性 · 压缩三角 · 信任边界

概念本阶段怎么体现 / 跨域同构
端到端完整性磁盘会静默腐坏(bit rot、固件 bug、坏道)。per-block crc 在使用点校验,而非信任硬件。同构:ZFS/Btrfs 块校验、TCP checksum、Ceph scrub、Git object SHA。"数据离开你视线就可能变,回来时要验"。
压缩三角压缩把空间换成CPU(还可能换读延迟)。lz4 选"快但压缩率中等",zstd 选"慢但更小",none 选"不压但最快"。这是 LSM 读/写/空间放大三角在块层的投影。算法选择 = 在你的瓶颈维度上让步最少的那个。
信任边界crc 那一行是代码里的信任分割线。之前=不可信外部字节,之后=可信内部数据。把边界画清楚,后续逻辑(解压、解码)就能假设输入良性而不必处处设防。安全工程的"parse, don't validate"同源。

迁移收获:任何"落盘 / 过网 / 跨进程"的字节,默认会坏、值得压。一个 [payload][type][checksum] 的信封,把"压缩(省)+ 类型(可演进)+ 校验(可信)"三件事钉在一个最小单位上——这个信封结构你会在无数协议和格式里再遇到。

🧪 Harness 系统:故障注入(fault injection)

为什么这样验

校验代码有个尴尬:不主动破坏数据,它永远不触发——测试全绿可能只是因为没东西坏过。所以 CRC 这类防御代码必须配故障注入:crc_catches_corruption 主动翻转盘上一个字节,断言"读该块报错(返回 None),而非返回错值"。这是在测失败路径,而非成功路径。同源 harness:WAL 的尾部截断测试(B 阶段你做过)、网络的丢包/乱序注入、kill -9 崩溃恢复测试。"防御性代码的测试,必须主动制造它要防的那个灾难"——否则你只是部署了一个从未演练过的灭火器。再加 seal_unseal_roundtrip 守正向(可压数据 lz4 确实更小、还原一致),一正一反两面包夹。

代码 4 — harness(sst.rs tests)

    // 正向:seal→unseal 还原一致;可压数据 lz4 更小。
    #[test] fn seal_unseal_roundtrip() {
        let raw = b"the quick brown fox the quick brown fox the quick brown fox";
        let lz4 = seal_block(raw, true);
        assert!(lz4.len() < seal_block(raw, false).len());
        assert_eq!(unseal_block(&lz4).unwrap().as_ref(), raw);
    }

    // 反向(故障注入):翻转盘上一字节 → 读该块必须报错,不返回错值。
    #[test] fn crc_catches_corruption() {
        // …build SST(block_size=64,多块)…
        let mut f = OpenOptions::new().read(true).write(true).open(&path)?;
        // 翻转第 8 字节(落在第一个 data block 的 payload 内)
        let mut one = [0u8; 1]; f.read_exact_at(&mut one, 8)?;
        one[0] ^= 0xFF; f.write_all_at(&one, 8)?; f.sync_all()?;

        let r = SstReader::open(&path)?;       // 索引未损,open 成功
        assert!(r.get(b"k000").is_none(),       // 损坏块 → None,绝不返回错值
            "损坏块不得返回(错)值");
    }

核心问题(自己答)

E5 想透这几个
  1. 压缩为何让热点查更慢、却该保留? readrandom 从 1440 涨到 2867ns。既然变慢,为什么生产 LSM 默认开压缩?(提示:磁盘 / SSD 寿命 / IO 带宽 / 内存里能缓存的有效数据量都翻倍。热读的解压成本会被 block cache 几乎抹平(热块只解一次)。算总账:省一半空间 + 缓存命中后无解压 远胜"每次纯内存偏移"。)
  2. crc 放块尾还是块头? 我们放尾(payload 之后)。放头(payload 之前)行不行?(提示:写是流式的——payload 多长、crc 是多少,要等 payload 写完才知道。crc 依赖全部 payload,只能事后追加。这跟块的 offsets 数组、SST footer 同理:依赖全文的元数据只能放尾。)
  3. 先验 crc 再解压,顺序能反吗? 若先解压再验 crc 会怎样?(提示:损坏的压缩流喂给解压器 = 在未验真的字节上跑复杂逻辑,可能 OOM / panic / 读出垃圾长度。校验是信任边界,必须在任何"理解"数据的动作之前。)
  4. 压缩与前缀压缩重复吗? E4 已前缀压缩,E5 又 lz4。是不是白压两遍?(提示:不。前缀压缩吃 key 的结构(排序相邻相似),lz4 吃 整块的统计冗余(含 value)。两者正交:前缀压缩先把 key 缩小,lz4 再把"缩小后的块"整体压。对随机 key + 文本 value,前者无效后者有效;对相似 key + 随机 value,反之。留两个都开,各吃各的冗余。)

🏁 Phase E 收官. 从 E1 的块,到 E2 的稀疏索引,到 E3 的诚实基准暴露两处洞(热点慢、热块重复读),到 E4 前缀压缩让数据区缩水,到 E5 块尾压缩 + 完整性。SST 终于长出真 RocksDB 的骨相:分块、稀疏索引、前缀压缩、块尾 CRC、可选块压缩。磁盘 −63%、索引内存 27×、读放大 O(一块)——而每一项都有 bench 数字背书,每一处代价都被诚实记下。
