STEP E4 · Phase E block-based SST

restart points + 前缀键压缩

E1 里我们留了一句"restart_interval=1,前缀压缩留作后续优化"。E3 的基准告诉我们:块化省了磁盘,但省的是索引,数据区还没动。E4 兑现这句承诺——块内每 K 条设一个 restart 点存完整 key,其余 entry 只存与前一条共享的前缀长度 + 后缀。排好序的 key 前缀高度相似(0000…0001、0000…0002),数据区第一次缩水;块内查找靠 restart 数组仍能二分。对标 RocksDB block_builder.cc 的 restart 机制——这是 RocksDB 块格式的签名招式。

GOAL

重写 block.rs 的 BlockBuilder/Block:entry 布局换成 [shared][non_shared][vlen][key 后缀][val],块尾从 offsets[] 换成 restarts[](只记 restart 点偏移)+ num_restarts。add 算前缀共享、每 K 条强制一个 restart;get 在 restarts 上二分定位段、段内线性扫并重建 key。

Done when: 前缀重建差分(vs BTreeMap,强制多 restart 段)绿;restart 边界键(每段首/末、跨段)命中;引擎 + E2 既有 SST 测试全过(索引块改用 restart_interval=1,行为对 reader 透明);回 E3 重跑,磁盘那一栏继续下降。

块布局:从 offsets 到 restarts(格式锚)

为什么要 restart 点、不直接全压? 若每条都只存"与前一条的差",那要定位第 i 条就得从块头一路重建到 i,块内查找退化成 O(条数) 线性。restart 点是压缩与可二分的折中:每 K 条放一个"锚"(完整 key),restarts 数组上二分锁定到某个 K 条的小段,段内才线性扫(最多重建 K 条)。K 就是 压缩率 ↔ 查找成本的旋钮:K 越大越省(完整 key 越少),但段内线性扫越长。RocksDB 默认 K=16。

table/block_based/block_builder.cc:Add(前缀共享计算)· :Finish(append restarts[] + num_restarts)· block.cc:BlockIter::Seek(restarts 上二分 + 段内 ParseNextKey)

步骤

  1. 重写 BlockBuilder(「代码 1」):字段换成 buf + restarts + last_key + counter + restart_interval;add 算 shared_prefix、每 K 条记一个 restart。finish 缀 restarts[] + num_restarts。
  2. 重写 Block(「代码 2」):decode 改解 restarts + 算出 entries_end(末 4B 从 count 变 num_restarts);get 在 restarts 上二分定位段、段内线性扫重建 key;加 for_each_entry(单趟解全块,供索引解码)。
  3. 改 sst.rs(「代码 3」):索引块用 BlockBuilder::with_restart_interval(1)(每条都是 restart、无前缀压缩 → len()==条数);open 用 for_each_entry 解索引。data block 用默认 K=16,自动获得前缀压缩。
  4. 敲入「代码 4」harness,cargo test 全绿;cargo run --release --bin bench 看磁盘续降。

代码 1 — BlockBuilder(前缀压缩 + restart)

pub const RESTART_INTERVAL: usize = 16;

/// 两个 key 的公共前缀长度。
fn shared_prefix(a: &[u8], b: &[u8]) -> usize {
    let n = a.len().min(b.len()); let mut i = 0;
    while i < n && a[i] == b[i] { i += 1; } i
}

pub struct BlockBuilder {
    buf: Vec<u8>, restarts: Vec<u32>, last_key: Vec<u8>,
    counter: usize,            // 自上个 restart 起的条数
    restart_interval: usize,
}
impl BlockBuilder {
    pub fn new() -> Self { Self::with_restart_interval(RESTART_INTERVAL) }
    pub fn with_restart_interval(k: usize) -> Self { /* k.max(1) */ }

    pub fn add(&mut self, key: &[u8], val: &[u8]) {
        let shared = if self.counter == 0 {
            self.restarts.push(self.buf.len() as u32); // restart 点:存完整 key
            0
        } else { shared_prefix(&self.last_key, key) };
        let non_shared = key.len() - shared;
        self.buf.extend_from_slice(&(shared as u32).to_le_bytes());
        self.buf.extend_from_slice(&(non_shared as u32).to_le_bytes());
        self.buf.extend_from_slice(&(val.len() as u32).to_le_bytes());
        self.buf.extend_from_slice(&key[shared..]); // 只存后缀
        self.buf.extend_from_slice(val);
        self.last_key.clear(); self.last_key.extend_from_slice(key);
        self.counter += 1;
        if self.counter == self.restart_interval { self.counter = 0; } // 满 K 条 → 下条强制 restart
    }

    pub fn finish(self) -> Bytes { /* buf + restarts[] + num_restarts */ }
    pub fn estimated_size(&self) -> usize { self.buf.len() + self.restarts.len()*4 + 4 }
    pub fn is_empty(&self) -> bool { self.restarts.is_empty() }
}

代码 2 — Block(restart 上二分 + 段内重建)

pub struct Block { data: Bytes, restarts: Vec<u32>, entries_end: usize }

impl Block {
    /// decode 大改:末 4B 从 count(条数)变 num_restarts(restart 数);
    /// 倒数第二段从 offsets[N] 变 restarts[R];且必须算出 entries_end(restarts 数组起点)。
    pub fn decode(data: Bytes) -> Self {
        let n = data.len();
        let num_restarts = u32::from_le_bytes(data[n-4..n].try_into().unwrap()) as usize;
        let entries_end = n - 4 - num_restarts * 4;   // entries 区结束 = restarts 数组起点
        let mut restarts = Vec::with_capacity(num_restarts);
        for i in 0..num_restarts {
            let p = entries_end + i * 4;
            restarts.push(u32::from_le_bytes(data[p..p+4].try_into().unwrap()));
        }
        Self { data, restarts, entries_end }      // 不再存 offsets;存 restarts + entries_end
    }

    /// 解析从 off 起一条:(shared, non_shared, 后缀, val, 下条 off)。
    fn parse_at(&self, off: usize) -> (usize, usize, &[u8], &[u8], usize) { /* 读 3×u32 + 后缀 + val */ }

    /// restart 点的完整 key(shared==0,后缀即整 key)。
    fn restart_key(&self, ri: usize) -> &[u8] {
        let (_, _, suffix, _, _) = self.parse_at(self.restarts[ri] as usize); suffix
    }

    pub fn get(&self, key: &[u8]) -> Option<&[u8]> {
        if self.restarts.is_empty() { return None; }
        // ① restarts 上二分:找最后一个 restart_key <= key 的段
        let (mut lo, mut hi) = (0, self.restarts.len());
        while lo < hi {
            let mid = (lo + hi) / 2;
            if self.restart_key(mid) <= key { lo = mid + 1; } else { hi = mid; }
        }
        if lo == 0 { return None; }          // 比第一个 restart 还小 → 不在本块
        let seg = lo - 1;
        let start = self.restarts[seg] as usize;
        let end = self.restarts.get(seg+1).map(|&r| r as usize).unwrap_or(self.entries_end);
        // ② 段内线性扫,逐条重建 key(前缀来自上一条)
        let mut cur = Vec::new(); let mut off = start;
        while off < end {
            let (shared, _ns, suffix, val, next) = self.parse_at(off);
            cur.truncate(shared); cur.extend_from_slice(suffix); // 重建 = 上一条[..shared] + 后缀
            match cur.as_slice().cmp(key) {
                Ordering::Equal   => return Some(val),  // 命中:val 借入块缓冲(零拷贝)
                Ordering::Greater => return None,       // 已越过,本段无此 key
                Ordering::Less    => {}
            }
            off = next;
        }
        None
    }

    /// 单趟解全块,回调 (key, val)。索引块解码用(O(n),不是 O(n²))。
    pub fn for_each_entry(&self, mut f: impl FnMut(&[u8], &[u8])) {
        let mut cur = Vec::new(); let mut off = 0;
        while off < self.entries_end {
            let (shared, _ns, suffix, val, next) = self.parse_at(off);
            cur.truncate(shared); cur.extend_from_slice(suffix);
            f(&cur, val); off = next;
        }
    }
    pub fn len(&self) -> usize { self.restarts.len() } // 注意:restart 数,非条数!
}
陷阱:len() 现在是 restart 数,不是条数。 E2 的索引解码 for i in 0..index_block.len() { entry_at(i) } 默认"len==条数"。前缀压缩后这不再成立(16 条才一个 restart)。两个修复缺一不可:① 索引块改用 restart_interval=1(每条都是 restart,len()==条数 重新成立、且 entry_at O(1));② 用 for_each_entry 单趟解码,避免"对每个 i 都从头扫"的 O(n²)。数据块用默认 K=16 吃满前缀压缩,索引块用 K=1 保持可随机解——同一个 BlockBuilder,两种 K,各取所需。

实测:数据区第一次缩水

同一份 1M×116B 数据,只把 data block 的 restart_interval 从 1 改成 16(+前缀压缩),再跑 E3 的基准:

指标E2(restart=1,无前缀压缩)E4(restart=16,前缀压缩)变化
file size on disk129.5 MB116.1 MB数据区 −13MB:每条 16B key 多数只存 ~3–6B 后缀
vs flat(152MB)−15%−24%索引瘦身 + 数据前缀压缩叠加
index bytes in RAM1.00 MB0.89 MB块更密 → 块更少 → 稀疏索引更短
data blocks31 25027 778同样 4KB 阈值下,每块装下更多条

这就是 E3 留的洞之一被补上的样子。磁盘赢面从"只省索引(−15%)"扩到"索引 + 数据前缀(−24%)"。注意它对排序键特别有效——key 越像(共同前缀越长),省得越多。随机散列 key 则几乎没有前缀可压(那要靠 E5 的通用压缩)。点查路径多了"段内重建 key"的开销,但每段最多 16 条,常数级,热点查延迟基本不动。

🧰 Coding style:增量编码 · 折中旋钮 · 单趟回调避免 O(n²)

  • 增量/差分编码(delta encoding):只存"与前一条的差"是无损压缩的一种基本招。前缀压缩、git 的 packfile、视频的 P 帧、protobuf 的 varint delta、时序库的 Gorilla 编码——全是"相邻数据高度相似 → 只记差异"。restart 点则是给"纯增量流"插的随机访问锚。
  • 用一个参数表达折中(restart_interval):K 一个旋钮,同时控制"压缩率"和"查找成本",且两端极值都有用——K=16 给数据块吃满压缩,K=1 给索引块换回随机访问。同一份代码靠参数服务两种需求,比写两个 builder 干净。
  • 单趟回调而非"按 i 查":for_each_entry(FnMut) 一趟 O(n) 解全块;若保留 entry_at(i) 在前缀压缩下就是"每个 i 从头重建" = O(n²)。当数据有顺序依赖(第 i 条要靠前 i−1 条),就别提供"随机第 i 个"的假象接口——顺序消费用回调/迭代器。

🧠 System mindset:相邻相似性 · 锚点索引 · 压缩/访问的对偶

概念本阶段怎么体现 / 跨域同构
利用相邻相似性排序让相邻 key 前缀相似 → 前缀压缩才有收益。先排序再压缩是通用前置(列存按列排序后压缩、LZ 的滑动窗口、字典编码)。块化 + 排序为这一步铺好了路。
锚点 + 增量restart=完整锚,中间=增量。这是"关键帧 + 差分帧"模式:视频 I/P 帧、数据库 full + incremental backup、日志 checkpoint + delta、LSM 本身(SST=快照 + WAL=增量)。锚的密度永远是"恢复/查找速度 ↔ 体积"的旋钮。
压缩与随机访问对偶压得越狠(K 越大)→ 随机定位越贵(段内扫越长)。这对张力无处不在:gzip 不可随机 seek、zstd 的 seekable 格式靠分块、parquet 的 row group、列存的 page。"可 seek 的压缩"几乎总是靠分块 + 块内锚点——你刚亲手实现了一个最小版。

迁移收获:任何"既要小、又要能随机查"的格式,答案几乎都是分块 + 块内锚点 + 锚间增量。块大小 ↔ IO 粒度,锚间隔 ↔ 压缩/查找折中。理解了 RocksDB 的 data block,你就理解了 parquet page、zstd-seekable、视频 GOP 的同一套骨架。

🧪 Harness 系统:前缀重建差分 + restart 边界

为什么这样验

前缀压缩最容易错在重建:shared 算错一位、truncate 用错长度、跨 restart 段时忘了 key 应从完整锚重置——结果是某些 key 读出来变形。差分(vs BTreeMap)能逮住它,但前提是强制多 restart 段:测试要么调小 restart_interval(如 4),要么喂足够多条,让块里出现 ≥2 个 restart,才覆盖"跨段重置"这条最危险的路径。再加 restart 边界键(每段第一条=完整 key、每段最后一条、段与段交界)逐个命中。还要一条前缀对抗用例:["a","ab","abc","b"] —— 共享前缀递增后又跌回 0("b"),专测 truncate(shared) 在 shared 变小时是否正确截断。

核心问题(自己答)

E4 想透这几个
  1. K 怎么选? restart_interval 大 → 省空间但段内扫长,小 → 反之。RocksDB 默认 16。如果你的 value 很大(key 占比小),K 该调大还是调小?(提示:key 占比小时,前缀压缩省的那点 key 字节无足轻重,不如调大 K 减少 restart 数组开销;反之 key 长 value 短时,压缩收益大,但也要留住可二分性。)
  2. 为什么索引块不前缀压缩? data block 用 K=16,index block 偏要 K=1。除了"len()==条数"的便利,还有什么道理?(提示:索引块小(每块才一条),压它省不了几个字节;但它要支持 reader 随机/顺序解全部,K=1 让每条都能独立解。小而要随机访问的结构,不值得为压缩牺牲可访问性。)
  3. 随机 key 怎么办? 我们的 {:016} 数字 key 前缀高度相似,压得动。但若 key 是 SHA256 这种随机散列,前缀压缩几乎白干。这时磁盘还能怎么省?(提示:前缀压缩吃结构相似性,通用压缩(lz4/zstd)吃统计冗余——后者对随机散列也无能为力,但对真实 value(JSON、文本)有效。两者正交,可叠加。这正是 E5。)
  4. 零拷贝兑现了吗? Block::get 重建 key 到一个临时 Vec(owned),但 value 仍返回借入块缓冲的 &[u8]。key 为何只能 owned?(提示:压缩后 key 在缓冲里不连续(前缀在别处),没法用一个切片表示;value 字节仍连续,所以 value 零拷贝、key 必须重建。Phase F 的迭代器会复用一个 cur buf 跨条增量重建,摊薄这点开销。)

下一步:E5 — 块尾 CRC + 压缩(lz4). 前缀压缩吃的是 key 的结构相似性;对 value 的统计冗余无能为力。E5 给每块加 RocksDB 式块尾 [type:1][crc32:4]:type 记压缩算法,crc 在读时校验完整性;用 lz4_flex 压整个数据块(可开关)。磁盘赢面在此叠满(对真实 value 可再降一截),代价是读路径多付解压 + CRC 校验的 CPU——回 E3 把这笔账量化出来。
