STEP B1 · Phase B WAL

WAL record 格式 + Writer

持久化的第一性问题：磁盘上一条记录长什么样，才能在断电后被可靠地读回、且认得出自己有没有坏？这一步只设计字节布局 + 写出去。

GOAL

交付 src/wal.rs 的写端：Wal::open_append + Wal::append(key,val) + Wal::sync。一次 put 编码成一条带校验和的 record 落盘。

Done when: 故障注入 harness——翻转日志一个字节，校验和把那条 record 挡在门外。

record 字节布局

MVP 用最朴素的"自描述帧"：每条 record = 校验和 + 负载长度 + 负载本身。负载里再放一次操作。

外层帧 = crc + len + payload；payload = tag(0=Put) + klen|key + vlen|value。全部小端。crc 覆盖整个 payload。

三个设计选择，每个对标 RocksDB（这才是要学的）

选择我们 MVPRocksDB
校验和放哪每条 record 头部，覆盖 payload同理，header 头 4 字节 crc32c（log_format.h:57 kHeaderSize=4+2+1）
长度宽度u32 定长（简单）header 用 u16 长度（因为有 32KB 分块）；payload 内 key/val 用 varint（write_batch.cc:10 varstring）省空间
大record整条写，不分块32KB block + kFirst/Middle/Last 分片（log_format.h:54 kBlockSize=32768）
诚实标注简化. 我们砍掉了 RocksDB 的 32KB 分块 + record 分片(kFirst/Middle/Last)。代价是无法在损坏后"重同步到下一个 block 边界"——见 B2 核心问题。MVP 阶段值得，但要知道砍了什么。

代码 1 — src/wal.rs 写端

// src/wal.rs — write-ahead log，断电不丢的底
use std::fs::{File, OpenOptions};
use std::io::{BufWriter, Write};
use std::path::Path;
use anyhow::Result;

pub const TAG_PUT: u8 = 0; // 操作类型；将来 Delete=1（MVP 后）

pub struct Wal {
    w: BufWriter<File>, // 缓冲写：攒够再下系统调用，减少 syscall
}

impl Wal {
    /// 追加模式打开（不存在则建）。append = O_APPEND，写永远落在文件尾。
    pub fn open_append(path: &Path) -> Result<Self> {
        let f = OpenOptions::new().create(true).append(true).open(path)?;
        Ok(Self { w: BufWriter::new(f) })
    }

    /// 编码一条 Put record 并写入（尚未 fsync，见 sync）。
    pub fn append(&mut self, key: &[u8], val: &[u8]) -> Result<()> {
        // 先拼 payload
        let mut payload = Vec::with_capacity(1 + 4 + key.len() + 4 + val.len());
        payload.push(TAG_PUT);
        payload.extend_from_slice(&(key.len() as u32).to_le_bytes());
        payload.extend_from_slice(key);
        payload.extend_from_slice(&(val.len() as u32).to_le_bytes());
        payload.extend_from_slice(val);
        // 帧头 = crc(payload) + len，再写 payload
        let crc = crc32fast::hash(&payload);
        self.w.write_all(&crc.to_le_bytes())?;
        self.w.write_all(&(payload.len() as u32).to_le_bytes())?;
        self.w.write_all(&payload)?;
        Ok(())
    }

    /// 把缓冲刷给 OS，再 fsync 强制落物理介质。
    /// 这一步才是"持久"的真正边界——见核心问题 2。
    pub fn sync(&mut self) -> Result<()> {
        self.w.flush()?;            // BufWriter -> OS page cache
        self.w.get_ref().sync_all()?; // OS -> 磁盘 (fsync)
        Ok(())
    }
}

Harness 系统 ④：故障注入

为什么这里用故障注入

校验和的存在意义只有一个：检出损坏。可正常测试里数据永远不坏，校验和这段代码从不被真正考验——它可能写错了你都不知道。 办法：主动往磁盘上的字节里翻一个 bit，断言"这条 record 必须被识别为坏、不被当好数据读回"。不制造坏，就验不了"抗坏"。这是 RocksDB fault_injection_fs 的核心思想。

代码 2 — 故障注入 harness（接 wal.rs 末尾）

注：这条 harness 依赖 B2 的 Wal::replay 来"读回并验校验和"。若你先做 B1，可先写一个最小的"读帧+验crc"内联函数；或按建议把 B1+B2 连着做，本测试放完整后跑。下面假设 replay 已就位。

#[cfg(test)]
mod tests {
    use super::*;
    use tempfile::tempdir; // 已在 dev-dependencies
    use bytes::Bytes;

    // 故障注入：翻转最后一条 record 的尾字节 → crc 必须把它挡掉
    #[test]
    fn corruption_caught_at_tail() {
        let dir = tempdir().unwrap();
        let path = dir.path().join("t.log");
        {
            let mut w = Wal::open_append(&path).unwrap();
            w.append(b"good", b"1").unwrap();
            w.append(b"bad",  b"2").unwrap();
            w.sync().unwrap();
        }
        // —— 注入故障：读出原始字节，翻转最后一字节，写回 ——
        let mut raw = std::fs::read(&path).unwrap();
        let n = raw.len();
        raw[n - 1] ^= 0xFF; // 改第二条 record 的 value 字节
        std::fs::write(&path, &raw).unwrap();

        // 断言：第一条完好被读回，第二条因 crc 不符被丢
        let recs = Wal::replay(&path).unwrap();
        assert_eq!(recs.len(), 1, "损坏record没被挡住!");
        assert_eq!(recs[0], (Bytes::from("good"), Bytes::from("1")));
    }
}
