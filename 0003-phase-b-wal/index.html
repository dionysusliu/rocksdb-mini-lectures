Phase B — WAL

给引擎装上"断电不丢"。写入先落 write-ahead log，再进 MemTable；重启时重放日志恢复。Phase A 的内存表第一次有了持久的底。

Mission 锚点. 你在 learning-record 0004/0005 读过 RocksDB 写路径 WriteBatch → WAL → MemTable。 Phase A 做完了链尾的 MemTable；本 Lab 补上链中那道持久化关卡。 做完，断电后 get 仍读得到断电前 put 的值——这是存储引擎和"一个 HashMap"的分水岭。
本 Lab 宣称的 GOAL

交付一个 Engine，open(dir) / put / get 三个 API，满足：

  1. 持久：put 返回后，进程崩溃 + 重开，该写入仍在。
  2. 顺序无关正确：重放恢复后，每个 key 读到最后写入值（含覆盖写）。
  3. 残record不致命：日志尾部被撕裂（torn write）时，恢复出完整前缀、丢掉残record、不 panic。

Done when: cargo test 全绿，含崩溃恢复 harness 与故障注入 harness。

本 Lab 的 harness 主题：把"坏"主动制造出来

Phase A 验的是"正常输入下对不对"。存储引擎真正的考场是异常：断电、半截写、磁盘翻位。这类 bug 不会自己冒头——你得主动注入故障去逼它现形。两种工业级 harness：

Harness 系统制造什么坏验什么
故障注入
fault injection		 翻转日志里一个 bit / 改一个字节校验和必须抓到，损坏record不被当好数据读进来
崩溃恢复
crash-recovery		 写入后不优雅关闭就"崩"（drop + 重开）；截断日志尾模拟 torn write已 ack 的写全在；残尾丢得干净、不 panic

这两种是 RocksDB db_stress + fault_injection_test 的核心套路，纯可迁移到任何持久化系统。

带着这个问题进 B1：为什么"WAL 先落盘、再写 MemTable"这个顺序不能反？如果先写 MemTable 再写 WAL，崩在两步之间会怎样？（这条规则叫 write-ahead，是名字的由来。）
