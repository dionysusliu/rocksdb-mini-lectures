STEP B2 · Phase B WAL

Reader + 写路径 + 崩溃恢复

把字节读回来变成操作、容忍被撕裂的尾巴，再用一个 Engine 把 WAL 和 MemTable 接成一条写路径，open 时重放恢复。这一步引擎第一次活过重启。

GOAL

交付 Wal::replay（解码 + 残尾容忍）与 src/engine.rs 的 Engine{open,put,get}。

Done when: 崩溃恢复 harness——写入、drop（模拟崩溃）、重开，所有 ack 的写都在；截断尾后重开不 panic、恢复完整前缀。

写路径的铁律（mission 锚）

RocksDB 恢复逻辑 DBImpl::RecoverLogFiles 干的就是：开 log reader，一条条 ReadRecord，InsertInto memtable。你这一步是它的极简版。但顺序铁律一致：put 里 WAL 先 append+sync、MemTable 后插。崩在中间——若 WAL 已落但 MemTable 没插，重放补上；若反过来先插 MemTable，崩溃后那次写既没进 WAL 又随内存蒸发 → ack 了却丢了，违反持久性。

代码 1 — wal.rs 读端（补在写端之后）

use std::io::Read;       // 加到文件顶部 use 区
use anyhow::bail;
use bytes::Bytes;

impl Wal {
    /// 重放所有完好 record。遇到第一条残缺/损坏的尾就干净停下。
    /// 返回 (key,value) 序列，按写入顺序——覆盖写靠重放顺序自然生效。
    pub fn replay(path: &Path) -> Result<Vec<(Bytes, Bytes)>> {
        let mut buf = Vec::new();
        match File::open(path) {
            Ok(mut f) => { f.read_to_end(&mut buf)?; }
            Err(e) if e.kind() == std::io::ErrorKind::NotFound => return Ok(vec![]),
            Err(e) => return Err(e.into()),
        }
        let mut out = Vec::new();
        let mut pos = 0;
        // 帧头 8 字节(crc4+len4)读不全 → 到尾了
        while pos + 8 <= buf.len() {
            let crc = u32::from_le_bytes(buf[pos..pos+4].try_into().unwrap());
            let len = u32::from_le_bytes(buf[pos+4..pos+8].try_into().unwrap()) as usize;
            let start = pos + 8;
            if start + len > buf.len() { break; }      // 残尾：payload 没写全
            let payload = &buf[start..start+len];
            if crc32fast::hash(payload) != crc { break; } // 损坏：crc 不符（B1 的故障注入靠这行）
            let (k, v) = decode_payload(payload)?;
            out.push((k, v));
            pos = start + len;
        }
        Ok(out)
    }
}

fn decode_payload(p: &[u8]) -> Result<(Bytes, Bytes)> {
    let mut i = 0;
    if p.is_empty() || p[i] != TAG_PUT { bail!("bad tag"); }
    i += 1;
    let klen = u32::from_le_bytes(p[i..i+4].try_into()?) as usize; i += 4;
    let key = Bytes::copy_from_slice(&p[i..i+klen]); i += klen;
    let vlen = u32::from_le_bytes(p[i..i+4].try_into()?) as usize; i += 4;
    let val = Bytes::copy_from_slice(&p[i..i+vlen]);
    Ok((key, val))
}

代码 2 — src/engine.rs（写路径接线）

// src/engine.rs — 把 WAL + MemTable 接成一条带持久化的写路径。对标 DBImpl 的极简版。
use std::path::Path;
use anyhow::Result;
use crate::kv::{Key, Value};
use crate::memtable::MemTable;
use crate::wal::Wal;

pub struct Engine {
    mem: MemTable,
    wal: Wal,
}

impl Engine {
    /// 打开 / 创建一个引擎目录。先重放已有 WAL 重建 MemTable，再开 WAL 续写。
    pub fn open(dir: &Path) -> Result<Self> {
        std::fs::create_dir_all(dir)?;
        let wal_path = dir.join("wal.log");
        let mem = MemTable::new();
        // —— 恢复：重放顺序 = 写入顺序，后写的覆盖先写的 ——
        for (k, v) in Wal::replay(&wal_path)? {
            mem.put(k, v);
        }
        let wal = Wal::open_append(&wal_path)?;
        Ok(Self { mem, wal })
    }

    /// 写路径铁律：① WAL 落盘 → ② fsync → ③ 才让读可见。顺序不可换（B1 核心问题1）。
    /// &mut self：WAL 是串行点，写入在此被序列化——正是 group commit 要优化的那个串行瓶颈。
    pub fn put(&mut self, key: Key, value: Value) -> Result<()> {
        self.wal.append(&key, &value)?; // 1. 先写日志
        self.wal.sync()?;             // 2. 强制落盘（MVP: 每次 put 都 sync）
        self.mem.put(key, value);     // 3. 现在才对 get 可见
        Ok(())
    }

    pub fn get(&self, key: &[u8]) -> Option<Value> {
        self.mem.get(key) // Phase B 只有 MemTable 一层；Phase C/D 会加 SST 查找链
    }
}
注意 &mut self. Phase A 的 MemTable::put 是 &self（可并发）。但 Engine::put 退回 &mut self——因为 WAL 是个有序追加点，必须串行。这正是你研究 backlog 里那个并发写课题的现场：WAL 把并发又收回成串行,RocksDB 用 group commit 把"N 次 fsync 压成 1 次"来救它。Phase B 先串行跑对，并发优化是后话。

Harness 系统 ⑤：崩溃恢复

为什么这里用崩溃恢复

持久性的唯一证明方式：真的"死"一次再活过来看数据在不在。 模拟崩溃不需要真 kill 进程——drop(engine) 关掉文件、不做任何额外清理，就等价于"进程在这一刻消失"。然后用同一个目录 open 第二个 Engine，断言所有 ack 的写都读得到。 再加一招 torn-write：把 WAL 尾截掉几字节，模拟"写到一半断电"，验证恢复出完整前缀且不 panic。这是 RocksDB db_stress --reopen 的核心。

代码 3 — 崩溃恢复 harness（接 engine.rs 末尾）

#[cfg(test)]
mod tests {
    use super::*;
    use bytes::Bytes;
    use tempfile::tempdir;

    // 崩溃恢复：写 → drop(模拟崩) → 重开 → 数据全在（含覆盖写）
    #[test]
    fn survives_reopen() {
        let dir = tempdir().unwrap();
        {
            let mut e = Engine::open(dir.path()).unwrap();
            e.put(Bytes::from("a"), Bytes::from("1")).unwrap();
            e.put(Bytes::from("b"), Bytes::from("2")).unwrap();
            e.put(Bytes::from("a"), Bytes::from("3")).unwrap(); // 覆盖 a
            // e 在此 drop = 模拟进程崩溃（文件关闭，无额外动作）
        }
        let e2 = Engine::open(dir.path()).unwrap();
        assert_eq!(e2.get(b"a").as_deref(), Some(&b"3"[..])); // 重放顺序保证读到最新
        assert_eq!(e2.get(b"b").as_deref(), Some(&b"2"[..]));
        assert_eq!(e2.get(b"missing"), None);
    }

    // torn write：截断 WAL 尾 → 恢复出前缀、丢残尾、不 panic
    #[test]
    fn truncated_tail_tolerated() {
        let dir = tempdir().unwrap();
        {
            let mut e = Engine::open(dir.path()).unwrap();
            e.put(Bytes::from("k1"), Bytes::from("v1")).unwrap();
            e.put(Bytes::from("k2"), Bytes::from("v2")).unwrap();
        }
        // 砍掉文件尾 3 字节，撕裂第二条 record
        let wal = dir.path().join("wal.log");
        let len = std::fs::metadata(&wal).unwrap().len();
        let f = std::fs::OpenOptions::new().write(true).open(&wal).unwrap();
        f.set_len(len - 3).unwrap();
        // 恢复：k1 完好读回，k2 残缺被丢，不崩
        let e2 = Engine::open(dir.path()).unwrap();
        assert_eq!(e2.get(b"k1").as_deref(), Some(&b"v1"[..]));
        assert_eq!(e2.get(b"k2"), None);
    }
}

核心问题（自己答）

Phase B 收尾
  1. 损坏在中间，不在尾：我们的 replay 遇到第一条坏 record 就 break——若坏的是中间那条，它后面所有完好 record 也被丢了。RocksDB 为什么能"跳过坏块、重同步到下一条"继续读？（提示：32KB block 边界 + record 分片，正是 B1 我们砍掉的东西。这就是分块的真正用途。）
  2. 这个 harness 证明了什么、没证明什么：drop+reopen 模拟的是进程崩溃（OS page cache 还在）。它没有证明断电下的持久性（page cache 没 fsync 会随电源消失）。要真验断电,harness 该怎么改?（提示:在 sync 前后的不同时机崩；或注入"fsync 不生效"的假文件系统。诚实标注 harness 的边界——这是 storage-benchmark-lab 纪律。）
  3. WAL 会无限涨：每次 put 都 append，文件只增不减。什么时候能安全删/截断 WAL？（提示：当 MemTable 被 flush 成 SST 持久化后，那部分 WAL 就没用了——这是 Phase C flush 完成后的事，对标 RocksDB 的 WAL number 与 MANIFEST。）
  4. 每次 put 都 fsync 的代价：fsync 是几毫秒级的慢操作。1 万次 put = 1 万次 fsync。learning-record 0006 的 group commit 怎么把它压成"一组一次 fsync"？你的 Engine::put 现在是反面教材——这正是后续可优化点，先跑对再说。
