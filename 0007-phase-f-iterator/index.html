Phase F — SST iterator(从"点查"到"有序游标")

到 Phase E 为止,SST 只会一件事:get(key) 点查——给一个 key,二分定位、读一块、返回值。但 LSM 的另一半能力是范围扫描:"从 key k 开始,按序把后面的 key 一个个吐出来"。这需要一个全新的抽象——迭代器(iterator):一个能 seek 到某处、然后 next 一步步前进、随时能问"当前停在哪(key/value)、还有没有(valid)"的有序游标。这一阶段先给一个 block、再给一个 SST 装上迭代器,并立起整条读路径未来都要遵守的 迭代器契约。它是 Phase G(k 路归并)、Phase K(对外扫描 API)的地基。

Mission 锚点. 迭代器是计算机科学里最通用的抽象之一:任何语言的 for-each、Python 的 generator、Java 的 Iterator、数据库的 cursor——本质都是"把一个数据源包装成"当前位置 + 前进一步"的惰性游标",调用方不必知道底下是数组、链表、文件块还是网络流。RocksDB 内部用一个统一的 InternalIterator 接口(Seek/Next/Valid/key/value)把 memtable、block、SST、甚至归并器全部抹平成同一种东西——正因如此,上层的归并、范围扫描、compaction 才能用一套代码处理所有数据源。我们这一阶段就立起这个接口(trait LsmIterator),对标 block_based_table_iterator 与 InternalIterator。
本 Lab 宣称的 GOAL
  1. 迭代器契约:定义 trait LsmIterator = seek_to_first / seek(target) / next / valid / key / value。这是所有数据源对外统一的"有序游标"协议。
  2. BlockIter:在一个 block 内实现该 trait。seek(target) 块内二分到第一个 ≥target 的 entry;next 前进一格;key/value 返回借入块缓冲的切片(零拷贝,承 E)。
  3. SstIter:在一个 SST(多块)内实现该 trait。借稀疏索引定位候选块、按需 read_block(惰性);next 走到块尾自动跨到下一块。

Done when: cargo test 全绿,含迭代器契约(seek_to_first 全程升序 == 排序 oracle;seek 落在第一个 ≥target;next 严格递增)、跨块边界(扫描穿过 block 接缝不漏不重)、惰性读放大(扫到第几块才读第几块)、零拷贝借用。引擎 get 行为一字不改。

🏁 本 Lab 不改对外 get,而是给 SST 长出"游标"这条新能力线。点查回答"k 的值是多少",迭代器回答"从 k 起,按序还有什么"——后者是范围扫描、归并、compaction 的共同原语。

为什么自定义 trait,而不用 std::iter::Iterator

Rust 已有 Iterator,为什么还要自己定义一个?三处对不上,而这三处恰是 LSM 的命脉:

能力std Iterator我们要的 LsmIterator
定位只能从头顺序 next,不能 seekseek(target) 直接跳到第一个 ≥target——范围查询的入口
窥视当前next() 一调就消费并交出当前元素valid()+key() 能反复看当前头而不前进——k 路归并必须先比较各源的"当前头"再决定推谁
零拷贝Item 多为拥有值(Move)key()/value() 返回借入块缓冲的 &[u8],不分配(承 E 的零拷贝)

所以我们立一个 RocksDB 风格的内部迭代器协议:seek_to_first / seek / next / valid / key / value。它把"窥视当前头"和"前进"拆成两步(std 把它俩捆在 next 里),这正是归并能工作的前提——Phase G 的 MergingIterator 会持有一堆 Box<dyn LsmIterator>,每轮比较所有源的 key()、推进最小的那个。trait object 多态让 block、SST、memtable、归并器自己,全成为可互换的"有序源"。

路线:三个最小步(每个数据源一个游标)

F 的主题是把每一种数据源都包成同一个 LsmIterator 游标:block(F1)、SST(F2)、memtable(F3)。三个源同契约 → Phase G 的归并器就能一视同仁地吃它们,不必知道底下是内存跳表还是磁盘块。关键设计:key()/value() 返回裸 &[u8]、比较谁小的规则不写死在 iterator 里(留给 G 的 MergeIterator 持比较器)——这样 Phase H 引入 InternalKey(seq+type)时,只换"存的 key 字节 + merge 比较器 + 上层解码"三处,trait 签名与各 Iter 的 next/valid 逻辑一字不动。本阶段只做正向(seek/next);反向 prev 是后续高级 phase。

另一条贯穿 F 的设计决策——游标「持什么数据」。 同一问题(谁拥有被迭代的数据、它活多久),两种源、两种答案,都绕开"独占 own"、也都不退快照:磁盘块不可变、且 SstIter 要跨块换块 → BlockIter/SstIter 持 Arc<Block> 引用计数共享句柄(多 reader 共享、与 Phase S block cache 咬合;不用 &'a 因换块会撞自引用、cache 驱逐又是运行期 refcount——对标 RocksDB CachableEntry<Block>);内存跳表本就支持游标、只在一张表上滑不换对象 → MemTableIter<'a> 持 crossbeam Entry<'a> 借用游标(借外部 map,O(log n) seek + O(1) next,零拷贝;"自引用只能快照"是把高层 map::Entry 和底层裸 skiplist 搞混的误判,F3 详述)。选 Arc 还是 &'a 看"游标要不要换底层对象"——换 → Arc('static),不换 → 借用(带 'a)。两条都没退到 O(n) 快照。

本 Lab 的 harness 主题:契约即可执行的测试

迭代器的正确性不是"某次 seek 对了",而是一组必须永远成立的契约。把契约写成属性断言,用随机操作流去撞,比手写几个例子强得多。

Harness 系统抓什么用在
迭代器契约(单块)seek_to_first 后一路 next 到 invalid == 块内全部 entry 升序;seek(t) 落在第一个 ≥t;next 后 key 严格递增F1
零拷贝借用断言key()/value() 返回的切片指针落在块缓冲地址区间内(借用,非副本)F1
扫描差分 vs oracle多块 SST,从随机 seek 点起的整条扫描 == BTreeMap::range 的同段输出F2
跨块边界seek 落在每块首/末键、块缝隙键;next 穿过 block 接缝不漏不重F2
惰性读放大seek 只读 1 块;扫 m 块 blocks_read 增量恰为 m(不全载文件)F2
带着这个问题进 F1:迭代器的 key() 返回 &[u8] 借入它内部持有的 block 缓冲。那么 next() 之后,上一次 key() 拿到的切片还有效吗?(提示:在一个 block 内 next 只是挪 idx,缓冲没动,旧切片其实还指着合法内存——但语义上你不该再用它。到 F2 跨块时,next 会换掉整个 block 缓冲,旧切片就真悬空了。Rust 的借用检查器会怎么帮你把这条"用完即弃"的契约编译期钉死?想想 key(&self) 的借用 vs next(&mut self) 的可变借用为什么天然互斥。)
