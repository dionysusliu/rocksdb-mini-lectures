STEP F1 · Phase F 迭代器

迭代器契约 + BlockIter

先立协议,再实现最小的一个数据源。协议 = trait LsmIterator,是 block / SST / memtable / 归并器将来全要遵守的"有序游标"接口。最小数据源 = 一个 block:它早已是升序、已在内存(Bytes 缓冲),把它包成一个能 seek/next 的游标,几乎只是"管一个下标 idx"。

GOAL

新建 iterator.rs:定义 trait LsmIterator(seek_to_first/seek/next/valid/key/value)。block.rs:加 Block::seg_start(restart 上二分定段)、前缀压缩感知的 BlockIter 实现 trait。lib.rs 加 pub mod iterator;。

Done when: 契约 harness 全绿——seek_to_first 后一路 next 到 invalid 吐出块内全部 entry 升序;seek(t) 落在第一个 ≥t;越界 seek 变 invalid。

为什么先有"窥视当前"再有"前进"(归并锚)

本课最该想透的设计:协议把 看当前(valid/key/value) 和 前进(next) 拆成两个独立动作。std::Iterator::next() 把这俩捆死——一调既交出当前、又前进,没法在不前进的情况下反复看头。可 Phase G 的 k 路归并必须:先看齐所有源的当前头 → 比出最小 → 只推进那一个源。被推进的源 next 一下,其它源的头纹丝不动、还要再被看一次。所以"窥视"与"前进"必须解耦。这就是为什么数据库内部迭代器(RocksDB InternalIterator、LevelDB Iterator)全长成 Valid/key/Next 这副样子,而不是语言自带的 Iterator。

table/internal_iterator.h — virtual void Seek(); virtual void Next(); virtual bool Valid() const; virtual Slice key() const;

代码 1 — trait LsmIterator(新建 iterator.rs)

    /// 统一的"有序游标"协议。block / SST / memtable / 归并器对外都长这样,
    /// 于是上层(归并、范围扫描、compaction)能用一套代码处理任意数据源。
    pub trait LsmIterator {
        /// 定位到第一个 entry(最小 key)。
        fn seek_to_first(&mut self);
        /// 定位到第一个 key >= target 的 entry;若无(target 比所有 key 都大)则变 invalid。
        fn seek(&mut self, target: &[u8]);
        /// 前进一格。仅 valid() 时调用有意义(契约:invalid 时别调)。
        fn next(&mut self);
        /// 当前是否停在合法 entry 上。
        fn valid(&self) -> bool;
        /// 当前 key / value;仅 valid() 为真时可调。借入内部缓冲——零拷贝。
        fn key(&self) -> &[u8];
        fn value(&self) -> &[u8];
    }

key(&self)->&[u8] 这个签名本身就把零拷贝钉成了类型契约:返回值借的是 &self,你没法在这里现造一份副本再返回它的引用(临时值的引用编译不过)。所以实现者被迫返回"指向自己内部持有的那块缓冲"的切片——零拷贝不是靠自律,是编译器逼的。

⚠️ 承接 E4:块格式变了,没有 entry_at(i) 了。 E4 给块加了前缀压缩 + restart 点:第 i 条 entry 只存"与前一条共享的前缀长 + 后缀",没法 O(1) 随机取第 i 条(要从 restart 段首往前重建)。所以本课的 seek 不能写成"entry_at(mid) 二分"。改两点:① 二分只在 restart 点上做(restart 点存完整 key,可比);② 游标持一个 cur_key 缓冲,前进时增量重建当前 key。这正是 RocksDB BlockIter 的真实形态。

代码 2 — Block::seg_start(restart 上二分;扩 block.rs)

    impl Block {
        /// 找最后一个 restart_key <= target 的段起点偏移;target 比首 restart 还小则从 0 起。
        /// 注意:二分的是 restart 数组(每个 restart 点存完整 key,可直接比),不是每条 entry。
        fn seg_start(&self, target: &[u8]) -> usize {
            let (mut lo, mut hi) = (0usize, self.restarts.len());
            while lo < hi {
                let mid = (lo + hi) / 2;
                if self.restart_key(mid) <= target { lo = mid + 1; } else { hi = mid; }
            }
            if lo == 0 { 0 } else { self.restarts[lo - 1] as usize }  // 段起点的块内偏移
        }
    }

这是 两级二分的外层:restart 数组上 lower_bound 把 target 锁进某个 ≤16 条的段;BlockIter::seek 再在段内线性扫(内层)。和 E4 的 Block::get 用的是同一套"restart 二分 + 段内扫",get 其实是它的退化(扫到判等即返回)。

代码 3 — BlockIter 实现 trait(前缀压缩感知;扩 block.rs)

    use std::sync::Arc;
    use crate::iterator::LsmIterator;

    /// 块内有序游标。持 Arc<Block> 共享句柄;前进时增量重建当前 key 到 cur_key。
    pub struct BlockIter {
        block: Arc<Block>,   // 共享句柄:块可被 cache 复用(Phase S),游标只持强引用不独占
        off: usize,          // 当前 entry 块内偏移;>= entries_end 即 invalid(哨兵)
        next_off: usize,     // 下一条偏移(load 时算好,next 直接用)
        cur_key: Vec<u8>,    // 增量重建出的当前完整 key
        val: (usize, usize), // 当前 value 在 block.data 内的 (start, len)
    }
    impl BlockIter {
        pub fn new(block: Arc<Block>) -> Self {
            let mut it = Self { block, off: 0, next_off: 0, cur_key: Vec::new(), val: (0, 0) };
            it.load(); it
        }
        /// 在 self.off 处重建 cur_key(承接已有 cur_key 增量)+ 记 value 范围 + next_off。
        fn load(&mut self) {
            if self.off >= self.block.entries_end { return; } // invalid:不动 cur_key
            let (shared, _ns, suffix, value, next) = self.block.parse_at(self.off);
            self.cur_key.truncate(shared);              // 留共享前缀
            self.cur_key.extend_from_slice(suffix);     // 接本条后缀 → 完整 key
            self.val = (next - value.len(), value.len());
            self.next_off = next;
        }
    }
    impl LsmIterator for BlockIter {
        fn seek_to_first(&mut self) { self.off = 0; self.cur_key.clear(); self.load(); }
        fn seek(&mut self, target: &[u8]) {
            if self.block.restarts.is_empty() { self.off = self.block.entries_end; return; }
            self.off = self.block.seg_start(target);       // ① restart 二分定段
            self.cur_key.clear(); self.load();
            while self.valid() && self.cur_key.as_slice() < target { self.next(); } // ② 段内扫到 ≥target
        }
        fn next(&mut self) {
            if self.off >= self.block.entries_end { return; }
            self.off = self.next_off; self.load();
        }
        fn valid(&self) -> bool { self.off < self.block.entries_end }
        fn key(&self) -> &[u8] { &self.cur_key }
        fn value(&self) -> &[u8] { &self.block.data[self.val.0..self.val.0 + self.val.1] }
    }

BlockIter 写在 block.rs 内(同模块),才能读 Block 的私有字段 parse_at / restarts / entries_end / data(Arc<Block> 自动 Deref 到 &Block,字段访问透明)。key() 借的是游标自己的 cur_key;value() 借的是块缓冲 block.data——value 仍零拷贝,key 因前缀压缩在缓冲里不连续、只能重建成 owned。

设计决策:迭代器该「持什么块」(domain modeling)

SST 落盘即不可变只读 → 同一个块本该能被多个 reader 安全共享。那 BlockIter 到底该 own 块 / 借 &'a 块 / 持引用计数句柄?三选一,排除法走一遍:

方案形态致命伤
own Block迭代器独占整块① 多 reader 无法共享同块(各拷一份);② 让 Phase S 的 block cache 失去意义——cache 存一份、迭代器又拷一份,缓存白做
&'a Block编译期借用,绑某 owner 生命周期① SstIter 跨块要换当前块 → 块得存在 SstIter 体内、同结构体内的 BlockIter 又借它 = 自引用借用,编译器直接拒;② cache 驱逐是"还有人用就别淘汰"的运行期 refcount,编译期 'a 表达不了
✅ Arc<Block>运行期引用计数的共享只读句柄无。clone 廉价(只 +1 计数);最后一个持有者 drop 才释放;与 cache 天然咬合

你的直觉"迭代器只持引用、块比迭代器活得久"方向正确——但落地不是编译期 &'a 借用,而是运行期共享引用 Arc。理由:块的真实寿命由"谁还在用"决定(cache + 多个游标 + 跨块换块),这是个动态集合,只有 refcount 能精确表达,'a 这种静态生命周期标注表达不了。RocksDB 同结论:其 BlockBasedTableIterator 持 CachableEntry<Block> —— cache 感知的引用计数句柄,正是 Arc<Block> 的工业版(迭代期 pin 住块、用完交还 cache)。

落到代码:read_block(Phase S 后)返回 Arc<Block>,cache 命中即 clone 一份递出 → 迭代器持着它把块 pin 在内存,即便 cache 条目被淘汰、块也活到游标放手。本课测试用 Arc::new(Block::decode(..)) 自造一个块;到 F2 就由 SstReader 递来。table/block_based/block_based_table_iterator.h — CachableEntry<Block> cachable_entry_; bool block_iter_points_to_real_block_;

off >= entries_end 当 invalid 哨兵,不用 Option. 越界偏移天然表达"扫完了":valid 一个比较、next 越过末尾自然 invalid、seek 超界时段内扫到尽头也落到 entries_end——同一套逻辑。这是用已有状态空间的边界值编码"无效",省掉一层包装(同 D1 空 SST 列表、A 的 approx_size)。前缀压缩把"下标 idx"换成了"偏移 off + cur_key 缓冲",但哨兵思路一字不变。

🧰 Coding style:接口/实现分离 · 哨兵编码 · 二分收敛为一

  • trait = 接口与实现解耦:协议定在 iterator.rs,谁都能实现。这让 Phase G 能写 Vec<Box<dyn LsmIterator>> 把异构数据源(block/SST/memtable)装进同一个容器、统一驱动——面向接口编程的教科书用法,对应 Java interface、Go interface、C++ 抽象基类。
  • 一个二分喂两处:seg_start(restart 上的 lower_bound)同时服务 BlockIter::seek(本课)和 E4 的 Block::get(seg_start + 段内判等)。把"restart 定段"这一个原语抽出来,胜过到处手写边界易错的二分。前缀压缩让"段内"无法再二分,但"段间"仍是干净的二分。
  • 哨兵优于 Option:off >= entries_end 表无效,比 Option<usize> 少一层解包,且 seek/next/valid 三者共享同一边界语义。
  • 不可变 + 引用计数 = 安全共享只读:BlockIter 持 Arc<Block> 而非 own/借用——块只读不变,Arc 让多个游标 + block cache 共享同一份、clone 只加计数。这是 Rust 缓存/快照/MVCC 类结构的主力范式,Phase S(block cache 存 Arc<Block>)、Phase H(MVCC 快照)还会反复用同一招。

🧠 System mindset:迭代器 = 惰性游标 · "窥视/前进"解耦

迭代器把"一个有序数据源"抽象成当前位置 + 前进一步的最小接口,调用方不必知道底层是数组、文件块还是网络流。这个抽象无处不在:

领域"有序游标"长这样
语言Python generator / Java Iterator / C++ begin..end / Rust Iterator——惰性:要一个算一个,不预先全展开
数据库(★)cursor / 火山模型(Volcano)算子:每个算子是个迭代器,next 向下游拉一行——整棵查询计划就是迭代器套迭代器
Unix管道 a | b | c:每段按需消费上游一块,背压自然形成
归并(本阶段去向)k 路归并要求"看 k 个源的当前头、推最小的一个"——必须能窥视而不消费

迁移收获:当你要"统一处理一堆形态不同、但都能按序产出的数据源"时,先定义一个 Valid/key/Next 式游标接口,把"窥视当前"和"前进"拆开。这一拆,归并、去重、范围裁剪、多路合并全都解锁——它们的共同前提就是"能反复看头、按需推进"。语言自带的 Iterator(next 即消费)在需要多源协同时反而不够用,这正是所有数据库都自造内部迭代器的原因。

🧪 Harness 系统:契约即可执行的测试

为什么验"契约"而非"几个例子"

迭代器的正确性是一组恒成立的性质,不是某次 seek 的结果:(1) seek_to_first 后一路 next 到 invalid,吐出的序列 == 块内全部 entry 的升序;(2) seek(t) 后若 valid,则 key() >= t 且是最小的这种;(3) 每次 next 后 key 严格递增。 把这些写成断言,再用各种 target(命中键、缝隙键、空串、超界键)去打,比手写三五个例子稳得多。下一课 F2 会把它升级成"vs BTreeMap 的全量差分"。

代码 4 — 契约 harness(接 block.rs 的 tests)

    use std::sync::Arc;
    use crate::iterator::LsmIterator;

    fn build_block() -> Bytes {                // k00..k19 升序,返回编码后的块字节
        let mut b = BlockBuilder::new();
        for i in 0..20u32 {
            b.add(format!("k{i:02}").as_bytes(), format!("v{i:02}").as_bytes());
        }
        b.finish()
    }

    #[test]
    fn block_iter_full_scan_ascending() {
        let bytes = build_block();
        let mut it = BlockIter::new(Arc::new(Block::decode(bytes)));
        it.seek_to_first();
        let mut n = 0u32;
        let mut prev: Option<Vec<u8>> = None;
        while it.valid() {
            if let Some(p) = &prev { assert!(p.as_slice() < it.key(), "key 必须严格递增"); }
            assert_eq!(it.key(), format!("k{n:02}").as_bytes());
            assert_eq!(it.value(), format!("v{n:02}").as_bytes());
            prev = Some(it.key().to_vec());
            it.next(); n += 1;
        }
        assert_eq!(n, 20);                       // 不多不少
    }

    #[test]
    fn block_iter_seek_lower_bound() {
        let bytes = build_block();
        let mk = || BlockIter::new(Arc::new(Block::decode(bytes.clone())));  // Bytes::clone = 引用计数+1,不拷贝缓冲

        let mut it = mk(); it.seek(b"k07");   assert_eq!(it.key(), b"k07");     // 命中
        let mut it = mk(); it.seek(b"k07x");  assert_eq!(it.key(), b"k08");     // 缝隙→下一个
        let mut it = mk(); it.seek(b"");      assert_eq!(it.key(), b"k00");     // 空串→第一个
        let mut it = mk(); it.seek(b"zzz");   assert!(!it.valid());            // 超界→invalid
    }

核心问题(自己答)

F1 想透这几个
  1. 为什么 next 是 &mut self 而 key 是 &self:这对签名让"前进"和"窥视"在借用层面天然互斥——持有 key() 返回的 &[u8](不可变借用)期间,你无法调 next()(需要可变借用)。借用检查器是怎么把"next 后旧 key 切片失效"这条契约在编译期替你钉死的?(对比:C++ 的迭代器失效是运行期 UB,得靠程序员记住。)
  2. seek 与 get 的关系:回看 E4 的 Block::get(k)——它已经是"seg_start 定段 + 段内扫到判等"。BlockIter::seek 用的是同一个 seg_start,只是段内扫到"第一个 ≥k"就停、不判等。所以 get(k) 等价于"seek(k) 后若 valid 且 key()==k 返回 value,否则 None"——点查是扫描的退化情形,一套定位逻辑两边共用。想清楚:既然能共用,为什么 RocksDB 仍保留独立的 Get 路径而不全建在 iterator 上?(提示:bloom 过滤 + 命中即停的栈优化;见 F 设计笔记的"两路"取舍。)
  3. 哨兵 vs Option:用 off >= entries_end 表 invalid 省了 Option,代价是什么?(key() 在 invalid 时返回的是上一次 load 留下的陈旧 cur_key、value() 用陈旧 val 范围——若调用方没先 valid() 就读,拿到的是脏值而非 panic。契约把"先 valid 再 key"的责任外推给调用方——性能与安全的取舍,RocksDB 同样如此。)

下一步:F2 — SstIter 跨块惰性扫描. 一个 block 的游标已通。SST 是多块:F2 用稀疏索引定位候选块、按需 read_block、next 走到块尾自动跨下一块。会证明一条漂亮性质——稀疏索引 lower_bound 选出的候选块必含答案,所以 seek 永不需要跨块修正。
