STEP F2 · Phase F 迭代器

SstIter — 跨块惰性扫描

F1 的 BlockIter 只在一个 block 内走。一个 SST 是多块 + 一张稀疏索引。SstIter 是两级游标:外层用稀疏索引在块之间定位/跨越,内层用 BlockIter 在块内走。关键是惰性——扫到哪一块,才 read_block 哪一块;绝不把整个文件载进来。

GOAL

sst.rs 加 SstIter<'a>(借 &SstReader):seek 用稀疏索引 partition_point 定位候选块 + 块内 seek;next 走到块尾自动 read_block 下一块;SstReader::iter 工厂方法。实现 LsmIterator。

Done when: 扫描差分(从任意 seek 点起 == BTreeMap::range)、跨块边界(全量扫 200 条穿过所有接缝不漏不重)、惰性读放大(seek 只读 1 块、全扫恰好读 num_blocks 块)全绿。

核心性质:稀疏索引 lower_bound 选出的候选块,必含第一个 ≥target

稀疏索引每项 = (该块的末键, BlockHandle)。seek(target) 在索引上 partition_point(|last| last < target) 得到第一个 末键 ≥ target 的块 i。这块的末键既然 ≥ target,块内必然存在 ≥ target 的 entry(末键本身就是)。所以在第 i 块内做 BlockIter::seek(target) 一定 valid——seek 永远不需要"块内没找到、再跨到下一块"的修正逻辑。唯一的"无块可选"是 i == 索引长度(target 比所有块的末键都大)→ 直接 invalid。

这条性质把 seek 和 next 的复杂度分得干干净净. seek:一次索引二分 + 一次块内二分,无跨块。next:块内 +1,仅当越过块尾才跨到下一块的开头。两个动作各管一段,互不渗透——这正是"把不变量(末键≥target ⇒ 块内有解)前移到数据布局里"换来的代码简洁,和 D1/E2 一脉相承。

步骤

  1. sst.rs 定义 SstIter<'a> + load 私有助手(「代码 1」)。
  2. 实现 LsmIterator:seek_to_first/seek/next/valid/key/value;加 SstReader::iter 工厂(「代码 2」)。
  3. 敲入「代码 3」差分 + 跨块 + 惰性三组 harness,cargo test 全绿。

代码 1 — SstIter 结构 + load

    use crate::block::BlockIter;
    use crate::iterator::LsmIterator;

    pub struct SstIter<'a> {
        reader: &'a SstReader,    // 借读者:按需 read_block + 看 index
        blk: usize,              // 当前在第几个稀疏索引块
        cur: Option<BlockIter>,  // 当前块的内层游标;None = invalid
    }

    impl<'a> SstIter<'a> {
        /// 载入第 i 块为当前块(越界则置空)。这是唯一摸盘的地方 → 惰性的闸口。
        fn load(&mut self, i: usize) {
            if i >= self.reader.index.len() { self.cur = None; return; }
            let h = self.reader.index[i].1;
            let block = self.reader.read_block(h).expect("read_block"); // MVP:IO 错即 panic(见核心问题)
            self.blk = i;
            self.cur = Some(BlockIter::new(Arc::new(block)));   // BlockIter 持 Arc<Block>(F1);此处包一层
        }
    }

read_block 是 E2 已有的私有方法(read_exact_at + blocks_read.fetch_add(1) + Block::decode),现返回 Block。BlockIter::new 吃 Arc<Block>(F1 定的,块要可共享),所以这里 Arc::new(block) 包一层。到 Phase S(block cache)read_block 会改成返回 Arc<Block>,届时这层 Arc::new 直接去掉——缓存命中递来的就是现成的 Arc。SstIter 与它同在 sst.rs,可直接用私有字段 index / 方法 read_block——模块边界即封装边界。(顶部记得 use std::sync::Arc;)

代码 2 — 实现 LsmIterator + iter 工厂

    impl LsmIterator for SstIter<'_> {
        fn seek_to_first(&mut self) {
            self.load(0);
            if let Some(c) = &mut self.cur { c.seek_to_first(); }
        }
        fn seek(&mut self, target: &[u8]) {
            // 稀疏索引 lower_bound:第一个 末键 >= target 的块
            let i = self.reader.index.partition_point(|(last, _)| last.as_ref() < target);
            self.load(i);
            if let Some(c) = &mut self.cur { c.seek(target); } // 候选块必含解 → 不需跨块
        }
        fn next(&mut self) {
            let Some(c) = &mut self.cur else { return; };
            c.next();
            if !c.valid() {                    // 当前块走完 → 跨下一块的开头
                let nxt = self.blk + 1;
                self.load(nxt);
                if let Some(c2) = &mut self.cur { c2.seek_to_first(); }
            }
        }
        fn valid(&self) -> bool { self.cur.as_ref().map_or(false, |c| c.valid()) }
        fn key(&self)   -> &[u8] { self.cur.as_ref().unwrap().key() }
        fn value(&self) -> &[u8] { self.cur.as_ref().unwrap().value() }
    }

    impl SstReader {
        /// 造一个借本 reader 的扫描游标。
        pub fn iter(&self) -> SstIter<'_> { SstIter { reader: self, blk: 0, cur: None } }
    }
为什么 next 跨块后要 seek_to_first. load(nxt) 新建的 BlockIter 的 new 已 off=0 重建好首条,理论上已在开头——但显式 seek_to_first 让"跨块后落在新块第一个 entry"这个意图写在脸上,不依赖 new 的默认值。块都非空(flush_block 只在非空时封口),所以跨过去必 valid,不会"空块导致提前结束"。

🧰 Coding style:两级游标 · 惰性闸口集中 · let-else 收窄

  • 两级迭代器:外层(索引/块间)+ 内层(BlockIter/块内)。每层只管自己那段,next 的跨块是唯一的层间衔接点。对标 RocksDB BlockBasedTableIterator 正是"index_iter_ + data block iter"两级。分层迭代让"块内逻辑"与"块间逻辑"各自可独立测试、独立替换(比如换索引格式不动块内)。
  • 惰性只有一个闸口:所有摸盘都过 load 一处。要加 prefetch、block cache(Phase S)、读计数,都只改这一个函数——把"副作用集中到单点"是可观测、可优化的前提。
  • let Some(c) = &mut self.cur else { return };:let-else 把"无效就早退"压成一行,主逻辑不再多一层缩进。这是 Rust 处理"提前返回 + 解包"的惯用法。

🧠 System mindset:火山模型 · 惰性 IO · 错误通道的取舍

SstIter 是火山模型(Volcano / iterator model)的微缩:每个算子是迭代器,上层 next 向下层按需拉数据,数据一条条流过,不物化中间结果。整个数据库查询引擎就是这种迭代器的层层嵌套——扫描算子、过滤算子、归并算子各是一个 LsmIterator,Phase G 的归并器会把多个 SstIter 当下游拉。

维度我们的 SstIter真 RocksDB / 通用做法
惰性 IO扫到第 i 块才 read_block(i)同;另加 prefetch / readahead(顺序扫时预读下一块摊薄延迟)、block cache(热块免重读,Phase S)
错误通道next/seek 不返回 Result → IO 错 expect panic(MVP)迭代器存一个 Status 字段,Valid() 转 false,调用方 iter.status() 取错——把错误从控制流挪到状态位,因为迭代器接口不便处处返回 Result

迁移收获:迭代器接口("当前/前进")天然不便携带错误(返回值已被 item 占用)。两条出路——要么像我们 MVP 直接 panic(简单、能跑),要么像 RocksDB 把错误沉淀成一个状态位(Valid()=false + status()),让"没有更多了"和"出错了"用同一个 invalid 表达、再用旁路区分。这个"主通道传数据、旁路传错误"的模式,在流式 API、游标、异步 stream 里反复出现。我们诚实标注:MVP 选 panic,稳健版选 status 位(可留作后续小练习)。

🧪 Harness 系统:扫描差分 + 跨块 + 惰性读放大

三组验法各抓一类 bug

① 扫描差分:从任意 seek 点起的扫描,必须逐条 == BTreeMap::range(seek..)。oracle 是标准库的有序 map,显然正确,抓"漏条/重条/顺序错/跨块丢"。
② 跨块边界:强制小 block_size 切出多块,全量扫描计数必须等于写入条数——专打块接缝(next 跨块时最易漏掉新块首条或重复旧块末条)。
③ 惰性读放大:给 blocks_read 计数器把"惰性"变成可执行契约——seek 只读 1 块、全量扫描恰好读 num_blocks 块。多读一块(误预载)或少读(漏块)立刻被抓。把性能承诺写成断言,承 E2。

代码 3 — 三组 harness(接 sst.rs 的 tests)

    use crate::iterator::LsmIterator;
    use std::collections::BTreeMap;

    fn build_multiblock(path: &Path) -> BTreeMap<Vec<u8>, Vec<u8>> {
        let mut b = SstBuilder::with_block_size(64);   // 小块 → 强制多块
        let mut model = BTreeMap::new();
        for i in 0..200u32 {
            let (k, v) = (format!("key{i:04}"), format!("val{i:04}"));
            b.add(k.as_bytes(), v.as_bytes()).unwrap();
            model.insert(k.into_bytes(), v.into_bytes());
        }
        b.finish(path).unwrap();
        model
    }

    // ① 扫描差分:从多个 seek 点起 == BTreeMap::range
    #[test]
    fn sst_iter_scan_differential() {
        let dir = tempdir().unwrap();
        let path = dir.path().join("t.sst");
        let model = build_multiblock(&path);
        let r = SstReader::open(&path).unwrap();
        assert!(r.num_blocks() > 1, "应切成多块");

        for probe in ["", "key0000", "key0099", "key01005", "key0199", "zzz"] {
            let mut it = r.iter();
            it.seek(probe.as_bytes());
            let mut got = Vec::new();
            while it.valid() {
                got.push((it.key().to_vec(), it.value().to_vec()));
                it.next();
            }
            let want: Vec<_> = model.range(probe.as_bytes().to_vec()..)
                .map(|(k, v)| (k.clone(), v.clone())).collect();
            assert_eq!(got, want, "seek({probe}) 起的扫描差分");
        }
    }

    // ② 跨块:全量扫描穿过所有接缝,不漏不重、严格递增
    #[test]
    fn sst_iter_full_scan_crosses_blocks() {
        let dir = tempdir().unwrap();
        let path = dir.path().join("t.sst");
        build_multiblock(&path);
        let r = SstReader::open(&path).unwrap();
        let mut it = r.iter(); it.seek_to_first();
        let (mut n, mut prev) = (0u32, None::<Vec<u8>>);
        while it.valid() {
            if let Some(p) = &prev { assert!(p.as_slice() < it.key(), "严格递增"); }
            prev = Some(it.key().to_vec()); it.next(); n += 1;
        }
        assert_eq!(n, 200);                      // 跨所有 block 不漏不重
    }

    // ③ 惰性读放大:seek 只读 1 块;全扫恰好读 num_blocks 块
    #[test]
    fn sst_iter_lazy_block_reads() {
        let dir = tempdir().unwrap();
        let path = dir.path().join("t.sst");
        build_multiblock(&path);
        let r = SstReader::open(&path).unwrap();

        let base = r.blocks_read();
        let mut it = r.iter(); it.seek(b"key0000");
        assert_eq!(r.blocks_read() - base, 1, "seek 只载 1 块");
        while it.valid() { it.next(); }
        assert_eq!(r.blocks_read() - base, r.num_blocks() as u64, "全扫恰好每块读一次");
    }

核心问题(自己答)

F2 / Phase F 收尾
  1. 为什么 seek 不需要跨块、next 才需要:用本课开头那条性质(候选块末键 ≥ target ⇒ 块内必有解)讲清。如果稀疏索引存的是每块的首键而非末键,这条还成立吗?seek 会不会就需要跨块了?(提示:存首键的话,partition_point 选出的块,其首键 ≤ target,但块内末键可能 < target → 解在下一块 → seek 就得跨块。存末键是让"候选块必含解"成立的关键设计。)
  2. 错误去哪了:我们 read_block(..).expect() 把 IO 错变 panic。改成 RocksDB 式"Valid()=false + status() 状态位"要加什么?(SstIter 加一个 status: Result<()> 字段,load 失败时填错并令 cur=None;valid 照旧 false。调用方扫完后查 status 区分"扫完了"与"扫炸了"。)
  3. 顺序扫描为什么该 prefetch:惰性"扫到才读"省了不需要的块,但顺序大扫描时每跨一块都同步等一次磁盘。真引擎怎么把这段延迟摊掉?(readahead/prefetch:扫第 i 块时异步预读第 i+1 块;Phase S 的 block cache 让热块免重读。我们 MVP 先要正确,优化留后。)
  4. 这个零件怎么拼进归并:Phase G 要把 memtable 迭代器 + 若干 SstIter 合成一条全局有序流。它会持有 Vec<Box<dyn LsmIterator>>,每轮看所有源的 key()、推最小的。回看 F1 为什么坚持"窥视(key)与前进(next)解耦"——没有这一步,归并根本写不出来。

下一步:Phase G — k 路归并 iterator. 把本课的零件(统一 LsmIterator 的多个有序源)用一个 MergingIterator 合并成一条全局有序、新值盖旧值的流。trait LsmIterator + Box<dyn ..> 的多态红利在那里兑现。harness:归并差分(vs 合并后的 BTreeMap)。
