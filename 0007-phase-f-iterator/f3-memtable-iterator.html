STEP F3 · Phase F 迭代器

MemTableIter — 跳表包成游标

F1 给 block、F2 给 SST 装了游标。还差最后一个数据源:memtable。它在内存里、用 crossbeam 跳表存、本就有序——给它包一个 LsmIterator,Phase G 的归并器就能一视同仁地吃 memtable + SST,不必知道底下是内存跳表还是磁盘块。这一步代码量最小,但藏着一个被广泛误判的 Rust 难点:"跳表游标会自引用,只能快照"——其实不然。本课走真·惰性跳表游标,O(log n) seek + O(1) next,零快照、零拷贝,纯安全 Rust。

GOAL

memtable.rs 加 MemTableIter<'a> 实现 LsmIterator + 工厂 MemTable::iter_lsm()。实现方式:持一个 crossbeam Entry 引用计数游标——front() 定位首条、lower_bound() 做 seek、move_next() 前进。不快照、不预拷。

Done when: 契约 harness 绿——seek_to_first 一路 next == BTreeMap 有序遍历;seek(t) 落第一个 ≥t(命中/缝隙/空串/超界);活游标语义(建 iter 后才插入的键,seek 时仍看得到);与 SstIter 同 trait(同数据两源吐同序,为 Phase G 差分铺路)。

为什么能持真跳表游标?(本课最该想透 · 一个被广泛误判的"自引用")

常见的退让是:"crossbeam 的 Entry 借内部 guard,把迭代器+当前 Entry 塞进一个 struct 会自引用 → 只能 collect 成 Vec 快照"。这个论断把两套 API 搞混了:

APIEntry 借谁能否安全持游标
底层 crossbeam_epoch 裸 skiplist借一个 pinned Guard(你得自己持有)❌ 持 Guard + 借 Guard 的 Entry = 真自引用
✅ 高层 crossbeam_skiplist::mapEntry<'a> 借 map 本身(外部),引用计数钉住节点✅ 无 guard 可借,无自引用

查 docs 实证:map::Entry<'a, K, V> 的 'a 绑的是 借 SkipMap;它有 move_next(&mut self)->bool(O(1) 往后一格)、next/prev,本身就是个可移动的引用计数游标,节点被删也靠 refcount 活着。于是:

pub struct MemTableIter<'a> {
    mt: &'a MemTable,                      // map 在 MemTable 体内 = 外部
    cur: Option<Entry<'a, Bytes, Bytes>>,  // 借外部 map(同一个 'a),不是借 self
}

两个字段都借结构体外面那个 MemTable('a)。cur 借的不是 self → 没有自引用 → 零 unsafe。seek = lower_bound 的 O(log n) 跳表下降,next = move_next 的 O(1),无 upfront collect、无 n 个指针的 Vec——完整保住 memtable 的 in-mem 高性能。这正是 RocksDB MemTableIterator 的真实形态(真跳表游标,非快照)。

自引用只在"我全都要"的写法里冒出来。 若想让迭代器 'static(不带 'a)又持 O(1) 游标,就得 struct { mt: Arc<MemTable>, cur: Entry<'???> }——cur 要借 self.mt 自己的字段,'??? 没法命名("借兄弟字段"的生命周期类型系统表达不了),这才是真自引用,只能 ouroboros/Pin/unsafe。我们不走那条:接受 MemTableIter<'a> 带生命周期(memtable 活过扫描,本就成立),换全程安全 + O(1)。C++ 的 RocksDB 能"全都要",只因它本就手动 refcount + 裸游标、没有借用检查器拦着。
docs.rs crossbeam_skiplist::map::{SkipMap::front / lower_bound, Entry::move_next} · RocksDB db/memtable.cc MemTableIterator(真跳表游标)
弱一致视图——不是 bug,Phase H 兜底。 active memtable 并发 insert(&self)(lock-free)时,move_next 游标看到的是弱一致视图:扫描开始后的并发插入,可能见、可能漏。但引入 seq + snapshot(Phase H/J)后,可见性由"版本号 vs 快照号"决定、与游标时机无关:seq > snapshot 的一律过滤,该见的(seq ≤ snapshot)扫描前已在表里、弱一致游标必然走到 → 物理弱一致 + 逻辑 seq 过滤 = 快照隔离。immutable memtable 已冻结,天然强一致。所以现在用弱一致游标零风险,Phase H 自动把它升级成强一致——正确性的担子从"游标"挪到了"版本号",这正是 seq 不是装饰、而是 scan 地基的原因。
回看 F 的所有权主线:同一问题、两种源、两种答案

F1/F2 与 F3 在回答同一个 domain modeling 问题:"游标拥有它正在迭代的数据吗?它活多久?"——都不 own 原始结构,但因源的性质分岔:

  • 磁盘块(F1/F2):不可变、需跨块换块 → 持 Arc<Block> 引用计数共享句柄。块不变可多 reader 共享、与 block cache 咬合;不用 &'a 因 SstIter 换块会撞自引用。用运行期 refcount 换"无生命周期 + O(1)"。
  • 内存跳表(F3):有序结构本就支持游标 → 持 Entry<'a> 借用游标。map 在外部,借它即可,refcount 钉节点;用编译期 'a 换"O(1) next + 安全 + 零拷贝",代价是带生命周期。

两种答案,同一条底线:谁拥有被迭代的数据、它活多久,是迭代器设计的核心建模题。 块用 Arc(自己持 refcount、'static),跳表用 Entry<'a>(借外部、带 'a)——选 Arc 还是 &'a,取决于"游标要不要换底层对象"(块要 → Arc;跳表不换、只在一张表上滑 → 借用够)。两条都避开了 own 整份数据,也都没退到快照。

步骤

  1. 在 memtable.rs 顶部 use std::ops::Bound; + use crossbeam_skiplist::map::Entry;。
  2. 加 MemTableIter<'a> 结构(mt: &'a MemTable + cur: Option<Entry<'a,..>>)与 MemTable::iter_lsm() 工厂(「代码 1」)。
  3. 实现 LsmIterator:seek_to_first=front()、seek=lower_bound(Included)、next=move_next()、key/value=cur.key()/value()(「代码 1」)。
  4. 敲入「代码 2」契约 harness(含活游标测试),cargo test 全绿。

代码 1 — MemTableIter(真·惰性跳表游标)

// memtable.rs
use std::ops::Bound;
use crossbeam_skiplist::map::Entry;

impl MemTable {
    /// Phase F3:游标借住本 memtable,故返回 MemTableIter<'_>(带其生命周期)。
    pub fn iter_lsm(&self) -> MemTableIter<'_> { MemTableIter::new(self) }
}

/// memtable 的有序游标:持一个 crossbeam refcount Entry,在跳表上滑。零快照零拷贝。
pub struct MemTableIter<'a> {
    mt: &'a MemTable,                      // map 在外部,活过游标
    cur: Option<Entry<'a, Bytes, Bytes>>,  // 当前节点的 refcount 游标;None = invalid
}
impl<'a> MemTableIter<'a> {
    fn new(mt: &'a MemTable) -> Self { Self { mt, cur: None } }
}
impl crate::iterator::LsmIterator for MemTableIter<'_> {
    fn seek_to_first(&mut self) { self.cur = self.mt.map.front(); }  // 最小 key 的 Entry
    fn seek(&mut self, target: &[u8]) {
        // 第一个 key >= target;O(log n) 跳表下降(Bytes: Borrow<[u8]>,零拷贝比较)
        self.cur = self.mt.map.lower_bound(Bound::Included(target));
    }
    fn next(&mut self) {
        if let Some(e) = &mut self.cur
            && !e.move_next() { self.cur = None; }  // 走过末尾 → invalid
    }
    fn valid(&self) -> bool { self.cur.is_some() }
    fn key(&self)   -> &[u8] { self.cur.as_ref().unwrap().key().as_ref() }
    fn value(&self) -> &[u8] { self.cur.as_ref().unwrap().value().as_ref() }
}

map 是 MemTable 的私有字段,MemTableIter 同模块可读。key()/value() 返回 cur 里 Entry 借出的 &[u8](Bytes::as_ref,零拷贝借进活节点)。next 用 let-chain:move_next 原地推进 Entry、返回是否还有下一条,走过末尾置 None——O(1),跳表 level-0 链表一格。对比 F1 的 BlockIter(前缀压缩要 cur_key 缓冲增量重建)、F2 的 SstIter(跨块换 Arc<Block>):三种实现繁简天差地别,对上层 Phase G 却完全一样——统一接口的意义。

🧰 Coding style:Option<Entry> 即哨兵 · lower_bound=库给的 seek · let-chain 早退

  • Option<Entry> 即哨兵:cur: None 直接表 invalid,valid 就是 is_some()。和 BlockIter 的越界 off、SstIter 的 cur: None 同思路——用类型自带的"空"状态当游标终点,不另造哨兵。
  • lower_bound = 库已实现的 seek:F1/F2 自己在 restart/索引上手写二分;memtable 的有序结构自带 lower_bound(Bound),seek 一行调用、O(log n)。能用数据结构自带的有序定位就别手写二分——少一处边界 bug。(Included = ≥target;要 >target 用 Excluded。)
  • let-chain 早退:if let Some(e)=&mut self.cur && !e.move_next() { self.cur=None } 把"解包 + 条件"压成一行。move_next 原地推进、借用在条件求值后即释放,故同句里能再写 self.cur=None。
  • 不退快照的底气:这次没有退到 collect 成 Vec。因为 map::Entry 借的是外部 map、非自引用——能持真游标就持真游标,保住 O(1) next 与零拷贝。Rust 里"借别人(外部)"永远比"借自己(自引用)"好走;真自引用才退 Arc/快照/ouroboros。

🧠 System mindset:迭代器抹平数据源 · 弱一致游标靠版本号兜底

概念本阶段怎么体现 / 跨域同构
统一接口抹平异构源memtable(内存跳表)、SST(磁盘块)、将来的归并器,全长成同一个 LsmIterator。上层不问"你是谁",只调 seek/next/key。同构:Unix "一切皆文件"(socket/pipe/device 都是 fd)、JDBC 抹平不同数据库、VFS 抹平不同文件系统。窄接口 + 多实现 = 上层一套代码通吃。
弱一致游标 + 版本号 = 快照隔离真跳表游标在并发写下是弱一致(可能见/漏并发插入)。系统不在游标层求强一致,而在逻辑层用 seq vs snapshot 过滤——物理弱一致被抹成逻辑快照隔离。同构:MVCC 读快照、Lamport/版本时钟、CRDT 的因果可见性。"正确性放在版本号,而非容器的瞬时状态",正是 Phase H/J 的核心,也是为什么 memtable 这里不必快照。
繁简由源决定,契约不变同一个 trait:BlockIter 增量重建 key(繁)、SstIter 跨块换 Arc(中)、MemTableIter 跳表游标滑一格(简)。接口稳定性吸收实现复杂度差异——复杂度封在实现里,不漏给调用方。

迁移收获:三个数据源装上同一个游标后,你手里已有归并的全部原料——Phase G 只需"持 N 个 LsmIterator、每轮比 key 取最小、推进那一个"。而 memtable 游标的弱一致不是缺陷而是分工:它只管"物理上有序滑动",一致性交给 Phase H 的 seq——这正是把"游标"和"可见性"解耦的预演,也是 Phase J 快照读能干净叠加上来的原因。

🧪 Harness 系统:契约 + 排序 oracle + 跨源同序(为 G 铺路)

为什么这样验

迭代器的正确性 = 契约:seek_to_first 后一路 next 必须吐出全部 entry、严格升序、key/value 配对正确——拿 BTreeMap 当 oracle 逐项对。seek 单独验四个临界:命中、缝隙落下一个、空串落首、超界变 invalid(lower_bound 的四种边界)。但 F3 真正的价值在 跨源同序:同一组 kv,喂给 MemTableIter 和(F2 的)SstIter,两个游标 seek_to_first 后吐出的序列必须一字不差。这条"同数据、不同源、同 trait → 同序"的差分,正是 Phase G 归并能成立的前提(归并假设每个源内部已正确有序),也是把"两个独立实现"锁死到同一语义的标准手法——它在 G 里会升级成"N 源归并 == 合并后的 BTreeMap"。

代码 2 — 契约 harness(memtable.rs tests)

    use crate::iterator::LsmIterator;

    // 契约:seek_to_first 后一路 next == BTreeMap 有序遍历。
    #[test]
    fn memtable_iter_full_scan_ascending() {
        let mt = MemTable::new();
        let mut model = BTreeMap::new();
        for i in [7u32, 3, 9, 1, 5, 8, 2] {          // 乱序插入
            mt.put(Bytes::from(format!("k{i:02}")), Bytes::from(format!("v{i}")));
            model.insert(format!("k{i:02}"), format!("v{i}"));
        }
        let mut it = mt.iter_lsm();
        it.seek_to_first();
        let mut got = Vec::new();
        while it.valid() {
            got.push((it.key().to_vec(), it.value().to_vec()));
            it.next();
        }
        // 与 BTreeMap 有序遍历一字不差(升序 + 全量 + 配对)
        let want: Vec<_> = model.into_iter()
            .map(|(k,v)| (k.into_bytes(), v.into_bytes())).collect();
        assert_eq!(got, want);
    }

    // seek = lower_bound:命中 / 缝隙 / 空串 / 超界。
    #[test]
    fn memtable_iter_seek_lower_bound() {
        let mt = MemTable::new();
        for i in 0..10u32 { mt.put(Bytes::from(format!("k{i:02}")), Bytes::from("v")); }
        let mut it = mt.iter_lsm();
        it.seek(b"k05");  assert_eq!(it.key(), b"k05");   // 命中
        it.seek(b"k05x"); assert_eq!(it.key(), b"k06");   // 缝隙 → 下一个
        it.seek(b"");     assert_eq!(it.key(), b"k00");   // 空串 → 第一个
        it.seek(b"zzz");  assert!(!it.valid());          // 超界 → invalid
    }

    // 活游标(非快照):iter 在插入前建,seek 时仍读到之后插入的键。
    // 锁住"惰性跳表游标"语义,防回退成构造期快照。
    #[test]
    fn memtable_iter_is_live_not_snapshot() {
        let mt = MemTable::new();
        mt.put(Bytes::from("k01"), Bytes::from("v1"));
        let mut it = mt.iter_lsm();                       // 此刻不快照任何东西
        mt.put(Bytes::from("k02"), Bytes::from("v2"));      // 建 iter 之后才插入
        it.seek_to_first();
        assert_eq!(it.key(), b"k01");
        it.next(); assert_eq!(it.key(), b"k02");          // 快照实现会漏掉 k02;活游标看得到
        it.next(); assert!(!it.valid());
    }

🔄 CI/CD:跨源同序作为差分门

F3 给 CI 加的不是新 job,而是一条结构性差分:同一组数据,MemTableIter 与 SstIter 吐出的序列必须相等。它和"读放大不变量""压缩 round-trip"一样,是把抽象契约钉成可执行断言——任何一个源的 iterator 实现退化(漏条、错序、seek 偏),这条差分立刻红。等 Phase G 落地,它升级成"N 源归并 == 合并后 BTreeMap",成为 Tier 2 最核心的回归门之一。

核心问题(自己答)

F3 想透这几个
  1. "自引用只能快照"哪里错了? 很多人会停在"crossbeam Entry 借 guard → 自引用 → collect 成 Vec"。错在哪一步?(提示:混淆了底层裸 skiplist(Entry 借你持的 Guard,真自引用)与高层 map::Entry<'a>(借外部 map、refcount 钉节点,无 guard 可借)。后者放进 struct { mt: &'a, cur: Option<Entry<'a>> } 两字段都借外部,毫无自引用。"借自己"才自引用,"借外部"从来不是。)
  2. 什么时候才真需要快照/ouroboros? 本课没退快照。那哪种写法会逼你上 unsafe/ouroboros?(提示:想要 'static 迭代器——struct { mt: Arc<MemTable>, cur: Entry<'???> },cur 借 self.mt 自己的字段,'??? 无从命名 → 真自引用。要么接受带 'a 的借用游标(本课),要么持 Arc<MemTable>+只存 cur_key 每步重新 lower_bound(O(log n) next、'static、无 Entry),要么 ouroboros 拼 Arc+Entry。"全都要(O(1)+'static)"才需 unsafe。)
  3. 多个 immutable memtable 怎么办? 引擎可能有 1 个 active + 多个 immutable memtable。Phase G 要归并它们 + 多个 SST。每个 memtable 一个 MemTableIter,对吧?那同一个 key 在 active 和某个 immutable 里都有(覆盖写),归并时谁赢?(提示:active 更新 → 该赢。但 plain key 的归并分不出新旧——只能靠源的优先级(active 排在 immutable 前、新 SST 排在旧 SST 前)。这正是 Phase G 的"源优先级去重",也正是为什么 Phase H 要引入 seq 把"新旧"变成 key 的一部分。)
  4. seek 之后 next,会不会重复吐出 seek 落点? seek(t) 把 idx 停在第一个 ≥t;接着 next 是 idx+1。那 seek 落点这一条,是先被 key() 读、还是被 next 跳过?(提示:契约是"seek 后当前就停在落点",调用方先 key()/value() 读它、再 next。所以 seek 落点会被读到,不跳过。这就是 F1 讲的"窥视与前进解耦"——seek 是定位+窥视,next 才前进。)

下一步:Phase G — MergeIterator(k 路归并). 三个源同 trait 后,G 持一个 Vec<Box<dyn LsmIterator>> + 一个外置比较器:每轮看齐所有源的当前 key()、用比较器选最小的、只推进那一个源;同 key 多源出现时按源优先级去重(active memtable > immutable > 新 SST > 旧 SST)。G 收尾切一片 minimal scan(start..end) 当里程碑。注意:G 这次在 plain key 上跑通;到 Phase H3 会把比较器换成 InternalKeyComparator——那次"重写"是刻意的教学点,让你亲历 seq 如何改变归并的去重语义。
