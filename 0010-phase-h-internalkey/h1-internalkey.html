STEP H1 · Phase H 多版本

InternalKey — 字节可比的 seq+type 编码

到 Phase G 为止,引擎对"同一个 key 的多个版本"只能靠源优先级猜新旧——可一旦 compaction 把多个版本写进同一个 SST,源顺序就分不清了。Phase H 给每个 key 编上序列号 seq(谁新)和类型 type(Put / Delete)。本课 H1 只做纯格式:定义 InternalKey 的编码 / 解码 / 比较,不碰任何存储。H2 把它穿进写路径,H3 穿进读路径 + 墓碑。

GOAL

新建 internal_key.rs:encode(user_key, seq, type) -> Bytes、user_key / seq / value_type 解析、lookup_key。编码须字节可比:裸 [u8]::cmp 即给出 user_key 升序、同 user_key 内 seq 降序。lib.rs 加 pub mod internal_key;。

Done when: round-trip 解析正确;排序契约绿(a@9 < a@5 < a@3 < b@1);lookup_key(k, snap) 的 lower_bound 落到 seq≤snap 的最新版本。

核心设计:为什么是「字节可比编码」而不是「穿一个 Comparator」

RocksDB 的 InternalKey = user_key + (seq<<8 | type),再配一个 InternalKeyComparator:user_key 升序、seq 降序(新值排前)。C++ 的容器(memtable skiplist、block)都收一个 comparator 仿函数,于是到处穿这个比较器即可。

Rust 这边有个硬约束:memtable 是 crossbeam_skiplist::SkipMap<Bytes, Bytes>,按 Bytes 的 Ord(裸字节)排序,不收外部 comparator;SST 块的二分也建在裸字节上。要把自定义 comparator 穿进这两处,代价很大。两条路:

方案做法取舍
✅ 字节可比编码把 seq 编成 be(u64::MAX - seq)(取反),让裸字节 Ord 直接 == 期望序memtable / block / 归并堆全用默认字节比较,零 comparator 穿线;代价:编码时算一下取反
comparator 穿线保留 user_key/seq 分离,把 InternalKeyComparator 一路传给 SkipMap/block/merge更贴 RocksDB;但 Rust 的 SkipMap<K: Ord> 不收 comparator,穿线侵入性大

我们选字节可比编码——它最契合 Rust 的 Ord 模型:存储层(memtable / SST)什么都不用改比较逻辑,只是 key 的字节变长了。唯一"懂 InternalKey"的地方收敛到读路径与归并去重(H3)。这正是 RocksDB 文档里 "bytewise-comparable encoding" 的思路。

编码布局 + 取反 seq 的魔法

    InternalKey = user_key ‖ big_endian(u64::MAX - seq) ‖ [type:1]
                             └──────── 8 字节 trailer ────────┘└ 1 ┘

    想要的序:  user_key 升序  →  同 user_key 内 seq 降序(新值排前)
    裸字节序:  user_key 升序(前缀主导)  →  trailer 升序
    要让 "trailer 字节升序" == "seq 降序",存 u64::MAX - seq:
        seq 大(新) → MAX-seq 小 → 字节靠前 → 排在同 user_key 最前 ✓

举例:a@9 的 trailer 字节 be(MAX-9) 比 a@5 的 be(MAX-5) 小 → a@9 排在 a@5 前。于是同一个 user_key 的所有版本,seq 最大(最新)的天然落在最前——seek 到这个 user_key,第一个就是最新版本。这把"找最新版本"从"扫所有版本比 seq"降成"lower_bound 一下"。

步骤

  1. 新建 internal_key.rs:常量 TYPE_DELETION=0 / TYPE_VALUE=1 / TRAILER_LEN=9;encode(「代码 1」)。lib.rs 加模块。
  2. 解析:user_key(切前缀)、unpack_trailer -> (seq, type)、lookup_key(「代码 1」)。
  3. 敲入「代码 2」:round-trip + 排序契约 + lookup_key harness,cargo test internal_key 全绿。

代码 1 — internal_key.rs

    use bytes::Bytes;

    pub type Seq = u64;                  // 每写自增,全局单调,越大越新
    pub const TYPE_DELETION: u8 = 0;      // 墓碑
    pub const TYPE_VALUE: u8 = 1;         // 普通写
    pub const TRAILER_LEN: usize = 9;     // 8(取反 seq 大端)+ 1(type)

    pub fn encode(user_key: &[u8], seq: Seq, vtype: u8) -> Bytes {
        let mut v = Vec::with_capacity(user_key.len() + TRAILER_LEN);
        v.extend_from_slice(user_key);
        v.extend_from_slice(&(u64::MAX - seq).to_be_bytes());  // 取反 → 字节升序即 seq 降序
        v.push(vtype);
        Bytes::from(v)
    }
    pub fn user_key(ik: &[u8]) -> &[u8] { &ik[..ik.len() - TRAILER_LEN] }  // 零拷贝切片
    pub fn unpack_trailer(ik: &[u8]) -> (Seq, u8) {
        let n = ik.len();
        let inv = u64::from_be_bytes(ik[n - TRAILER_LEN..n - 1].try_into().unwrap());
        (u64::MAX - inv, ik[n - 1])
    }
    pub fn seq(ik: &[u8]) -> Seq { unpack_trailer(ik).0 }
    pub fn value_type(ik: &[u8]) -> u8 { unpack_trailer(ik).1 }

    /// 查找键:user_key 在 seq<=snapshot 里的最新版本 = lower_bound(lookup_key(uk, snap))。
    /// type 取**最小**(0):同 seq 上墓碑(type 0)排在 value(type 1)前,lookup 用最小 type
    /// 才排在该 seq 任何条目前或相等,lower_bound 不会跳过墓碑。(Phase J 有限快照下这点关键。)
    pub fn lookup_key(user_key: &[u8], snapshot_seq: Seq) -> Bytes {
        encode(user_key, snapshot_seq, 0)   // 0 = 最小 type,seek 用
    }
lookup_key 是 H3/J 的钥匙。 点查"user_key 在快照 snap 下的值" = 对所有源 seek(lookup_key(uk, snap)),落点若 user_key()==uk 即该 uk 在 seq≤snap 中的最新版本(seq 降序的最前那个);看它的 type 决定返回 value 还是 None(墓碑)。snap 给 u64::MAX 就是"读最新"(H3);给一个历史 seq 就是"读快照"(J)。同一把 lookup_key,H3 与 J 只差传入的 snap。

🧰 Coding style:取反编码 · 切片解析零拷贝 · 常量命名对齐 RocksDB

  • 取反编码让默认 Ord 干活:与其到处穿自定义比较器,不如把"期望的序"烘进字节布局,让语言默认的 Ord 直接给对结果。这是"order-preserving encoding"的通用思路(整数大端、有符号数翻符号位、时间戳倒序…),数据库索引键编码里无处不在。
  • 解析全是零拷贝切片:user_key(ik) 返回 &ik[..n-9],不分配;trailer 用 from_be_bytes 定长解。InternalKey 在内存里就一段连续字节,user_key 与 trailer 是它的两个切片视图。
  • 常量对齐 RocksDB:TYPE_DELETION=0 / TYPE_VALUE=1 与 RocksDB kTypeDeletion / kTypeValue 同值。照搬业界约定,日后读 RocksDB 源码无缝对照。

🧠 System mindset:多版本 = 把"时间"编进 key

LSM(以及一切 MVCC 系统)的根本一招:不原地改数据,而是给每次写盖一个版本戳(seq),让新旧并存、读时按版本挑。InternalKey 就是这个版本戳的载体。一旦 key 带了 seq:

能力怎么来的
覆盖写不再靠源顺序同 user_key 多版本,seq 大者即最新——跨任何源、任何层都成立(compaction 混层也不怕)
删除变成一种写TYPE_DELETION 墓碑:删 = 写一条 type=Delete 的 InternalKey,归并时遮蔽旧值(H3)
快照读 = 时间旅行带一个 snapshot_seq,只看 seq≤snap 的版本,无视之后的写——读到"过去的一致视图"(J)

迁移收获:把"版本/时间"编进 key,是 MVCC、Git 对象、事件溯源、时序数据库的共同地基。"不可变 + 版本戳 + 读时挑版本"替代"可变 + 原地改",换来:并发读不加锁、历史可回溯、删除可延迟。代价是空间(旧版本堆积)——所以才需要 compaction(Tier 3)去回收。H1 立的这个小小 9 字节 trailer,是后面所有多版本能力的起点。

🧪 Harness 系统:排序契约 + lookup 落位

编码的正确性 = 排序 + 可逆

两条性质钉死编码:① 可逆——encode 后 user_key/seq/type 原样取回(含空 user_key、seq=0 边界)。② 排序契约——一组 InternalKey 用裸 sort() 排,落位必须是 "user_key 升、同 user_key 内 seq 降"。第三条 lookup_key:对一列同 user_key 多版本(已 seq 降序),partition_point(< lookup_key(k, snap)) 必须落到 seq≤snap 的最新那条。这条直接预演 H3/J 的点查,把"快照可见性"在纯格式层先验一遍。

代码 2 — harness(internal_key.rs tests)

    use std::cmp::Ordering;

    #[test]
    fn roundtrip_extracts_parts() {
        let ik = encode(b"apple", 42, TYPE_VALUE);
        assert_eq!(user_key(&ik), b"apple");
        assert_eq!(seq(&ik), 42);
        assert_eq!(value_type(&ik), TYPE_VALUE);
        let tomb = encode(b"", 0, TYPE_DELETION);       // 空 uk + seq 0 边界
        assert_eq!(unpack_trailer(&tomb), (0, TYPE_DELETION));
    }

    #[test] // 裸字节序 = user_key 升、同 user_key seq 降
    fn bytewise_order_is_userkey_asc_then_seq_desc() {
        let (a9, a5, a3, b1) = (encode(b"a",9,1), encode(b"a",5,1), encode(b"a",3,1), encode(b"b",1,1));
        assert_eq!(a9.as_ref().cmp(a5.as_ref()), Ordering::Less);  // 同 uk seq 大者前
        assert_eq!(a3.as_ref().cmp(b1.as_ref()), Ordering::Less);  // uk 主导
        let mut v = vec![b1.clone(), a3.clone(), a9.clone(), a5.clone()];
        v.sort();
        assert_eq!(v, vec![a9, a5, a3, b1]);              // 落位:a9,a5,a3,b1
    }

    #[test] // lookup_key:lower_bound 落到 seq<=snap 的最新版本
    fn lookup_key_lands_on_newest_visible() {
        let versions = [encode(b"k",10,1), encode(b"k",7,1), encode(b"k",3,1)]; // seq 降序
        let lk = lookup_key(b"k", 8);
        let idx = versions.partition_point(|ik| ik.as_ref() < lk.as_ref());
        assert_eq!(seq(&versions[idx]), 7);             // 看到 <=8 的最新 = 7(10 被跳过)
    }

🏁 H1 达成:多版本的「字」造好了。InternalKey 能编、能解、字节序天然对。但还没人用它——memtable / WAL / SST 存的还是 plain key。H2 把它穿进写路径:每次 put 分配一个 seq、存 InternalKey;WAL record 带上 seq+type。

核心问题(自己答)

H1 想透这几个
  1. 为什么 seq 要取反存,而不是直接存?(直接存大端 seq,字节序 = seq 升序 = 旧值排前;但我们要新值排前(seek 第一个就是最新)。取反 MAX-seq 把升序翻成降序。换个等价做法:存原 seq、但比较时把 trailer 当"降序"——那又得自定义 comparator,绕回老问题。取反编码把降序烘进字节,默认 Ord 就对。)
  2. type 放在 trailer最后一字节,会影响排序吗?(几乎不会:同 user_key 的两条若 seq 也相同,才轮到 type 比——而 seq 全局唯一(每写自增),同 user_key 同 seq 不会出现。所以 type 在排序里是"死字节",纯粹用来在读路径区分 Put/Delete。RocksDB 把 type 和 seq 打包进一个 u64,我们分开放,语义一致、更易读。)
  3. 为什么存储层(memtable/SST)不用改比较逻辑?(因为编码是字节可比的:它们本就按裸字节 Ord 排,而我们的字节序已经== 期望的 InternalKey 序。它们只是"key 变长了 9 字节",排序代码一行不改。这就是字节编码相对 comparator 穿线的最大红利——把语义负担从"每处比较"挪到"一次编码"。)
  4. seq 会用完吗?(u64,每秒百万写也要约 58 万年。实践中不担心。RocksDB 同样 u64(实为 56-bit,留 8-bit 给 type)。设计上"够大到不用考虑回绕"是一种常见的工程简化。)

下一步:H2 — 写路径穿 seq. 引擎加一个全局 seq 计数器,每次 put 自增并分配;memtable 存的 key 从 user_key 变成 encode(uk, seq, TYPE_VALUE);WAL record 带上 seq+type(replay 时重建带版本的 InternalKey)。get/scan 也要顺势改成"按 user_key 取最新版本"(否则存了 InternalKey 却用 plain key 查会失配)。
