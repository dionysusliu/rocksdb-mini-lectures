墓碑 + delete() — 删除是一种写

H2 把 seq 穿透了引擎,但所有写都是 TYPE_VALUE——还不能删。H3 引入 TYPE_DELETION 墓碑:删一个 key,不是去存储里把它抹掉(它可能散在 memtable + 好几个 SST 里,逐个抹掉既慢又违反"只追加"),而是写一条墓碑——一个 type=Delete、更大 seq 的新版本。读到墓碑即"此 key 已删";归并时墓碑作为最新版本,遮蔽下层所有旧值。真正回收空间要等 compaction 把墓碑和被它遮的旧值一起丢掉(Tier 3)。

GOAL

Engine::delete(key):分配 seq、写 encode(key, seq, TYPE_DELETION) 墓碑(value 空)进 WAL + memtable。读路径在 H2 已留好口子(get 见 type != VALUE 返回 None、scan 跳过非 VALUE),H3 主要是加 delete 并验证语义。

Done when: 删后 get=None / scan 不见;删后重新 put 又复活;memtable 墓碑遮蔽已 flush 的 SST 旧值(跨 flush 仍 None);get(k) ≡ scan(k, succ(k)).first() 在 put/delete/flush 随机流下恒成立。

删除为什么是"写一条墓碑",而不是"抹掉"

LSM 的铁律是只追加、不原地改。一个 key 的历史版本可能散在 active memtable、好几个 immutable、一摞不同层的 SST 里。要"抹掉"它,得把每一处都找出来删——既要随机写(违反顺序写)、又要改不可变的 SST(违反只读)。所以 LSM 把删除也变成一次顺序追加:

写一个值删一个 key
动作put → encode(k, seq, VALUE) + valuedelete → encode(k, seq, DELETION) + 空
本质追加一个新版本追加一个新版本(只是 type 不同)
读到它返回 value返回 None(此 key 已删)
对旧值遮蔽(seq 更大)遮蔽(seq 更大)

墓碑和普通写走完全一样的路径:同样分配 seq、同样进 memtable / WAL / SST、同样靠"seq 大者最新"在归并里胜出。唯一区别是读到它时,type==DELETION 让结果变成 None 而非 value。删除被统一进了"写 + 多版本"框架——这就是 H1/H2 把 type 编进 key 的回报。

墓碑会"复活"被删的 key 吗?——不会,只要 seq 单调。 put k=v1(seq 1)→ delete k(seq 2,墓碑)→ put k=v2(seq 3)。三个版本按 seq 降序:v2@3, 墓碑@2, v1@1。读最新 = v2@3 → 复活成 v2。删除不是终态,只是"此刻最新版本恰好是墓碑"。这正是 delete_hides_then_reput_revives 验的。反过来:墓碑必须有比被删值更大的 seq,否则删不掉——又一次 seq 单调的重要性。

步骤

  1. Engine::delete(key):分配 seq、encode(.., TYPE_DELETION)、WAL + memtable 写空 value(「代码 1」)。
  2. 确认读路径已就位:get 的 (t == TYPE_VALUE).then_some(v)、scan 的 value_type == TYPE_VALUE 过滤(H2 已写)。
  3. 敲入「代码 2」三组 harness(藏/复活 · 遮蔽 SST · get≡scan首元素),cargo test 全绿。

代码 1 — Engine::delete(engine.rs)

    pub fn delete(&mut self, key: Key) -> Result<()> {
        let s = self.seq; self.seq += 1;
        let ik = internal_key::encode(&key, s, internal_key::TYPE_DELETION);
        self.wal.append(&ik, &Bytes::new())?;      // 墓碑 value 空
        self.wal.sync()?;
        self.mem.put(ik, Bytes::new());
        if self.mem.is_full(self.flush_threshold) { self.flush()?; }
        Ok(())
    }

delete 和 put 几乎逐行一样——只差 type 和空 value。这正是"删除是一种写"在代码上的体现。读路径一行不用改:H2 写 get_newest 返回 (type, value) 时就预留了 type 分支,get 的 (t == TYPE_VALUE).then_some(v) 对墓碑天然给 None。好的抽象让后续语义"免费"加上——这是 H1 把 type 编进 key、H2 让读路径携带 type 的复利。

🧰 Coding style:删除即写 · 读路径预留 type 分支 · 互锁双路径

  • 删除收敛进"写 + 多版本":不为删除造一套新机制,而是复用整条写路径,只换一个 type。能用已有框架的一个新枚举值表达的语义,就别造新机制——更少代码、更少 bug、更易推理(墓碑和值同样参与 seq 定序、归并、compaction)。
  • 读路径早留扩展点:H2 让 get_newest 返回 (type, value) 而非裸 value,当时还没有删除,但留了 type。H3 直接受益——加 delete 即可,读路径零改。在抽象边界多带一点元信息(type),给未来语义留口子。
  • 双路径互锁当回归网:get(k) ≡ scan(k, succ(k)).first() 把点查和范围读绑成"必须给同一答案"。任一条路径处理墓碑/覆盖出错,这条 fuzz 差分立刻红。两个本应一致的实现互为 oracle,比各写各的断言更狠。

🧠 System mindset:逻辑删除 vs 物理删除

H3 划清了一条重要的线:逻辑删除(墓碑标记"已删",读不可见)与物理删除(真正回收空间)。LSM 把它俩解耦到不同时间:

逻辑删除(H3)物理删除(Tier 3 compaction)
何时delete 的瞬间compaction 归并到底层时
做什么写墓碑,读路径遮蔽墓碑 + 被遮旧值一起丢弃,空间回收
代价O(1) 一次写后台批量,摊薄

迁移收获:"标记删除 + 延迟回收"是高写入系统的通用招式——避免删除时的随机 IO,把回收攒成批后台做。同构:文件系统的 unlink(先解链接、空间后回收)、GC 语言的标记-清除、Kafka 的 log compaction、SQL 的软删除(deleted_at 标记)。代价是墓碑会占空间、还可能拖慢读(读得跳过一堆墓碑)——所以才有 Tier 3 的墓碑 GC(底层 compaction 才能真正丢弃,因为只有到了最底层才能确定"下面没有更旧的值需要这个墓碑去遮")。H3 标记,Tier 3 回收,这条分工后面会反复出现。

🧪 Harness 系统:藏/复活 · 跨层遮蔽 · 双路径互锁

三组各盯一类

① 藏 + 复活:put→get 有值;delete→get None & scan 空;再 put→get 新值。验"墓碑遮蔽 + seq 让复活生效"。② 跨层遮蔽:put→flush(值进 SST)→delete(墓碑在 memtable)→get None;再 flush(墓碑也进 SST)→仍 None。验"高层墓碑遮蔽低层 SST 旧值、且墓碑落盘后不失效"。③ 双路径互锁:2000 步随机 put/delete/freeze/flush,每步抽查所有 key get(k) == scan(k, succ(k)).first()。succ(k) = k ‖ 0x00 是 k 的紧邻后继,[k, succ) 恰好只含 k。这条把"点查正确"和"范围读正确"绑成一个不可分割的不变量。

    #[test] // 藏 + 复活
    fn delete_hides_then_reput_revives() {
        let mut e = Engine::open(dir).unwrap();
        e.put("k".into(), "v1".into()).unwrap();
        e.delete("k".into()).unwrap();
        assert_eq!(e.get(b"k"), None);                    // 墓碑遮蔽
        assert!(e.scan(b"", b"\xff").is_empty());
        e.put("k".into(), "v2".into()).unwrap();
        assert_eq!(e.get(b"k").as_deref(), Some(&b"v2"[..]));  // 复活
    }

    #[test] // 高层墓碑遮蔽 SST 旧值
    fn tombstone_shadows_sst_value() {
        let mut e = Engine::open(dir).unwrap();
        e.put("k".into(), "v".into()).unwrap(); e.flush().unwrap();  // v 落 SST
        e.delete("k".into()).unwrap();                       // 墓碑在 memtable
        assert_eq!(e.get(b"k"), None);
        e.flush().unwrap();                                  // 墓碑也落 SST
        assert_eq!(e.get(b"k"), None);                          // 仍遮蔽
    }

核心问题(自己答)

H3 / Phase H 收尾
  1. 墓碑能立刻从存储里删掉吗?(不能。墓碑要"遮蔽"散在下层的旧值,只要下层还可能有被删 key 的旧版本,墓碑就得留着挡。只有 compaction 把数据归并到最底层、确认下面再无更旧版本时,才能把墓碑和它遮的旧值一起丢——Tier 3 的墓碑 GC。过早删墓碑 = 旧值复活,是 LSM 经典 bug。)
  2. 为什么读路径在 H2 就"免费"支持了墓碑?(因为 H2 把 get_newest 设计成返回 (type, value)、get 写成 (t==VALUE).then_some(v)。当时没有删除,但 type 分支留好了。H3 只加 delete 这个写入侧,读侧零改。抽象边界上多带一点元信息,后续语义就能低成本叠加——这就是为什么值得在 H1 就把 type 编进 key。)
  3. get(k) ≡ scan(k, succ) 为什么是强约束?(get 和 scan 是两套独立实现(逐层命中即停 vs 归并去重),却必须对"k 的最新可见版本"给出同一答案。让它们互为 oracle,任一处理墓碑/覆盖/层序出错都会被另一条逮住。比"各自跟手写期望值比"更狠——因为手写期望也可能错,而两条独立实现同时错到一块儿的概率极低。)
  4. 墓碑的 value 为什么是空?(墓碑只表达"此 key 在此 seq 被删",不携带数据。空 value 省空间。读到墓碑根本不看 value(直接 None)。RocksDB 的 kTypeDeletion 同样无 value(另有 kTypeSingleDeletion、范围墓碑 kTypeRangeDeletion 等变体,我们 MVP 只要最基本那种)。)

下一步:Phase J — 快照(时间旅行). H3 的读路径处处用 snap = Seq::MAX 读最新。Phase J 把它参数化:get_as_of(k, snapshot_seq) / scan_as_of(.., snapshot_seq) 只看 seq ≤ snapshot 的版本,无视之后的写——读到"过去某一刻的一致视图"。钥匙还是 H1 的 lookup_key(uk, snap),只是 snap 不再恒为 MAX。MVCC 的核心就此落地。
