STEP H2 · Phase H 多版本

写路径穿 seq — InternalKey 进存储

H1 造好了 InternalKey 这个"字",但还没人用。H2 把它穿进引擎:每次 put 分配一个递增 seq,memtable / WAL / SST 存的 key 从 user_key 变成 encode(uk, seq, TYPE_VALUE)。一旦写的是 InternalKey,读也得跟着改——get / scan 不能再用 plain key 精确匹配,得"按 user_key 找最新版本"。所以本课同时落:写路径分配 seq + 读路径取最新 + 归并按 user_key 去重。本阶段全是 TYPE_VALUE(墓碑/删除是 H3)。

GOAL

① Engine 加 seq 计数器,put 分配 seq、存 InternalKey、WAL 也存 InternalKey;open 从 WAL+SST 恢复 seq 续号。② MemTable::get_newest / SstReader::get_newest(seek lookup_key 取最新版本)+ SstReader::max_seq。③ get 逐层 get_newest;scan 用 MergeIterator::with_dedup(user_key) 去重 + 剥 trailer。

Done when: 既有 F/G 测试全回归;同 memtable 多版本 get 取最新、scan 去重为一条;seq 跨重启单调(重启后覆盖写赢过旧值)。

关键链条:写 InternalKey ⇒ 读必须改

这是本课最该想透的连锁。一旦 memtable 的 key 是 encode(uk, seq, type):

原来(plain key)现在(InternalKey)
mem.get(uk) 精确匹配uk 不再是存储 key → 精确匹配必失配。改 seek(lookup_key(uk, MAX)) 落到该 uk 最新版本,再比 user_key()==uk
归并按整 key去重同 uk 不同 seq = 不同 key → 整 key 去重漏掉,一个 uk 会吐出所有版本。改按 user_key 去重
scan 吐存储 key存的是 InternalKey,得剥掉 9 字节 trailer再吐给用户
seq 不存在重启要恢复 seq:WAL 里的 ik 带 seq,但 flush 后 WAL 截断 → 还得扫 SST 取 max seq

所以 roadmap 把"写路径"和"读路径"分两步,但工程上它们必须同时落——存储格式一变,读路径不跟着改就直接挂。本课一并做掉;H3 只在此之上加"删除"这一种新 type 的语义。

步骤

  1. MemTable::get_newest(uk, snap) / SstReader::get_newest / SstReader::max_seq(「代码 1」)。
  2. MergeIterator 去重外置:加 dedup_key: fn(&[u8])->&[u8],with_dedup;next 按 dedup_key 分组(「代码 2」)。
  3. Engine:seq 字段;open 恢复 seq;put 编码 ik;get 逐层 get_newest;scan with_dedup + 剥 trailer(「代码 3」)。
  4. 敲入「代码 4」H2 harness,cargo test 全绿。

代码 1 — get_newest / max_seq(memtable.rs · sst.rs)

    // 两个 reader 同款逻辑:seek lookup_key 落到 user_key 最新可见版本。
    pub fn get_newest(&self, user_key: &[u8], snap: Seq) -> Option<(u8, Bytes)> {
        use crate::internal_key as ik;
        let lk = ik::lookup_key(user_key, snap);
        let mut it = self.iter();          // memtable: iter_lsm()
        it.seek(&lk);
        if it.valid() && ik::user_key(it.key()) == user_key {
            Some((ik::value_type(it.key()), Bytes::copy_from_slice(it.value())))
        } else { None }              // 本源无此 uk 的可见版本
    }

    // open 恢复 seq 用:本 SST 最大 seq(MVP O(n) 全扫)。
    pub fn max_seq(&self) -> Seq {
        let mut it = self.iter(); it.seek_to_first();
        let mut m = 0;
        while it.valid() { m = m.max(crate::internal_key::seq(it.key())); it.next(); }
        m
    }

seek(lookup_key(uk, snap)) 为什么落到"最新可见版本"?H1 讲过:同 uk 的版本 seq 降序排列,lookup_key(uk, snap) 的 trailer = be(MAX-snap) 恰是 seq≤snap 区段的下界——lower_bound 它,落在 seq≤snap 里 seq 最大(最新)那条。snap=MAX 即"读最新"。

代码 2 — 归并去重外置(merge.rs)

    fn identity_key(k: &[u8]) -> &[u8] { k }     // 默认:整 key 去重(plain key/G1)

    pub struct MergeIterator<'a> {
        sources: Vec<Box<dyn LsmIterator + 'a>>,
        heap: BinaryHeap<Reverse<(Vec<u8>, usize)>>,
        dedup_key: fn(&[u8]) -> &[u8],          // 物理 key → 逻辑 key
    }
    impl<'a> MergeIterator<'a> {
        pub fn new(sources: …) -> Self { Self::with_dedup(sources, identity_key) }
        pub fn with_dedup(sources: …, dedup_key: fn(&[u8])->&[u8]) -> Self { … }
    }
    // next:把所有 dedup_key == 当前堆顶 dedup_key 的源全推进(同 user_key 的多版本归一)
    fn next(&mut self) {
        let cur = match self.heap.peek() {
            Some(Reverse((k, _))) => (self.dedup_key)(k).to_vec(), None => return,
        };
        while let Some(Reverse((k, _))) = self.heap.peek() {
            if (self.dedup_key)(k) != cur.as_slice() { break; }
            let Reverse((_, idx)) = self.heap.pop().unwrap();
            self.sources[idx].next();
            if self.sources[idx].valid() { self.heap.push(Reverse((self.sources[idx].key().to_vec(), idx))); }
        }
    }

这就是 roadmap 说的"比较器重写",但更轻。 排序仍是裸字节(字节可比编码已让 ik 字节序 == user_key 升 / seq 降);只有去重分组换成 user_key。堆顶(最小 ik)= 该 user_key 最新版本,被读走后,next 把这个 user_key 的其余版本(含同源更旧版、低优先级源)全吞掉。fn 指针外置,G1 的 plain-key 测试用 new(identity)一字不改。

代码 3 — Engine 穿 seq(engine.rs)

    struct Engine { …, seq: Seq }   // 下一个要分配的 seq

    // open:WAL 的 ik 带 seq;flush 后 WAL 截断 → 还得扫 SST 取 max seq
    let mut next_seq = 0;
    for (k, v) in Wal::replay(…)? { next_seq = next_seq.max(ik::seq(&k) + 1); mem.put(k, v); }
    for r in &ssts { next_seq = next_seq.max(r.max_seq() + 1); }

    pub fn put(&mut self, key: Key, value: Value) -> Result<()> {
        let s = self.seq; self.seq += 1;
        let ik = internal_key::encode(&key, s, TYPE_VALUE);
        self.wal.append(&ik, &value)?; self.wal.sync()?;   // WAL 存 ik,replay 重建版本
        self.mem.put(ik, value);  …
    }

    pub fn get(&self, key: &[u8]) -> Option<Value> {
        let snap = Seq::MAX;                            // 读最新
        if let Some((t, v)) = self.mem.get_newest(key, snap) { return (t == TYPE_VALUE).then_some(v); }
        for m in &self.imm { if let Some((t, v)) = m.get_newest(key, snap) { return (t==TYPE_VALUE).then_some(v); } }
        for r in &self.ssts { if let Some((t, v)) = r.get_newest(key, snap) { return (t==TYPE_VALUE).then_some(v); } }
        None                                           // 第一个含该 uk 的源即最新(高优先级=更新 seq);命中即停
    }

    pub fn scan(&self, start: &[u8], end: &[u8]) -> Vec<(Key, Value)> {
        … sources 按优先级 …
        let mut it = MergeIterator::with_dedup(sources, internal_key::user_key);
        it.seek(&internal_key::lookup_key(start, Seq::MAX));
        while it.valid() {
            let uk = internal_key::user_key(it.key());
            if uk >= end { break; }
            if internal_key::value_type(it.key()) == TYPE_VALUE {   // 墓碑跳过(H3);H2 全 VALUE
                out.push((Bytes::copy_from_slice(uk), Bytes::copy_from_slice(it.value())));
            }
            it.next();
        }
        out
    }
get 仍是逐层命中即停(不走归并)。 为什么第一个含该 user_key 的源就是全局最新?因为 seq 随时间单调、数据只从 mem 往 SST 单向下沉:一个 key 的最新版本必在含它的最高优先级源里。所以 get 保留"逐层问、命中即停"的独立优化路径(后面 Phase R 还能加 bloom 跳过整层),不必为点查开一整套归并——这正是 F 阶段定的"get 与 scan 两条路径"。两路对"谁最新"必须一致,H3 用 get(k) ≡ scan(k..=k).first() 锁死。

🧰 Coding style:格式变更先想"读怎么跟" · 去重外置 · seq 续号双来源

  • 改存储格式,先问"读路径怎么跟":写格式一变,所有读它的地方(精确匹配、去重、对外解码)都得同步。本课的连锁(get/scan/merge/seq 恢复)就是一次完整演练。"谁写、谁读、谁解码"列清单再动手,比改一处编译一处稳。
  • 去重规则外置成 fn 指针:把"什么算同一逻辑 key"从 MergeIterator 里拎出来当参数,plain key 与 InternalKey 共用同一台归并机。这就是策略模式的最轻量形态(一个函数指针),换语义不改结构。
  • seq 续号要查所有持久层:WAL(未 flush 的)+ SST(已 flush 的)都可能藏着最大 seq。漏一个就会续号倒退、新写被旧值压住。"恢复单调计数器"必须扫尽所有可能的来源——这也暴露了"没有 manifest 持久化 seq"的 MVP 短板(Phase L 补)。

🧠 System mindset:版本戳进存储 = 不可变 + 单调时钟

H2 把"逻辑时钟 seq"焊进了每一次写。这一步之后,引擎里没有"修改"了,只有"追加新版本":覆盖 = 写一个更大 seq 的版本,旧版本仍躺在那(直到 compaction 回收)。这正是 MVCC / 事件溯源 / 不可变基础设施的共同形态:

系统它的"seq"
RocksDB / LevelDBSequenceNumber(全局单调,本课同款)
Postgres MVCCxmin/xmax(事务 ID)标记行版本可见区间
Gitcommit 链(每次提交一个不可变快照)
分布式系统Lamport 时钟 / 向量时钟(定因果序)

迁移收获:给写盖一个全局单调的版本戳,是把"并发 + 历史 + 删除"一次性讲清的地基。代价是 seq 必须真单调——重启不能倒退、并发不能重复(本课单线程暂不涉并发分配,那是 Tier 4 的题)。一个倒退的时钟会让旧值压住新值,是这类系统最隐蔽的 bug 之一,所以 seq_monotonic_across_reopen 单独立一条测试盯它。

🧪 Harness 系统:多版本去重 + seq 跨重启单调

两条新性质

① 同 memtable 多版本:连写 k=v1,v2,v3 不 flush,get(k)=v3、scan 里 k 只出一条(去重把 v1/v2 吞掉)。抓"去重漏版本 / 取错版本"。② seq 跨重启单调:写 k=before → flush(WAL 截断,seq 只剩在 SST)→ 重开 → 写 k=after → 必须 get=after。抓"重启 seq 没从 SST 恢复 → after 拿到 ≤before 的 seq → 旧值反而赢"。既有 F/G 全部测试照常回归——它们现在全程跑在 InternalKey 上,本身就是最强的"没改坏"证明。

代码 4 — H2 harness(engine.rs tests)

    #[test] // 同 memtable 多版本:get 取最新,scan 去重为一条
    fn multiple_versions_same_memtable() {
        let mut e = Engine::open(dir.path()).unwrap();
        for v in ["v1","v2","v3"] { e.put("k".into(), v.into()).unwrap(); }
        assert_eq!(e.get(b"k").as_deref(), Some(&b"v3"[..]));
        let s = e.scan(b"", b"\xff");
        assert_eq!(s.len(), 1);
        assert_eq!(s[0], (Bytes::from("k"), Bytes::from("v3")));
    }

    #[test] // seq 跨重启单调:重启后覆盖写赢过重启前旧值
    fn seq_monotonic_across_reopen() {
        { let mut e = Engine::open(dir).unwrap();
          e.put("k".into(), "before".into()).unwrap(); e.flush().unwrap(); }  // before@0 落 SST
        let mut e2 = Engine::open(dir).unwrap();         // 须从 SST 恢复 seq=1
        e2.put("k".into(), "after".into()).unwrap();
        assert_eq!(e2.get(b"k").as_deref(), Some(&b"after"[..]));   // after 赢
    }

核心问题(自己答)

H2 想透这几个
  1. 为什么 WAL 直接存 InternalKey,而不存 (uk, seq, type, v) 四元组?(InternalKey 本就把 uk+seq+type 编进一段连续字节 → WAL 的 "key" 字段塞 ik 即可,WAL 格式一行不改。replay 拿回 (ik, v) 直接 mem.put,seq/type 自带。少存一份、少一层解析。这是"把结构编进字节、让通道无脑搬运"的好处。)
  2. 为什么 get 第一个命中的源就是最新?能逐层命中即停吗?(能。seq 单调 + 数据单向下沉 ⇒ 一个 key 的最新版本必在含它的最高优先级源。所以逐层问、命中即停就对,不必归并。反例要警惕:若哪天允许"低层 seq 比高层新"(乱序写入/导入),这个前提就破,get 必须改成跨层比 seq。我们的引擎不会,故安全。)
  3. 同一个 user_key 在一个 memtable 里有多版本,next 怎么保证全吞掉?(while dedup_key==cur 循环:pop 胜者(最新)、推进它的源——该源很可能正好停在同 uk 的更旧版本,re-push 后循环再判 dedup_key==cur 仍真 → 再 pop 再推进。直到该源越过这个 user_key。所以"同源多版本"被循环自然吞干净,不漏。)
  4. seq 恢复为什么要扫 SST,不能只看 WAL?(flush 后 WAL 被截断,已落盘数据的 seq 只在 SST 里。只看 WAL → 重启后 seq 从 0 重来 → 新写的 seq 撞上甚至小于 SST 里的旧 seq → 旧值压新值。所以必须 max(WAL, 每个 SST 的 max_seq)。代价 O(n) 全扫——暴露了"seq 没被持久化"的短板,Phase L 的 manifest 会存下 seq,open 直接读,免全扫。)

下一步:H3 — 墓碑 + delete() + 两路径互锁. delete(k) = 写一条 encode(k, seq, TYPE_DELETION) 的墓碑(value 空)。get 见墓碑返回 None;scan 跳过墓碑;归并里墓碑作为该 user_key 的最新版本,遮蔽下面的旧值。再补 get(k) ≡ scan(k..=k).first() 把点查与范围读互锁。其实读路径在 H2 已留好 type != VALUE → None 的口子,H3 主要是加 delete 和验证语义。
