STEP C3 · Phase C SST

Engine::flush — 倒表闭环

把前两步接成一个动作:满了的 MemTable → 有序 SST 文件 → 清空内存表 → 截断 WAL。这就是 LSM 名字里的 "Log-Structured Merge" 中那个把内存日志结构倒进磁盘的关键一跳。

GOAL

交付 Engine::flush() -> PathBuf:遍历 MemTable(有序)喂 SstBuilder → 落盘 → mem 换新 → WAL 截断。

Done when: 跨持久化边界 round-trip——flush 前的逻辑内容 == flush 出的 SST 读回内容(覆盖写取最新);flush 后 WAL 不再重放旧数据。

为什么 flush 后能截断 WAL(mission 锚)

WAL 的唯一职责:保护还在内存、尚未落盘的写。一旦 MemTable 的内容被 flush 成持久 SST,那段 WAL 就完成使命、可以丢。RocksDB 用 log number + MANIFEST 精确记录"哪个 WAL 覆盖哪个 MemTable",安全删旧 WAL。我们 MVP 简化成"flush 后直接截断当前 WAL"——单 MemTable 够用,多 MemTable/并发 flush 的精确生命周期是 Phase L(Manifest)+ X(WAL 回收) 的活。

对标 flush job · 真实 WAL 生命周期:db_impl_open.cc / wal_manager.cc + MANIFEST

步骤

  1. 给 src/wal.rs 加一个 Wal::reset(截断重开,「代码 1」)。
  2. 扩展 src/engine.rs:Engine 加 dir + next_sst 字段,加 flush()(「代码 2」)。
  3. 敲入「代码 3」跨边界 round-trip harness,跑 cargo test 全绿。

代码 1 — wal.rs 加 reset

impl Wal {
    // ... 已有 open_append/append/sync/replay ...

    /// flush 后该段 WAL 已进 SST,截断重来。truncate=true 清空文件。
    pub fn reset(path: &Path) -> Result<Self> {
        let f = OpenOptions::new().create(true).write(true).truncate(true).open(path)?;
        Ok(Self { w: BufWriter::new(f) })
    }
}

代码 2 — 扩展 src/engine.rs（+ 为新增）

use std::path::{Path, PathBuf};
+ use crate::sst::SstBuilder;

pub struct Engine {
+   dir: PathBuf,        // flush 要往这个目录写 SST
    mem: MemTable,
    wal: Wal,
+   next_sst: u64,       // SST 文件编号,递增
}

impl Engine {
    pub fn open(dir: &Path) -> Result<Self> {
        std::fs::create_dir_all(dir)?;
        let wal_path = dir.join("wal.log");
        let mem = MemTable::new();
        for (k, v) in Wal::replay(&wal_path)? { mem.put(k, v); }
        let wal = Wal::open_append(&wal_path)?;
        Ok(Self { dir: dir.to_path_buf(), mem, wal, next_sst: 0 })
    }

    // put / get 不变(get 仍只查 mem;Phase D 才接 SST)

+   /// 冻结的 MemTable -> 一个有序 SST;成功后清空 MemTable + 截断 WAL。返回 SST 路径。
+   pub fn flush(&mut self) -> Result<PathBuf> {
+       let path = self.dir.join(format!("{:06}.sst", self.next_sst));
+       self.next_sst += 1;
+       let mut b = SstBuilder::new();
+       for (k, v) in self.mem.iter() { b.add(&k, &v)?; } // iter 有序 -> add 升序成立
+       b.finish(&path)?;
+       // SST 已持久 -> 该段 WAL 作废,截断;MemTable 换新
+       self.mem = MemTable::new();
+       self.wal = Wal::reset(&self.dir.join("wal.log"))?;
+       Ok(path)
+   }
}
顺序也是铁律. flush 里:先 finish(SST 落盘 fsync)成功,才截断 WAL。反过来——先截 WAL 再写 SST,崩在中间 → 数据既不在 WAL 也不在 SST = 丢。和 B 的 write-ahead 同一个道理:新持久副本就绪前,绝不毁旧副本。

🧰 Coding style：RAII 生命周期 + establish-new-before-destroy-old

  • RAII 资源生命周期:self.mem = MemTable::new() 整个替换旧 MemTable——旧的失去所有者,自动析构释放内存。self.wal = Wal::reset(...) 同理,旧 Wal(含 BufWriter/文件句柄)被 drop 时自动 flush+关闭。资源生命周期绑定对象生命周期,不用手动 free/close。迁移到任何 RAII 语言(C++ 析构、Python with、Go defer)。
  • establish-new-before-destroy-old(原子替换):flush 里先 finish 把新 SST 落盘成功,才 reset 截断旧 WAL。新副本就绪前,绝不毁旧副本。这是个反复出现的安全切换范式——下面 System mindset 展开,它直通 blue-green 部署。
  • 顺序即正确性:这段代码的对错不在"做了什么"而在"什么顺序做"。把关键顺序写成注释 + 不变量,是系统代码的基本功。

🧠 System mindset：crash-consistency · write-ordering · 新就绪前不毁旧

flush 的 finish → reset 顺序,和 B 的 "WAL 先于 MemTable",是同一条铁律的两次出现。这条铁律是所有有状态系统的命根:

领域"新就绪前不毁旧"长这样
文件系统journaling:先写 journal,再改原地;rename() 原子替换(写新临时文件 → fsync → rename 覆盖)
数据库checkpoint 前 WAL 不截;double-write buffer 防 torn page
部署(!)blue-green / 滚动发布:新版本起来、健康检查过,才切流量摘旧版本。和 flush 一模一样的形状
网络/分布式两阶段提交 prepare→commit;先复制到多数派,才回复客户端

迁移收获:任何"用新状态替换旧状态"的地方,问三个问题——(1) 新状态持久/就绪了吗(我们的 finish+fsync)?(2) 中间崩了会落在哪个半成品态,能恢复吗(C3 核心问题 #1/#2)?(3) 切换那一下是原子的吗(rename 是,"删旧+写新"两步不是)?把这三问刻进肌肉,你在 Linux 文件操作、DB 运维、服务发布、HPC checkpoint/restart 里都不会丢数据。LSM 的 flush 只是这条原理的一个实例。

Harness 系统：跨持久化边界 round-trip

为什么这样验

flush 跨越了一道边界:内存逻辑结构 → 磁盘字节格式。要证这道转换不丢不变:flush 前 MemTable 里"每个 key 的最新值",必须等于 flush 后 SST 读回的值。 再加一条:flush 后重开引擎,不能把已进 SST 的数据又从 WAL 重放一遍(否则 WAL 截断没生效,数据会重复/复活)。两条一起,锁死 flush 的正确性与 WAL 生命周期。

代码 3 — 跨边界 harness（接 engine.rs 的 tests）

    use crate::sst::SstReader;

    // flush 前 MemTable 内容 == flush 出的 SST(覆盖写取最新)
    #[test]
    fn flush_produces_correct_sst() {
        let dir = tempdir().unwrap();
        let mut e = Engine::open(dir.path()).unwrap();
        e.put(Bytes::from("banana"), Bytes::from("yellow")).unwrap();
        e.put(Bytes::from("apple"),  Bytes::from("red")).unwrap();
        e.put(Bytes::from("apple"),  Bytes::from("green")).unwrap(); // 覆盖
        let sst_path = e.flush().unwrap();
        assert_eq!(e.get(b"apple"), None); // flush 后 memtable 空
        let r = SstReader::open(&sst_path).unwrap();
        assert_eq!(r.len(), 2);                                   // 覆盖后只剩 2 个 key
        assert_eq!(r.get(b"apple").as_deref(), Some(&b"green"[..])); // 最新值
        assert_eq!(r.get(b"banana").as_deref(), Some(&b"yellow"[..]));
    }

    // flush 后 WAL 被截断:重开不重放已 flush 的数据
    #[test]
    fn wal_truncated_after_flush() {
        let dir = tempdir().unwrap();
        { let mut e = Engine::open(dir.path()).unwrap();
          e.put(Bytes::from("x"), Bytes::from("1")).unwrap();
          e.flush().unwrap(); }
        let e2 = Engine::open(dir.path()).unwrap();
        assert_eq!(e2.get(b"x"), None);                  // 不在 WAL/memtable(Phase D 才从 SST 读)
        assert!(dir.path().join("000000.sst").exists()); // SST 确实写出
    }

核心问题（自己答）

Phase C 收尾
  1. flush 的原子性:finish 写到一半崩了,会留下一个半截 SST 文件。重开时怎么知道它不完整、该忽略?(提示:footer/magic 在最后写;没 footer 的文件 = 没写完。RocksDB 还靠 MANIFEST 只认"已登记"的 SST。)
  2. WAL 截断的时机风险:我们 finish 成功后才 reset WAL。如果 finish 成功但 reset 前崩了,会怎样?重开时 WAL 重放 + SST 都在 → 数据重复吗?对 get 结果有害吗?(想想覆盖写的幂等性。)
  3. 为什么不直接改 MemTable 而要换新:self.mem = MemTable::new() 整个换掉,而非清空旧的。如果有读线程正拿着旧 MemTable,换新 vs 清空,哪个安全?(这预告 Phase D 的 immutable memtable + SuperVersion:flush 时旧表转 immutable 仍可读,不是就地清。)
  4. flush 何时自动触发:现在 flush 要手动调。真实引擎在 put 里检测 mem.is_full(threshold) 自动触发(A3 的信号!)。为什么 RocksDB 是"后台线程 flush"而非"在 put 里同步 flush"?(提示:flush 是慢 IO,别堵写路径——关联 write stall,Phase W。)
