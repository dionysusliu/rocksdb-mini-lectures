STEP C1 · Phase C SST

SstBuilder — 有序写盘

SST 的第一性问题:一堆有序 KV,在磁盘上排成什么样,既能高效点查、又自描述(能独立读回)?这一步设计布局 + 写出去。

GOAL

交付 src/sst.rs 的 SstBuilder:add(key,val)(强制升序)+ finish(path) 落盘成 data+index+footer 三段。

Done when: 升序不变量被强制(乱序 add 报错);文件结构自洽(footer magic 正确)。完整验证连 C2 一起跑。

SST 文件布局(MVP)

data: entry* (升序)
index: count + (key,offset)*
footer: idx_off idx_len magic (24B)

entry = klen(u32) key vlen(u32) val。 index 每条 entry 一个 (key, data内偏移)。 读时从尾部读固定 24B footer → 定位 index → 二分。footer 在尾,因为写时长度未知;放尾才能一遍流式写。

三个设计选择(对标 RocksDB,这是要学的)

选择我们 MVPRocksDB
索引粒度每 entry 一项(全量索引)每 data block(~4KB)一项(稀疏),块内再线性/restart 找。block_builder.cc
key 存储完整 key,每条都存restart points 间前缀压缩(只存与前一 key 的差异部分)
footeridx_off + idx_len + magicindex handle + metaindex handle(指向 bloom/属性)+ magic + 版本。format.cc Footer
诚实标注简化. 砍了 block 分块、前缀压缩、metaindex(bloom/属性块)。代价:索引随 key 数线性涨(不是按块)、读时整文件入内存。这些正是 Phase E(block 化)、R(bloom)、S(block cache) 要补的。先把"有序自描述文件"跑通。

步骤

  1. 先给 src/memtable.rs 加一个有序遍历 iter()(「代码 1」)——flush 要靠它按序取出。
  2. 新建 src/sst.rs,敲入「代码 2」SstBuilder(SstReader 留 C2)。
  3. src/lib.rs 加 pub mod sst;。
  4. 升序不变量 harness(「代码 3」)。完整 round-trip 要等 C2 的 reader,这步先验"乱序被拒"。

代码 1 — memtable.rs 加有序 iter()

impl MemTable {

    /// 有序遍历(skiplist 天然字节序)。flush 成有序 SST 的基础。
    pub fn iter(&self) -> impl Iterator<Item = (Bytes, Bytes)> + '_ {
        self.map.iter().map(|e| (e.key().clone(), e.value().clone()))
    }
}

代码 2 — src/sst.rs 的 SstBuilder

// src/sst.rs — Sorted String Table:冻结 MemTable 倒成的有序不可变磁盘文件。
use std::fs::File;
use std::io::Write;
use std::path::Path;
use anyhow::{bail, Result};
use bytes::Bytes;

const MAGIC: u64 = 0x6d696e695f737374; // "mini_sst",读时校验格式
const FOOTER_LEN: usize = 24;            // idx_off(8) + idx_len(8) + magic(8)

/// add 必须按 key 升序调用(调用者保证,通常来自有序 MemTable::iter)。
pub struct SstBuilder {
    data: Vec<u8>,
    index: Vec<(Bytes, u64)>, // (key, data 内偏移)
    last_key: Option<Bytes>,
}
impl SstBuilder {
    pub fn new() -> Self { Self { data: Vec::new(), index: Vec::new(), last_key: None } }

    pub fn add(&mut self, key: &[u8], val: &[u8]) -> Result<()> {
        // 不变量:严格升序(输入应是去重后的有序流)。let-chain(edition 2024)折叠嵌套 if。
        if let Some(last) = &self.last_key
            && key <= last.as_ref()
        {
            bail!("SstBuilder.add 非升序: {:?} <= {:?}", key, last);
        }
        let off = self.data.len() as u64;
        self.data.extend_from_slice(&(key.len() as u32).to_le_bytes());
        self.data.extend_from_slice(key);
        self.data.extend_from_slice(&(val.len() as u32).to_le_bytes());
        self.data.extend_from_slice(val);
        self.index.push((Bytes::copy_from_slice(key), off));
        self.last_key = Some(Bytes::copy_from_slice(key));
        Ok(())
    }

    pub fn finish(self, path: &Path) -> Result<()> {
        let mut f = File::create(path)?;
        f.write_all(&self.data)?;
        let index_offset = self.data.len() as u64;
        // 编码 index:count,然后每条 klen|key|offset
        let mut idx = Vec::new();
        idx.extend_from_slice(&(self.index.len() as u32).to_le_bytes());
        for (k, off) in &self.index {
            idx.extend_from_slice(&(k.len() as u32).to_le_bytes());
            idx.extend_from_slice(k);
            idx.extend_from_slice(&off.to_le_bytes());
        }
        f.write_all(&idx)?;
        // footer:让文件自描述,读时从尾部固定 24B 解析
        let mut footer = Vec::with_capacity(FOOTER_LEN);
        footer.extend_from_slice(&index_offset.to_le_bytes());
        footer.extend_from_slice(&(idx.len() as u64).to_le_bytes());
        footer.extend_from_slice(&MAGIC.to_le_bytes());
        f.write_all(&footer)?;
        f.sync_all()?; // SST 也要落盘才算数(同 WAL 的持久边界)
        Ok(())
    }
}
impl Default for SstBuilder { fn default() -> Self { Self::new() } }

🧰 Coding style：builder 模式 + consuming self + 不变量在构造点强制

三个可迁移范式,全在这段小代码里:

  • Builder 模式:SstBuilder 累积状态(add 多次),finish 一次性产出成品。复杂对象的构造与表示分离——任意语言通用(Java StringBuilder、Rust *Builder、protobuf builder)。
  • Consuming self 终结:finish(self, ...) 按值取走 self——调用后 builder 被移动消费,无法再用。类型系统保证"finish 后不会再 add",非法状态在编译期就不可表达。这是 Rust 把生命周期编进 API 的典型,迁移到别处就是"一次性资源/终态对象"设计。
  • 不变量在构造点强制:升序检查放在 add 里,坏数据进不来而非"进来后再查"。"parse, don't validate" / "make illegal states unrepresentable" 的精神——把约束前移到边界。
  • let-chain(edition 2024):if let Some(x) = ... && cond 折叠嵌套 if。clippy 会盯这个(见下方 CI/CD)。

🧠 System mindset：自描述文件格式 · footer/trailer 在尾

"长度未知时,把目录/索引写在尾部,读时从尾往回解析"——这不是 LSM 独有,是文件格式的通用范式:

系统尾部放了什么
ZIP"中央目录"在文件末尾(End of Central Directory record)——这就是为什么能往 zip 尾追加而不重写
ELF(Linux 可执行)section header table 偏移记在头,但布局同理:元数据与数据分离、按偏移寻址
SQLite / PDFPDF 的 xref 表在尾;增量更新只追加 + 新 xref
我们的 SSTfooter(index 偏移 + magic)在尾,一遍流式写完即自描述

迁移收获:为什么"trailer 在尾"?因为流式写入时总长度未知——你不能在开头写"index 在偏移 X",X 那时还不存在。放尾 = 写一遍、不回头、不预留。这条直觉在设计任何 append-only 格式/协议(日志、列存、网络帧)时反复出现。magic 数则是格式自我标识:读到不符立刻拒,防止把别的文件/旧版本误读成数据。

Harness 系统 ⑥：格式稳定性 / 不变量

为什么这里这样验

磁盘格式的两类致命 bug:违反约定的输入悄悄写进去(乱序 → 二分失效),和结构写歪(偏移/magic 错 → 读不回)。 C1 先用不变量 harness 把"升序"这条契约钉死(乱序必报错);完整的"写回再读"格式 round-trip 等 C2 的 reader 到位一起验。先证"坏输入进不来"。

代码 3 — 升序不变量 harness（接 sst.rs 末尾）

#[cfg(test)]
mod tests {
    use super::*;
    // 不变量:非升序 add 必须报错(相等也拒——输入应已去重)
    #[test]
    fn rejects_unsorted() {
        let mut b = SstBuilder::new();
        b.add(b"b", b"1").unwrap();
        assert!(b.add(b"a", b"2").is_err()); // 逆序
        assert!(b.add(b"b", b"3").is_err()); // 相等
    }
}

CI 教学点:lint 是会咬人的

C2 会给 CI 加 clippy -D warnings 门禁。届时你会发现 clippy 真的报错:本课 add 的嵌套 if 会被 collapsible_if 抓到 → 提示用 let-chain 折叠(就是上面代码已经写成的样子)。 这是 CI 的价值现场:机器逼你写更地道的代码。我们提前按 clippy 的建议写了,所以门禁能过——你也可以故意写回嵌套 if,看 CI 变红,再改回来,体会这个回路。
