Phase C — SST

把冻结的 MemTable 倒成一个有序、不可变的磁盘文件。从此数据有了一个能二分查找、能被 compaction 合并的持久形态——这是 LSM 里 "SS" (Sorted String) 的由来。

Mission 锚点. WAL(Phase B)让数据持久,但它是无序追加——查一个 key 得重放全部。 SST 把数据按 key 排好、写成只读文件。还记得 A3 那个 is_full() "满了"信号吗?它就是 flush 的触发器:满 → 冻结 → 倒成 SST。 对标 RocksDB 的 BlockBasedTable + flush job。
本 Lab 宣称的 GOAL

交付 src/sst.rs(SstBuilder + SstReader)与 Engine::flush:

  1. 有序落盘:MemTable 的内容按 key 升序写成一个自描述的 SST 文件(data + index + footer)。
  2. 可点查:重新打开 SST,二分索引,O(log n) 读到任意 key 的 value,无需载入全部 value。
  3. flush 闭环:Engine::flush 把当前 MemTable 倒成 SST、清空内存表、截断 WAL。

Done when: cargo test 全绿,含格式 round-trip、二分差分、flush 闭环三类 harness。

本 Lab 起,每节课五块. 除 🎯Goal+代码 与 🧪Harness 外,新增三支柱(可迁移收获,引擎之外): 🧰 Coding style(编程范式)· 🧠 System mindset(跨域到 Linux/存储/网络/算法/HPC 的系统思维)· 🔄 CI/CD(逐课搭真流水线)。 总览见 路线图的"五块 + 三条横切学习弧"。

本 Lab 的 harness 主题：文件格式的可信

SST 是磁盘格式——它要被另一次进程、另一个版本、甚至另一台机器读回。格式 bug(偏移算错、字节序、footer 写歪)是静默的、迟发的。新增两类侧重:

Harness 系统抓什么用在

跨持久化边界 round-tripflush 前 MemTable 的逻辑内容 == flush 后 SST 读回的内容(含覆盖写取最新)C3
带着这个问题进 C1:SST 为什么必须有序?如果乱序写,(a) 点查、(b) 范围扫描、(c) 两个 SST 做 compaction 归并,分别会贵多少?(这三点正是 "Sorted" 在 SSTable 里值钱的原因。)
