STEP J · Phase J 快照

snapshot + get_as_of — 把"读哪个版本"变成参数

H3 的读路径处处写死 snap = Seq::MAX。本课把它抽成参数,落地 MVCC 时间旅行的点读版。

GOAL

Engine::snapshot() -> Seq 抓当前序列号;get_as_of(key, snapshot) 只看 seq < snapshot 的版本;get 改成 get_as_of(key, self.seq)。修正 internal_key::lookup_key 的 seek-type。

Done when: 历史快照读到当时版本(无视之后写/删);快照跨 flush 仍有效;空快照/未来写不可见;既有 H 测试全回归。

快照 = 一个独占上界 seq

快照就是一个数:序列号上界。约定 snapshot() 返回 self.seq(下一个要分配的 seq),语义是"seq < snapshot 的版本可见,>= 的不可见"。为什么这样就对?

事实推论
已写入的版本 seq ∈ [0, self.seq)snapshot = self.seq ⇒ 全部已写可见
之后的写拿到 seq ≥ self.seq≥ snapshot ⇒ 对这个快照不可见
空引擎 self.seq = 0snapshot = 0 ⇒ 无任何版本可见(之后的写 seq 0 也 ≥ 0,挡住)

所以"抓快照"就是记下此刻的 self.seq——O(1),不拷数据、不锁。读时拿这个数过滤版本。RocksDB 的 snapshot 本质也是一个 sequence number(SnapshotImpl::number_),外加一个引用让 compaction 别回收它需要的旧版本(我们 MVP 单线程不涉及那层 GC 协调)。

独占上界 vs 包含上界——差一就错。 我们用独占(visible iff seq < snapshot),配合 snapshot()=self.seq,空引擎和未来写都干净。落到 get_newest(uk, s)(它找 seq <= s),要传 s = snapshot - 1;snapshot==0 时直接返回 None(没有 seq < 0)。这个 off-by-one 是快照实现最容易错的地方——先把"可见 = seq ? snapshot"的不等号方向钉死,再决定传给 lower_bound 的值。

步骤

  1. internal_key::lookup_key 的 type 从 TYPE_VALUE 改成 0(最小 type),修 seek 跳墓碑的潜伏 bug(「代码 1」)。
  2. Engine::snapshot + get_as_of;get 转发 get_as_of(key, self.seq)(「代码 2」)。
  3. 敲入「代码 3」时间旅行 harness,cargo test 全绿。

代码 1 — 修 lookup_key 的 seek-type(internal_key.rs)

    pub fn lookup_key(user_key: &[u8], snapshot_seq: Seq) -> Bytes {
        // type 取最小(0):同一 seq 上墓碑(0)排在 value(1)前,
        // lookup 用最小 type 才排在该 seq 任何条目前/相等,lower_bound 不跳墓碑。
        encode(user_key, snapshot_seq, 0)
    }
为什么 H 阶段没暴露这个 bug? H 的读路径只用 snap = MAX。lookup_key(uk, MAX) 的 trailer = be(0),严格小于所有真实版本的 be(MAX-seq)(seq<MAX)——type 字节根本轮不到比较,seek 必落到最新版本。一到有限快照(J),lookup 的 be(MAX-snap) 与 seq=snap 版本的 be(MAX-snap) 相等,type 成了 tiebreaker:墓碑(type 0)排在 TYPE_VALUE(1) 的 lookup 之前 → seek 跳过它 → 删除"读不出来"。把 lookup 的 type 降到 0 即修。潜伏 bug 常被"某个参数恰好让它不发作"掩盖,换参数空间(MAX→有限)才现形——这也是为什么 J 的快照测试是 H 墓碑逻辑的真正试金石。

代码 2 — snapshot + get_as_of(engine.rs)

    pub fn snapshot(&self) -> Seq { self.seq }   // 独占上界:已写全可见,未来不可见

    pub fn get(&self, key: &[u8]) -> Option<Value> {
        self.get_as_of(key, self.seq)              // "读最新" = 在当前快照下读
    }

    pub fn get_as_of(&self, key: &[u8], snapshot: Seq) -> Option<Value> {
        if snapshot == 0 { return None; }            // 空快照:无可见版本
        let snap = snapshot - 1;                    // get_newest 是 seq<=snap;snap=snapshot-1 ⇔ seq
        if let Some((t, v)) = self.mem.get_newest(key, snap) { return (t == TYPE_VALUE).then_some(v); }
        for m in &self.imm { if let Some((t, v)) = m.get_newest(key, snap) { return (t==TYPE_VALUE).then_some(v); } }
        for r in &self.ssts { if let Some((t, v)) = r.get_newest(key, snap) { return (t==TYPE_VALUE).then_some(v); } }
        None
    }

逐层 get_newest(uk, snap) 各自只返回"本源里 seq≤snap 的最新版本"。第一个有可见版本的源即给出该快照下的最新——若高优先级源只有 seq>snap 的(不可见)版本,get_newest 自然返回 None,继续往下层找。所以历史快照能穿过较新的不可见写,落到当时的旧值。这正是 snapshot_survives_flush 里"v2 已写但快照仍读 v1"的机制。

🧰 Coding style:参数化代替写死 · off-by-one 先定方向 · 修 bug 连根

  • 把策略做成参数:H 把 snap 一路带到 get_newest 却恒传 MAX——等于"为参数化预留接口,先用默认值"。J 接上真参数,几乎零改。预见到"将来要可变"的维度,先做成参数留 MAX/默认,比日后重构便宜得多。
  • off-by-one 先定不等号:快照可见性 seq < snapshot(独占)一旦定死,传给 lower_bound 的 snapshot-1 和 snapshot==0 的早退都顺势推出。边界 bug 的根治是先把语义不等号写在注释里,再推实现值,而非反过来试。
  • 修 bug 要连根、并补能复现的测试:lookup_key 的 type bug 不是"J 新引入",是 H1 潜伏、J 暴露。修在根上(internal_key),并让 J 的快照测试成为它的回归网。潜伏 bug 修复后,要留一个"会触发它的参数"的测试钉住。

🧠 System mindset:快照隔离 = 读一个不动的过去

MVCC 的精髓:读者读一个固定版本号下的一致视图,写者照常往前写,二者互不阻塞。没有读写锁、没有读等写——读拿快照号过滤,写追加更大 seq。这套"多版本 + 版本号可见性"是现代数据库并发控制的主流:

系统快照是什么
RocksDB一个 sequence number(本课同款)+ 阻止 compaction 回收它需要的旧版本
PostgreSQL事务可见性快照(xmin/xmax + 活跃事务集)
Git一个 commit hash —— 整棵树的不可变快照
时序/分析库读到某个 watermark 之前的全部数据

迁移收获:要"一致地读一个变动中的系统",别去锁住它,而是给它的状态打版本号、读一个固定版本。代价是旧版本要保留到没有快照需要它为止——这就引出快照与 GC/compaction 的协调(谁还引用着最老的快照,compaction 就不能丢早于它的版本)。我们 MVP 单线程、不主动回收,所以暂不涉及;但真实系统里"快照引用计数 ↔ compaction 安全点"是核心一环(Tier 3 会再碰)。

🧪 Harness 系统:时间旅行三问

三条性质

① 读历史:put v1 → 抓 s1 → put v2 → 抓 s2 → delete。get=None(最新已删),get_as_of(s2)=v2,get_as_of(s1)=v1。一条线验三个时间点。② 跨 flush:抓快照后把新旧版本都 flush 进 SST,历史快照仍读到旧值(版本进哪层不影响可见性,只看 seq)。③ 未来不可见:空引擎抓 s0 → 之后 put,get_as_of(s0)=None。盯死"独占上界 + 空快照"边界。

代码 3 — harness(engine.rs tests)

    #[test] // 时间旅行
    fn snapshot_time_travel() {
        let mut e = Engine::open(dir).unwrap();
        e.put("k".into(), "v1".into()).unwrap();   let s1 = e.snapshot();
        e.put("k".into(), "v2".into()).unwrap();   let s2 = e.snapshot();
        e.delete("k".into()).unwrap();
        assert_eq!(e.get(b"k"), None);                          // 最新 = 已删
        assert_eq!(e.get_as_of(b"k", s2).as_deref(), Some(&b"v2"[..]));
        assert_eq!(e.get_as_of(b"k", s1).as_deref(), Some(&b"v1"[..]));
    }

    #[test] // 跨 flush 仍有效
    fn snapshot_survives_flush() {
        e.put("k".into(), "v1".into()).unwrap(); let s1 = e.snapshot();
        e.put("k".into(), "v2".into()).unwrap(); e.flush().unwrap();
        assert_eq!(e.get_as_of(b"k", s1).as_deref(), Some(&b"v1"[..]));  // 穿过 v2 读 v1
    }

    #[test] // 未来不可见
    fn snapshot_hides_future_writes() {
        let s0 = e.snapshot();                // 空引擎,s0 = 0
        e.put("k".into(), "v".into()).unwrap();
        assert_eq!(e.get_as_of(b"k", s0), None);   // s0 之后的写不可见
    }

核心问题(自己答)

J 想透这几个
  1. 抓快照为什么是 O(1)、不拷数据?(快照只是记下一个 seq 数。数据本就多版本并存(只追加,旧版本没被删),所以"过去的视图"一直躺在那,读时按 seq 过滤即可,无需在抓快照时复制任何东西。多版本存储让"快照"退化成一个数——这是 LSM/MVCC 比"拷贝整库"高明的地方。)
  2. 历史快照怎么"穿过"较新的写读到旧值?(逐层 get_newest(uk, snap):高优先级源若只有 seq>snap 的新版本,它返回 None(那版本不可见),于是继续往低层找,直到某层有 seq≤snap 的版本。所以新写不会挡住旧快照——它们 seq 太大,被过滤掉了。)
  3. 为什么 lookup_key 的 type 必须最小?和墓碑什么关系?(同一个 seq 上,墓碑 type 0 排在 value type 1 前。若被删 key 在 seq=snap 处是墓碑,lookup 用 type 1 会排在它后面 → lower_bound 跳过墓碑 → 删除在该快照下"读不出来"、旧值复活。type 取 0 让 lookup 排在该 seq 任何条目前/相等,墓碑不被跳过。这个 bug 在 snap=MAX 下永不发作,是 J 的有限快照才逼出来的——参数空间扩大是发现潜伏 bug 的常见途径。)
  4. 旧版本能一直留着吗?(不能,会涨爆空间。但有快照引用的旧版本不能被回收(否则该快照读到空)。所以真实系统把"最老的活跃快照 seq"当 compaction 的安全水位:早于它、且被新版本遮蔽的旧值才能丢。我们 MVP 不回收,所以不涉及;Tier 3 compaction 会正面处理"快照安全点"。)

下一步:Phase K — 对外范围扫描(快照 + 墓碑感知). 把 G2 的 scan 升级成一个真正的对外游标 DBIter:底层 MergeIterator 吐所有版本(不在归并层去重),上层 DBIter 按 snapshot 过滤 seq、按 user_key 取最新可见版本、跳墓碑。scan_as_of(start, end, snapshot) 就是范围版的时间旅行。这也终于把"去重 + 快照 + 墓碑"从 MergeIterator 里拆出来,放回它该在的 DBIter 层(对标 RocksDB 的分工)。
